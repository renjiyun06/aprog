状态与持久化

进程状态存储（Git 模型）

一个进程的状态表现为一个目录、带一串保存点——这正是 git 的模型。aprog 据此把存储定死: 一个进程 = 一个 git 仓库,一个检查点 = 一次 commit;耐久性靠 git push 到控制平面托管的 bare 仓库, 唤醒靠 git clone,前端查看状态直接按 ref 拉 git。本页是这套模型的唯一出处, 进程目录与状态 / 生命周期流程 / 数据模型 / wake 流程 等页与此对齐。

这取代了什么 早期设计里检查点是「进程目录的 state 子集经 DriverChannel(B 平面)流回 CP」的一套自定义传输, 唤醒时再 pushBundle 灌回沙箱。本决策用 git 取代这套自定义传输:传输就是 git push / clone, 对象库就是 git 自带的内容寻址存储。什么是检查点、何时打、装什么(程序 mark + turn.end 去抖、不含 session.jsonl)不变;变的只是它怎么存、怎么传。

进程 = 仓库,检查点 = commit

把「目录 + 保存点」逐字映射到 git,存储方案就自洽了:

git 概念aprog 对应
仓库（bare，权威副本）一个进程的耐久状态,托管在 CP 侧 /srv/aprog-repos/<pid>.git
工作树运行中沙箱里的进程目录(沙箱临时盘,不挂持久卷)
一次 commit一个检查点 / 保存点(不可变快照)
commit SHA(HEAD)PCB 的 checkpoint_ref
对象库(内容寻址 + 去重)检查点存储——跨保存点、跨进程去重,全历史很便宜
checkout <sha>rewind 到某个保存点
分支从某保存点 fork 一个进程(分享 / 试错)
为什么是 git 而不是自建 内容寻址、增量去重、全历史、diff、分支、push/pull 传输——全是现成的。自建 CAS 等于重造半个 git。 状态子集主要是 yaml/md/jsonl 文本,对 git 近乎完美;大二进制(artifacts/)另议(见 开放点)。

检查点 = 一次 commit（语义不变）

检查点的语义沿用 进程目录与状态 已定的:一致性归程序、耐久性归平台。 git 只是把它落成具体动作:

问题谁负责git 落法
此刻一致吗(commit mark)程序在语义干净点调检查点工具喊一声(mark / 意向)
何时真捕获平台 / driverturn.end 时若 mark 仍干净 → git add -A && git commit
怎么耐久平台 / drivergit push 到 CP bare remote(去抖,见 耐久传输)
装什么协议进程目录的 state 子集(meta.yml / input.md / execution-state/ / artifacts/);不含 session.jsonl(引擎私有缓存,artifacts 见开放点)

「mark 武装、turn.end 捕获」的去抖纪律不变:调用只是意向,捕获前若又写脏则等下一个干净 mark。checkpoint_ref = 这次 commit 的 SHA。

耐久传输:git push,不再走 B 平面

沙箱本地盘随时可能没(回收 / OOM / 崩溃)。耐久靠把 commit 推到 CP 托管的 bare 仓库—— 这就是「同步到控制平面」的全部,不再有经 DriverChannel 的自定义检查点传输。

沙箱(工作树) turn.end → commit → git push 到 CP remote。持每进程一把、仅授权该 repo 的 deploy 凭证(短时、可轮换)。
CP 托管 bare repo /srv/aprog-repos/<pid>.git。耐久真相在此,跨厂商。CP 退化成一个哑 git 端点 + 鉴权门,不再是自定义 relay。
wake = clone provider.create 新沙箱 → git clone --depth=1 + checkout <checkpoint_ref> → harness resume。取代旧的 pushBundle 灌回。
迁移 = 在厂商 Y 起沙箱、clone 同一个 remote——天然跨厂商,不依赖任何厂商热令牌。 历史留在 remote(供 rewind / 分支),沙箱只 --depth=1 拿最新态,冷启不必拉全历史。
不因换 git 而消失 崩溃窗口照旧:mark 后、push 成功前崩,丢的是上次 push 以来那段(git 改的是传输,不是去抖丢失语义)。 外部副作用照旧:程序自己开发的项目里的 git push 和「进程状态 repo」是两个不同的 repo,后者变 git 不解决前者的逃逸(见 外部副作用)。

前端:状态拉 git,事件走 CP

前端数据收敛成两个来源,不碰沙箱:

前端要的来源机制
进程状态 / 文件 / 产物 / 保存点历史 / diffgit(CP 托管 bare repo)smart-HTTP 按 ref 读 tree/blob;只读已提交
对话 / 事件(live + 回放)CP 事件流SSE subscribe({from:seq})(见 事件流协议)
新保存点通知CP 事件流(一种事件)收到 checkpoint-committed{ref} → 按新 ref 重拉 git
spawn / wake / hibernate / kill · input / 打断CP REST不变
只展示一致快照 尚未提交的态不给用户看。文件浏览器永远只显示程序 mark 过的一致快照,绝不出现写到一半的撕裂态——文件树停在上个保存点,等新 checkpoint 落地再整体跳一格。 活动看 live(事件流里有 tool_use: 写文件 X),产物看 checkpoint(git)。 故沙箱 fs 读穿透被移除(见 DriverChannel)。
鉴权 + 单对端 浏览器不能拿裸 repo 访问——「谁能看这个进程」的授权仍在 CP(authorize(user,pid,role))。 做法:CP 给 git smart-HTTP 加一道薄鉴权门,前端用同一个会话请求 …/<pid>.git/,同源、同 token。 于是前端唯一网络对端仍是 CP,只是对它说三种话:REST + SSE + git-http。CP 退出数据搬运,仍是策略权威。

DriverChannel 瘦身

检查点传输(→ git push)与 fs 读穿透(→ 前端拉 git)都拿走后,DriverChannel(B 平面) 只剩两件实时的事:

事件上行harness 产出 → driver 定序 → CP 盖全局 seq 落库 + 扇出前端
input 下行前端 → CP → DriverChannel → driver → 喂引擎
耐久(检查点)= git、读文件 = git,都不再是 B 平面的职责。原 FsServer 读路径(interaction.html)随之移除。

rewind 与分支

检查点能把进程倒带到上一个 commit(checkout)。但别 force-push 覆盖历史—— 倒带后往前跑就开一条分支,顺带把「从某保存点 fork 一个进程(分享 / 试错)」这个能力白送了。 倒带不了已逃出 repo 的外部副作用(典型 git push 到远端项目),处理仍见 外部副作用与游标。

开放点

  • artifacts/ 大二进制:权威产物但可能大。进同一 git(靠 LFS),还是单列 blob 路径?倾向 LFS 起步,实测再调。
  • remote 托管位置:默认 CP 托管 bare repo(保住单对端 + 单鉴权);是否允许指向外部 host(Gitea/Forgejo)做卸载?那会让前端多一个对端,暂不开。
  • shallow / 历史保留:沙箱 --depth=1;CP 侧全历史保留(去重后很便宜);是否需要 GC / 老 commit 归档,留待规模上来再定。
  • checkpoint-committed 事件:作为 协议 的一种事件类型落地,让前端据此重拉 git。
