协议与会话

事件流协议

一次会话就是一条 append-only 的事件流：底层引擎吐的每一片思考/工具/回复、以及用户的每次输入， 都是流上一个带顺序号的事件。这条流是唯一数据源，实时推送、断线恢复、持久存档 全部由它统一承载。引擎是可替换的，所以这套协议是 engine-neutral 的——各家引擎怎么映射进来见 Harness 适配。

会话即一条事件流

每个进程一条事件流。流上每个事件由 Driver 盖两个标识：

# seq 位置 单调递增，每个事件（含用户输入）都有。游标单位，回答“落后多少 / 补哪些”。
+
id 身份 逻辑分组，流一开始就定。把属于同一条消息的 delta 折叠成一条。回答“是不是同一条”。
seq 管重放到哪、id 管哪些折成一条。id 由 Driver 赋予；seq 由 control-plane 落库时盖全局单调（Driver 只保证一次运行内的局部顺序，见 沙箱交互界面 · seq 与 Harness 适配 · seq）。

事件种类

协议把任何引擎的产出归一成下面几类（harness-neutral）。一个“项(item)”= 一条思考 / 一条回复 / 一次工具调用 / 一次命令执行……它有 start → delta* → end 的生命周期：

事件含义关键字段
turn.start一个 agent 回合开始（用户提交后引擎开干）turn
user用户输入（完整，非流式；由 Driver 回显进流）id, content
item.start一个项开始id, item_type ∈ thinking / reply / tool / command / file_change …
item.delta项的增量id, patch（text / args / output…）
item.end项完成，带 coalesced 全量id, value
turn.end回合结束turn, usage?
error错误message

线上每条事件的信封：

{"seq":182, "ts":"…", "kind":"item.delta", "id":"msg_B:1", "patch":{"text":"找到了"}}
{"seq":183, "ts":"…", "kind":"item.end",   "id":"msg_B:1", "value":{"item":"reply","text":"找到了，是 join 写错"}}
不做授权 产品形态自动批准引擎的所有操作，所以协议里没有 allow/deny 事件——审批由 Driver 自动应答、不进流（Codex 那种审批往返见 Harness 适配）。

每类的字段

信封字段（每个事件都有）：seq:number（全局单调，control-plane 盖）、ts:string（ISO-8601，Driver 盖）、parent?:ItemId（子代理归属，可选，主流为空——见下）、raw?:unknown（引擎原始载荷，可选——见下「保真兜底」）。其余按 kind：

kind字段
turn.startturn:TurnId
userid:ItemId、content:string
item.startid:ItemId、item_type
item.deltaid:ItemId、patch:ItemPatch（见下）
item.endid:ItemId、value:ItemValue（见下，已合并全量）
turn.endturn:TurnId、stop_reason:StopReason（见下）、raw_stop_reason?:string、usage?:{ input?, output?, context_window?, extra? }
compactiontrigger:'auto'|'manual'、pre_tokens?:number（引擎压缩了自己的上下文，见下）
errorseverity:'fatal'|'transient'|'warning'、message:string（见下）

item.delta.patch（delta 增量）与 item.end.value（end 全量）是同一个项的两种形态，按项类型一一对应：

项类型patch（增量载荷）value（合并后全量）
thinking{ kind:"text", text }{ item:"thinking", text, signature? }
reply{ kind:"text", text }{ item:"reply", text }
tool{ kind:"tool_args", partial_json }{ item:"tool", name, args, result? }
command{ kind:"command_output", chunk }{ item:"command", command, output, exit_code? }
file_change{ kind:"file_change", diff }{ item:"file_change", path, diff }
权威类型见 packages/protocol/src/events.ts 与 envelope.ts。

turn.end.stop_reason（StopReason）归一成「用户不同反应」的小集——取值粒度 = 用户会做出的不同动作的数量，不再细；引擎原词进 raw_stop_reason。用户关心“为什么停了”，所以建模进流：

stop_reason用户的反应Claude 来源Codex 来源
completed正常说完，轮到我"end_turn" / "stop_sequence"TurnComplete
interrupted被打断/取代（中性）用户中断TurnAborted：Interrupted / Replaced / ReviewEnded
limit可能没说完，要不要继续"max_tokens" / error_max_turnsTurnAborted：BudgetLimited
refused它拒答了，换个问法"refusal"—
error它崩了error_during_executionError
max_tokens 与 max_turns（及 Codex 的 BudgetLimited）都塌进 limit——用户反应都是“撞天花板了，要不要继续”；哪种上限留给 raw_stop_reason 的调试视图。Claude 的 tool_use / pause_turn 不是回合结束（回合继续），不映射成 turn.end。
compaction ≠ 流的合并 compaction 事件记录的是引擎压缩了它自己的上下文（context compaction），和前面说的“流的合并 / 读时投影”是两码事——事件流本身永不删减。它如实落在流里，只为让用户在历史里看到“此处之前的上下文被引擎压缩过”。取自 Claude 的 compact_boundary / SDKCompactBoundaryMessage（带 trigger / pre_tokens）。
子代理：parent 字段 每个事件信封可带可选 parent:ItemId——子代理（subagent）产生的事件，parent 指向派生它的 Agent 工具调用的 id（Claude 的 parent_tool_use_id）；主流（顶层 agent）事件为空。前端据此把子代理的 thinking/tool/reply 折叠成一棵子树，而不是摊平进主流。
error 的 severity error 带严重级别，让 UI 区别对待，别把“正在重连”误报成“挂了”：fatal 致命、回合无法继续（红色，Codex Error）；transient 瞬时、系统正在自愈（黄色「重连中…」，Codex StreamError）；warning 继续了但需提示（黄色提示行，Codex Warning）。
保真兜底：raw 字段，与「typed 收敛」之道 协议要同时完美适配底层 harness（Claude / Codex）又保持自身抽象，靠的是一条生成规则：typed 字段只建模「用户该知道、且会据此行动」的东西，归一粒度 = 用户不同反应的数量；其余引擎丰富度原样进 raw?:unknown，携带而不建模。「完美适配」不等于「类型里建模一切」（那会毁掉抽象），而是一切都没丢——通用 UI 读 typed，高级检视器/调试/未来需求读 raw。Driver 在归一有损的事件（item.end / turn.end / error / compaction）上填 raw。用户的反应集是有限且引擎中性的（轮到我了吗 / 要不要继续 / 要担心吗 / 它怎么忘事了），贴合它，就同时贴合了所有 harness。

实时与持久是同一条流

这条流上既有 delta（细粒度，供“正在打字”的实时感），也有 item.end（coalesced 的完整条目）。 所以不需要两套通道：

  • 实时 = tail 流的末尾，delta 即推 → “正在打字”；
  • 持久 = 原始流本身就是存档，跟进程目录一起落盘、一起持久化——不删 delta、不做破坏性压实。因为 item.end 已带合并全量，原始流既能逐字回放、又不增加历史读取成本（压实只是将来可选的冷段省空间优化，不改协议、可后加）。
session.aprog.jsonl = 流的读时投影 那个“干净的会话记录”不是第二份数据源，而是把这条流合并后的样子（丢 delta、每个 id 取 item.end）。它由原始流随时按需重建，不必落地；留它只为给人/agent 省去每次折叠的开销。删了不丢信息。详见 三层数据。

合并：看历史给完整态，实时才打字

这条 delta 流可合并，因为每个 patch 都是对单个 id 缓冲区的追加（append-only）——没有乱序、没有冲突，合并就是顺序拼接，纯函数：

  • text / command_output / file_change：直接拼接；
  • tool_args：拼接 partial_json 分片，末尾一次 JSON.parse。

更省事的是：item.end.value 本身就是合并后的全量，所以已完成的项根本不用自己折叠——直接取 value。于是同一条流支持两种读法：

读法怎么读呈现
实时 attachtail：item.delta 即推，item.end 到了定稿替换逐字“打字”
看历史（如休眠进程）丢 turn.* / item.start / item.delta，每个 id 只取 item.end.value，连同 user 按 seq 排直接给合并后的完整记录，不打字
休眠进程被打开 → control-plane 从持久化的原始流回放 → 历史段全是已合并的 item.end（看历史不打字）；用户若继续 attach 实时，只有尾部那个未完成项才回到 delta 播放。seq 保证所有界面拿到同一顺序。

一个完整示例：一轮交互

把上面所有概念串起来——用户问“为什么登录偶尔失败？”，引擎想一下、跑一次 Grep、再回复。线上的原始流（持久化落盘的就是它，逐字 delta 全在）：

{"seq":41,"ts":"…","kind":"user","id":"u_41","content":"为什么登录偶尔失败？"}
{"seq":42,"ts":"…","kind":"turn.start","turn":"t7"}
{"seq":43,"ts":"…","kind":"item.start","id":"msg_a:0","item_type":"thinking"}
{"seq":44,"ts":"…","kind":"item.delta","id":"msg_a:0","patch":{"kind":"text","text":"先翻登录"}}
{"seq":45,"ts":"…","kind":"item.delta","id":"msg_a:0","patch":{"kind":"text","text":"相关代码"}}
{"seq":46,"ts":"…","kind":"item.end","id":"msg_a:0","value":{"item":"thinking","text":"先翻登录相关代码","signature":"Eq…"}}
{"seq":47,"ts":"…","kind":"item.start","id":"toolu_01","item_type":"tool"}
{"seq":48,"ts":"…","kind":"item.delta","id":"toolu_01","patch":{"kind":"tool_args","partial_json":"{\"q\":\""}}
{"seq":49,"ts":"…","kind":"item.delta","id":"toolu_01","patch":{"kind":"tool_args","partial_json":"login\"}"}}
{"seq":50,"ts":"…","kind":"item.end","id":"toolu_01","value":{"item":"tool","name":"Grep","args":{"q":"login"},"result":"auth.ts:42  if (expired) renew()"}}
{"seq":51,"ts":"…","kind":"item.start","id":"msg_b:0","item_type":"reply"}
{"seq":52,"ts":"…","kind":"item.delta","id":"msg_b:0","patch":{"kind":"text","text":"是 token "}}
{"seq":53,"ts":"…","kind":"item.delta","id":"msg_b:0","patch":{"kind":"text","text":"过期判断写反了。"}}
{"seq":54,"ts":"…","kind":"item.end","id":"msg_b:0","value":{"item":"reply","text":"是 token 过期判断写反了。"}}
{"seq":55,"ts":"…","kind":"turn.end","turn":"t7","stop_reason":"completed","usage":{"input":1840,"output":210,"context_window":42000}}

读法：id 把碎片归组——msg_a:0 的两条 text delta（seq 44/45）合成 thinking 全文；toolu_01 的两段 partial_json（48/49）拼成 {"q":"login"}，工具结果在 item.end（50）补上；msg_b:0 同理。每个项的 item.end 都已是合并全量。

休眠后回看历史：丢掉 turn.* / item.start / item.delta，每个 id 只取 item.end.value、连同 user 按 seq 排——同一条流立刻变成合并好的完整记录（即 session.aprog.jsonl 的样子），不打字：

user      为什么登录偶尔失败？
thinking  先翻登录相关代码
tool      Grep {"q":"login"} → auth.ts:42  if (expired) renew()
reply     是 token 过期判断写反了。
同一段 seq 41–55：实时 attach 时逐字播 delta；休眠后回看只取那 4 条 item.end 直接成文。原始流一字未删，两种呈现都从它派生。

再看子代理 + 上下文压缩 + 停止原因怎么落在同一条流里——主代理派一个 code-reviewer 子代理（Agent 工具调用 toolu_rev），子代理内部的项都带 parent:"toolu_rev"；中途引擎压了一次上下文；回合带停止原因结束：

{"seq":70,"ts":"…","kind":"item.start","id":"toolu_rev","item_type":"tool"}
{"seq":71,"ts":"…","kind":"item.end","id":"toolu_rev","value":{"item":"tool","name":"Agent","args":{"agent":"code-reviewer"},"result":"见子代理产出"}}
{"seq":72,"ts":"…","kind":"item.start","id":"msg_c:0","item_type":"thinking","parent":"toolu_rev"}
{"seq":73,"ts":"…","kind":"item.end","id":"msg_c:0","parent":"toolu_rev","value":{"item":"thinking","text":"先看 auth.ts"}}
{"seq":74,"ts":"…","kind":"item.start","id":"msg_c:1","item_type":"reply","parent":"toolu_rev"}
{"seq":75,"ts":"…","kind":"item.end","id":"msg_c:1","parent":"toolu_rev","value":{"item":"reply","text":"第 42 行 expired 判断写反了"}}
{"seq":76,"ts":"…","kind":"compaction","trigger":"auto","pre_tokens":152000}
{"seq":77,"ts":"…","kind":"turn.end","turn":"t8","stop_reason":"completed","usage":{"input":2010,"output":180}}
parent:"toolu_rev" 让前端把 seq 72–75 折成 code-reviewer 的子树（不摊平进主流）；compaction（seq 76）如实记录“引擎在此压了上下文”——流本身一字未删；turn.end.stop_reason 告诉用户回合为何结束。

断线恢复

客户端只记一个数：已渲染到的最后一条事件的 seq（游标）。重连时报上去，服务端在同一条订阅上：

前端 重连subscribe { from: 游标 seq }
↓
Driver 在一条流上依次回放
  • seq > 游标 的事件：落在已压实段→给折叠后的 item.end；落在尾部→给原始 delta
  • 紧接着续 live delta（同管交接，原子无缝，不漏不重）
↓
前端 按 id 合并delta 追加、item.end 定稿、快照整条替换——全按 id 幂等，多收也不出错
“落后多少”由 seq 差精确回答；那条正打到一半的事件，重放尾部 delta 自然补全——不需要单独的快照机制。投递 at-least-once，前端按 id 去重。
一个陷阱 resync 与 live 必须走同一条有序订阅：服务端先吐“游标后的事件”、紧接着吐 live delta。别拆成“HTTP 拿全量 + 另开 WS 收增量”两次调用——那中间会漏/重。

用户输入：echo-from-stream

用户回车后，前端不乐观地把消息画上去。它只是把输入提交给 Driver（经 DriverChannel）；Driver 接受后注入一个 user 事件到流（局部定序，control-plane 落库时盖 seq），再顺着流回到所有 viewer——前端这时才显示它。

前端回车 → 提交，不本地渲染
submit→
Driver接受 → 注入 user 事件(CP 盖 seq)
stream→
前端从流里收到 → 才显示
界面只有“流”这一个真相源。代价是用户输入看着略有延迟，但逻辑彻底统一：用户消息和引擎产出走同一条流、同一套 seq/id/恢复语义，多设备/多 viewer 天然一致。
