通道与协议

本篇定义跨线的全部类型化消息契约。所有契约集中于单一的包 @aprog/protocol，内分两组： 组① harness 事件——引擎中立的语义事件流，由 web、控制平面、driver 三方共用； 组② 通道通信——driver 与控制平面之间的全双工通道帧，由两方共用。 二者经子路径分别导出，且组②对组①零依赖。本篇逐组给出帧格式、传输形态、握手与信任流程， 并按 ✅ 已实现 / 🚧 部分实现 / ❌ 未实现 如实标注各环节状态。数据平面在体系中的定位见 01 · 总览。

标注约定 全文以 ✅ 已实现 / 🚧 部分实现 / ❌ 未实现 标注各环节状态；尚未落地的设计如实标注，不以「已实现」混淆。

1 · 两组协议:一个包,零耦合

@aprog/protocol 是单一的包，源码内分两组，经 exports 子路径分别导出，使用方可只引入所需的一组：

维度组① harness 事件组② 通道通信
子路径导出@aprog/protocol/harness@aprog/protocol/channel
参与方web、控制平面、driver 三方driver、控制平面两方
职责引擎中立的语义事件流（思考 / 工具 / 回复 / 用户输入…）driver↔控制平面通道的帧传输（握手、双向通信）
方向以事件上行、向 web 扇出为主双向全双工，一条连接承载两个方向
耦合关系纯值类型，不依赖组②对组①零依赖：harness 事件作为不透明载荷被通道携带，通道不 import、不解析其结构
实现状态类型已定义 ✅；事件流经通道落地 🚧 / ❌（见 §6）握手 ✅；其余帧待扩展 ❌（见 §5）

两组解耦的关键在于组②是纯传输契约。未来的事件流上行帧（待加）只是上行方向并集里的一个成员， 其载荷为一个不透明类型：driver 在其中放入一条 harness Event，控制平面落库时解读，而通道本身既不导入也不解析该结构。 于是两组各自独立演进——harness 事件的语义变化不触动通道传输层，反之亦然。

2 · 统一帧格式 Frame<t,p>

组②的每条通信帧采用统一格式 Frame<T extends string, P> = { t: T; p: P }。 字段 t 为类型判别标签（discriminator），接收方据其分派；载荷收于 p。

字段类型含义
tT extends string类型判别标签；一组方向并集以其区分各帧
pP该类型的专属载荷

扩展规则：新增一个通信帧 = 定义一个 Frame<'标签', { …载荷… }>， 并将其加入对应方向的并集（driver→控制平面加入 DriverFrame，反向加入 ControlPlaneFrame，见 §5）。 当前仅保留真正在用的握手帧；其余帧待实际用到时按此格式机械扩展，不预先铺设。

协议版本 CHANNEL_PROTOCOL_VERSION = 1 driver 将协议版本烘入镜像，控制平面独立部署，握手时比对（见 §4）。版本不一致时控制平面主动关闭连接， 以关闭码 1002 表示版本不匹配。

3 · 传输:WebSocket 通道

沙箱处于 NAT 之后，控制平面无法主动拨入，只有沙箱能向外拨出。因此连接必然由 driver 主动拨出一条全双工 WebSocket， 两个方向的帧都在这条连接上往返。帧以 NDJSON 编码——每条 WebSocket 消息为一个 JSON 对象。

维度取值
通道路径/v1/driver/channel
发起方driver（沙箱受 NAT 限制，只能主动拨出）
形态单条全双工 WebSocket，双向帧共用
帧编码NDJSON：逐帧 JSON.stringify / JSON.parse
控制平面端Bun.serve 原生 WebSocket（http.ts 为唯一 Bun.serve，升级判定后转交通道模块）
driver 端node 全局 WebSocket
URL 推导控制平面基址 http(s)://host/base → ws(s)://host/base/v1/driver/channel
带外大块不走本通道 OCI 闭包拉取与进程态 git clone 等大块传输不经此通道（见 02 · 程序与发版 与 03 · 镜像与沙箱）。待扩展 seed 帧时，通道只下发 URI 与短时凭证，真正的字节走带外。

4 · 握手与信任

driver 自启后的第一件事即拨向控制平面建立可信连接。它持 create 时注入的 bindToken， 经注入的控制平面地址拨出 WebSocket，发送 Hello 帧；控制平面据 bindToken 认领出连接所属的进程，回以 Welcome 帧。

信任凭证 bindToken 由控制平面网关在 create 沙箱时生成（crypto.randomUUID），凭证所有权归控制平面， 供应商仅负责注入。bindToken 每沙箱唯一，因此一张令牌同时充当能力证明与身份标识。

register-before-launch（登记早于启动）：供应商在「沙箱已起、driver 未启」之间回调 onProvisioned(sandboxId)， 控制平面在回调内将 bindToken → { pid, sandboxId } 登记进 DriverRegistry，await 其返回后供应商才启动 driver。 于是 driver 首拨即命中登记簿，杜绝「拨号早于登记」的竞态。 

sequenceDiagram
    autonumber
    participant GW as 控制平面网关
    participant REG as DriverRegistry
    participant SP as SandboxProvider
    participant D as driver
    participant CH as DriverChannelServer
    GW->>GW: bindToken = crypto.randomUUID()
    GW->>SP: create(image, res, {bindToken, onProvisioned})
    SP-->>GW: onProvisioned(sandboxId)（沙箱已起、driver 未启）
    GW->>REG: register(bindToken → {pid, sandboxId})
    Note over GW,REG: 登记早于 driver 启动
    SP->>D: 注入 bindToken + 控制平面地址，后台启动 driver
    D->>CH: WS connect /v1/driver/channel
    D->>CH: Hello {protocolVersion, bindToken}
    CH->>REG: resolve(bindToken)
    REG-->>CH: {pid, sandboxId}
    CH-->>D: Welcome {protocolVersion, pid, mode}
    Note over CH,D: 钉死连接到 pid；连接保持，留待后续双向帧

握手两帧的载荷字段如下：

握手过程中控制平面以 WebSocket 关闭码拒绝异常连接：

关闭码触发driver 应对
1002协议版本不匹配视为致命，由顶层兜底退出

Welcome.mode 现状 🚧 类型层面 mode 取值为 resume（瞬时重连）或 restore（冷唤醒）；当前实现恒回 restore， 两态分流尚未落地。resume 路径（driver 端缓冲重放、按已存 seq 去重）待事件流上行帧接入后补全。

5 · 通道帧:现状与扩展点

组②按方向分为两个并集：DriverFrame（driver 能发的全部帧）与 ControlPlaneFrame（控制平面能发的全部帧）。 当前各自仅含握手一帧；其余帧为规划中的扩展点，按 §2 的统一格式在真正用到时加入对应并集。

6 · 组① harness 事件流

每个进程对应一条 append-only 的事件流。底层引擎产出的每一片思考、工具调用、回复，以及用户的每次输入， 都是流上一个带顺序号的事件。这条流是唯一数据源：实时推送、断线恢复、持久存档统一由它承载。 引擎可替换，故本组协议引擎中立——各引擎私有消息由其对应的 driver 翻译为本组事件（适配细节见 Harness 适配）。

流上每个事件由两个标识定位：seq 为单调递增的流位置（控制平面落库时盖，跨生命周期全局单调，回答「落后多少 / 补哪些」）； id 为逻辑分组标识（driver 取引擎稳定标识赋予，把同一条消息的增量折叠成一条，回答「是不是同一条」）。 二者分工：seq 管重放到哪，id 管哪些折成一条。

事件信封

所有事件共有以下信封字段：

字段类型含义
seqnumber流位置，控制平面盖，全局单调
tsstringISO-8601 时间戳，driver 盖
parent?ItemId子代理归属：值为派生它的 Agent 工具调用 id（Claude 的 parent_tool_use_id）；主流事件为空
raw?unknown引擎原始载荷，归一前的 harness 原文。driver 在归一有损的事件上填，作保真兜底

事件种类

任何引擎的产出归一为以下几类（以 kind 判别）。一个「项（item）」= 一条思考 / 一条回复 / 一次工具调用 / 一次命令执行 / 一次文件改动，生命周期为 start → delta* → end：

kind含义关键字段
turn.start一个 agent 回合开始（用户提交后引擎开始执行）turn
user用户输入（完整、非流式；由 driver 回显进流）id、content
item.start一个项开始id、item_type
item.delta项的增量（实时「正在打字」）id、patch
item.end项完成，带 coalesced 全量id、value
turn.end回合结束turn、stop_reason、raw_stop_reason?、usage?
compaction引擎压缩了自身上下文（与事件流的合并无关，流永不删减）trigger、pre_tokens?
error错误 / 非致命提示severity、message
不做授权 产品形态自动批准引擎的全部操作，故协议中没有 allow / deny 事件——审批由 driver 自动应答、不进流。

项的两种形态:patch 与 value

item.delta.patch（增量载荷）与 item.end.value（合并后全量）是同一个项的两种形态，按项类型一一对应。 item.end.value 本身即合并全量，故看历史时无需自行折叠 delta：

项类型patch（增量载荷）value（合并后全量）
thinking{ kind:'text', text }{ item:'thinking', text, signature? }
reply{ kind:'text', text }{ item:'reply', text }
tool{ kind:'tool_args', partial_json }{ item:'tool', name, args, result? }
command{ kind:'command_output', chunk }{ item:'command', command, output, exit_code? }
file_change{ kind:'file_change', diff }{ item:'file_change', path, diff }

同一条流支持两种读法：实时 attach 时 tail 末尾，item.delta 即推、item.end 到达时定稿替换，呈现逐字「打字」； 看历史时丢弃 turn.* / item.start / item.delta，每个 id 仅取 item.end.value，连同 user 按 seq 排，即得合并后的完整记录，不打字。 原始流一字未删，两种呈现均由它派生。

停止原因 stop_reason

turn.end.stop_reason 归一为「用户不同反应」的小集——取值粒度等于用户会做出的不同动作的数量，不再细分；引擎原词存入 raw_stop_reason：

stop_reason用户的反应Claude 来源Codex 来源
completed正常说完，轮到用户end_turn / stop_sequenceTurnComplete
interrupted被打断或取代（中性）用户中断Interrupted / Replaced / ReviewEnded
limit可能没说完，是否继续max_tokens / error_max_turnsBudgetLimited
refused安全拒答，换个问法refusal—
error执行中出错error_during_executionError
保真兜底:typed 收敛, raw 携带 typed 字段只建模「用户该知道、且会据此行动」的内容，归一粒度等于用户不同反应的数量；其余引擎丰富度原样进 raw（携带而不建模）。 通用 UI 读 typed，高级检视器 / 调试 / 未来需求读 raw。用户的反应集有限且引擎中立，贴合它即同时贴合所有 harness。

断线恢复与用户输入

客户端只记一个数——已渲染到的最后一条事件的 seq（游标）。重连时以 Subscribe { from } 报上去， 服务端在同一条订阅上先回放 seq > from 的事件、紧接着续 live delta，原子交接、不漏不重。投递为 at-least-once，前端按 id 幂等合并： 

sequenceDiagram
    autonumber
    participant FE as 前端
    participant CP as 控制平面
    FE->>CP: Subscribe { from: 游标 seq }
    CP-->>FE: 回放 seq > from 的事件（已压实段给 item.end，尾部给原始 delta）
    CP-->>FE: 紧接续 live delta（同一订阅，原子交接）
    Note over FE: 按 id 幂等合并：delta 追加、item.end 定稿；多收去重

用户输入采用 echo-from-stream：前端回车后不本地乐观渲染，只把输入提交给 driver；driver 接受后注入一个 user 事件到流（控制平面落库盖 seq），再顺流回到所有 viewer——前端此时才显示它。 界面只有「流」这一个真相源，用户消息与引擎产出走同一套 seq / id / 恢复语义，多设备天然一致。

本组实现状态 🚧 / ❌ 组① 的全部事件类型已在 @aprog/protocol/harness 中定义（✅，类型层面），但当前仅从旧位置平移过来、内部结构待整理； 且这些事件尚未经通道落地——通道目前只完成握手，事件流上行帧、seq 盖章、断线重放均待 §5 的扩展点接入（❌）。 本节描述的读法与恢复语义为已定的设计目标，非当前运行态。
