三处存储的分界

为什么这样分：沙箱里的 harness 只有文件工具、没有 SQL，且跨引擎中立要求「文件即状态」；而用户/归属/进程索引是要查询的关系事实，归数据库。密钥不进任何快照。

🗄 SQLite 关系管理 用户、认证码、会话 token、程序目录镜像、安装记录（已落地）；进程 / 镜像注册（规划）。可查询，存指针 + 关系事实，不存重内容。
📁 文件系统 权威内容 /var/lib/aprog/skills/<name>/<ver>/（版本化技能包）、~/.aprog/<pid>/（进程目录，resume 权威）、事件流。
🔐 Vault 密钥 capability 用的 token/secret 值。DB 只存引用，spawn 时按授权注入沙箱，永不烘进镜像/快照。
DB 行里的 state 是控制平面驱动的生命周期态；程序内部的 phase / status 住在进程目录 meta.yml（resume 权威，冲突以目录为准），不进 DB。见 三层数据。

已实现：控制平面 SQLite 库表 现状

控制平面库为单 VM 嵌入式 SQLite（bun:sqlite，WAL），见 control-plane/src/db/index.ts。 目前共 5 张表：认证三件套（users / auth_codes / tokens） 与程序模型两表（programs / installations）。下面字段与索引均与建表 SQL 一一对应。

users —— 用户

注册只填 name + email，password_hash 留空、status = pending；邮箱验证 + 设密码后才置 active。自助注册，无需管理员预置。

字段类型说明
idTEXT PK唯一标识
nameTEXT用户名（登录用）
emailTEXT邮箱（登录 / 验证用）
password_hashTEXT?激活后才设；pending 期为空
statusTEXTpending → active
created_atTEXTISO 字符串
唯一性只约束 active 用户 name、email 大小写不敏感唯一，但用的是局部唯一索引 （WHERE status = 'active'）。未激活（pending）用户不占用标识符：注册了却忘记激活的人， 应能用同一用户名/邮箱重新注册——此时旧的 pending 行连同其 auth_codes / tokens 一并回收（见 users.createPending）。 
CREATE UNIQUE INDEX users_name_lower  ON users (lower(name))  WHERE status = 'active';
CREATE UNIQUE INDEX users_email_lower ON users (lower(email)) WHERE status = 'active';

auth_codes —— 邮箱验证 / 登录码

一张表承载两种短时凭据，靠 kind 区分：verify（邮箱验证用的长随机 token）与 login（6 位登录验证码）。均带 expires_at，用后/过期作废。

字段类型说明
idTEXT PK
user_idTEXTusers.id
kindTEXTverify（长随机）/ login（6 位短码）
codeTEXTtoken / 验证码本体
expires_at / created_atTEXTISO 字符串

索引：auth_codes_verify (kind, code)（验证 token 反查）、auth_codes_login (user_id, kind)（取某人最近的登录码）。

tokens —— 会话

登录成功后签发的不透明会话串，客户端持有。登出 / 过期即失效。一个用户可同时持多个（多设备）。

字段类型说明
tokenTEXT PK客户端持有的不透明串
user_idTEXTusers.id
created_at / expires_atTEXTISO 字符串

programs —— 程序目录（薄镜像）

一个智能程序（skill）的目录级元数据。权威态是磁盘上的 skill 注册表（SKILL.md frontmatter）， 这张表只是可查询的薄镜像：服务启动时由内置目录 INSERT OR REPLACE 播种，供 GET /programs 列出商店。图标不入库——前端按 id 从本地 registry 渲染。

字段类型说明
idTEXT PKslug = skill id（稳定标识）
nameTEXT展示名
versionTEXT?来自 SKILL.md
summaryTEXT?一句话简介
categoryTEXT?商店分组（规划与设计 / 开发与质量 / 文档…）
publisherTEXT?发布者
派生为主 因目录权威在磁盘，installations.program_id 只引用 slug，不向本表做强制外键； 本表丢了可随时从 skill 注册表重建。它是规划中 Skill / SkillVersion 模型的扁平先行版—— 目前一行只记一个 version 字符串，尚无不可变版本注册与依赖闭包锁。

installations —— 用户安装的程序

表达"用户 X 把程序 Y 装在了桌面上"。一个 (user, program) 至多一条。装即插行、卸即删行（硬删）。

CREATE TABLE installations (
  user_id    TEXT NOT NULL,   -- users.id
  program_id TEXT NOT NULL,   -- 程序 slug（= skill id）
  PRIMARY KEY (user_id, program_id)   -- 兼作唯一性 + “是否已装”查询
);
CREATE INDEX installations_user ON installations(user_id);
  • 列出我的程序 = WHERE user_id = ?（桌面图标据此渲染）；是否已装 = 主键命中。
  • 卸载只是从桌面移除：进程与数据在别处（进程目录），重装即恢复 → 行硬删即可，无需软删/历史，也无 installed_at。
为什么这么简单（设计取舍） 这套模型从前端实际交互反推、刻意压到最简：
  • 安装即在桌面，没有"桌面 / 状态栏"摆放管理 → 不含 surface / position 字段。
  • 任务栏 = 当前已打开的程序，是运行时 UI 态（开窗即现、关窗即消），不落库。
  • 系统应用不是智能程序：程序商店、设置只是前端 chrome（商店为桌面常驻入口），不进 programs / installations。
  • 无程序详情页：只需目录级元数据，更丰富的介绍按需读 SKILL.md，不在库里冗余。

规划中：运行时 / 版本 / 能力 未落地

以下尚未实现 本节是目标架构，描述程序运行时（Process / Snapshot）、不可变版本（SkillVersion）与能力授权（Capability）模型。 当前代码里都还没有对应表——程序模型先以扁平的 programs 占位。按既定优先级，运行时部分待沙箱接入后落地。

SkillVersion 规划中

一个技能的某个版本，发布后不可变，进程钉死的就是它。把现 programs.version 字符串升级成独立的版本注册：

字段类型说明
skill_name + versionstring + semver主键，如 design@0.4.0
descriptionstring简介（界面 ? 与程序商店展示）
depends_lockjson依赖闭包锁：解析后的库版本，如 {state@0.3.0, live-annotate@0.4.0}，保证闭包可复现
capabilitiesjson声明需要的 capability + scope（spawn 时据此授权）
store_pathpath/var/lib/aprog/skills/<name>/<version>/（只读副本，publish 时拷入）
源 → publish → 副本 repo 的 programs/ 是编写源；一次 publish 把目录拷成版本化只读副本并写 SkillVersion 行。 Control Plane 只从副本注入，不读可编辑的源——所以改程序发新版不影响钉旧版的进程。

Capability & CapabilityGrant 规划中

capability = 给 harness 调的工具（CLI / MCP），全量烘在沙箱镜像里；按 program 差异化在 spawn 时控制暴露面。详见 沙箱与镜像。

实体 / 字段说明
Capability（name, kind=cli|mcp, version）能力目录；binary 在镜像 /opt/aprog/bin
CapabilityGrant（user_id + skill_name → capability + scopes, secret_ref）谁的哪个程序获准用哪个 capability、什么 scope、用 vault 里哪个 secret。spawn 时据此注 PATH/MCP/token/egress

Process 已实现 · 沙箱 mock

后端已按此模型落地（沙箱动作 mock） 第三层 Process 的数据模型（PCB 表 + 状态机 + 操作副作用 + 事实源边界）已实现： PCB 进 processes 表、生命周期 spawn / ps / wake / hibernate / kill 走 control-plane/src/process。所有「碰沙箱」的动作（起停沙箱、检查点）经 SandboxGateway 收口，当前用 MockSandboxGateway 顶——只造假 sandbox_id，不起真实算力、不真 push。 运行时机制（沙箱编排、检查点存储 / git push、唤醒）的深入细节见 进程状态存储（Git 模型）、进程目录与状态、沙箱与镜像、生命周期流程。

Process = 一个 program 的一次运行，类比 OS 进程，平台的主 noun。版本在 spawn 时钉死、一生不变（升级是显式迁移）。 它有两处载体：控制平面的 PCB 行（可查询的关系事实）与磁盘上的进程目录 ~/.aprog/<pid>/（权威内容态）。 进程目录的状态如何存储 / 传输 / 唤醒取回（进程 = git 仓库、检查点 = commit）见 进程状态存储（Git 模型）。

① 进程控制块（PCB）—— processes 表

沿用现有 db/index.ts 风格（TEXT 主键、ISO 时间字符串）。刻意精简：只放需要查询/编排的字段，内容态一律留在进程目录。

字段类型说明
pidINTEGER PK进程 id，自增整数（类比 OS pid），如 1042
nameTEXT必填的进程名（spawn 时校验，不可空）；UI 列表显示「圆点 + 名字」
user_idTEXTowner，users.id
program_idTEXT钉死的程序 slug（= skill id）
program_versionTEXT钉死的版本。当前映射到扁平 programs.version；SkillVersion 落地后升级为 (skill_name, version) + 依赖闭包锁
stateTEXTspawned / running / hibernating（见状态机）
providerTEXT?沙箱厂商：daytona / e2b …（见 provider 抽象）；首次 attach 前为 null
sandbox_idTEXT?当前沙箱 ID；无沙箱（spawned / hibernating）时为 null
repo_urlTEXT进程 git 仓库的实际 clone URL（不含凭证），如 https://<cp>/aprog-proc-<pid>.git；spawn 建库时写入（见 寻址）
created_atTEXTspawn 时间（ISO）
last_active_atTEXT?末次活跃（驱动 UI uptime/最近、休眠回收策略）
与「保持精简」一致 无终止态、不设 deleted 列：进程跑完 / 被杀也只是回到 hibernating（释放沙箱、留检查点）——状态永不删除，进程目录与产物始终保留、随时可唤醒。 created_at / last_active_at 是功能字段（uptime、回收策略），非审计样板，故保留；不加 updated_at。
仓库：存地址、不存 commit 指针 进程 = 一个 git 仓库，名为 aprog-proc-<pid>（pid 仍是数字内部 PK；裸数字虽合法但不全局唯一、难读，故加前缀）。 仓库的实际地址存 repo_url——名带前缀 / 命名空间、外部 host 的 URL 由对方给定，真实地址不再是 pid 的纯函数，故 store, not derive。 但不存 commit 指针：「最新检查点是哪个 commit」= 仓库 HEAD，git ls-remote 可查、可推导故不入 PCB；spawned/hibernating 的区分由 state 列承载，默认 wake = git clone 自动检出 HEAD。详见 仓库创建与寻址。

② 状态机 —— 由「有没有关联沙箱」驱动

spawned 待起 刚 spawn，从未起过沙箱。无 sandbox_id，仓库还空（无 commit），只 eager 存了初始 input。不写骨架、不烧算力。
running 运行 sandbox_id 非空，有 live 沙箱与对话。检查点 commit 后持续 git push 到 CP 托管的 bare 仓库。
hibernating 休眠 休眠（含跑完 / 被杀）。sandbox_id=null，仓库 HEAD 即最新检查点。wake 即在新沙箱 git clone 取回（默认 HEAD）续跑。
状态只由「有没有关联沙箱」决定：有沙箱 = running；无沙箱 = spawned（刚创建、还没检查点）或 hibernating（跑过、留检查点）。 稳态在 running ⇄ hibernating 间往复。没有终止态——跑完 / 被杀都只是回到 hibernating，状态永不删除（见 进程目录与状态）。

③ 操作 → 字段副作用（命令动词如何改 PCB；机制细节见 生命周期流程）

操作前置 state后置 state对 PCB / 沙箱的副作用
spawn—spawned插入 PCB 行、钉死 program@version、eager 存初始 input；建空仓库 aprog-proc-<pid> 并写 repo_url（见 寻址）；sandbox_id=null，不起沙箱
attach / wakespawned / hibernatingrunningprovider.create 起新沙箱 → 回填 sandbox_id + provider；git clone --depth=1 + checkout（hibernating 取回该 commit；spawned 为空 / 初始 input）；拉起 harness
detachrunningrunning（不变）仅前端退订；沙箱与 harness 照常跑。属运行时 UI 态，不落库
hibernaterunninghibernating末次 commit + git push → provider.destroy；sandbox_id=null（仓库 HEAD 即最新检查点）
完成 / killrunninghibernating同 hibernate（尽力末次检查点 → provider.destroy、sandbox_id=null、记 last_active_at）；只是触发方不同（程序自身完成 / 用户强制），不产生新状态

④ 一个进程的事实源：DB vs 进程目录

东西放哪权威性
PCB（pid / owner / 钉死版本 / state / sandbox_id / repo_url…）DB processes关系元数据权威；state 为控制平面驱动的生命周期态
meta.yml（phase / input_cursor / status）进程目录resume 权威；phase/status 是程序内部 FSM 态，不进 PCB
input.md（append-only 输入流）进程目录权威意图流
execution-state/（state.yaml + by-ref blob）进程目录有 schema 的执行状态，权威
artifacts/（程序产物）进程目录权威
session.aprog.jsonl（会话逐字流）进程目录不权威、不进快照——读时投影，仅展示/审计
检查点（commit；运行中持续 git push）进程 git 仓库（bare，CP 托管 /srv/aprog-repos/aprog-proc-<pid>.git）跨厂商耐久真相；最新检查点 = 仓库 HEAD（不在 PCB 存指针，见 Git 模型）
沙箱（临时算力，工作树）厂商侧（沙箱临时盘）易失，hibernate / kill 即销毁；wake 由 git clone 重建
已定（本轮评审） 无 exited 终止态：状态只由「有没有关联沙箱」决定——有沙箱 = running，无沙箱 = spawned（刚创建）/ hibernating（休眠）；跑完 / kill 都只是回到 hibernating，状态永不删除。
