状态与持久化

数据模型

Control Plane 管理平台要存三类数据，存储介质按一条铁律分开：关系管理数据进数据库（SQLite）、 进程工作态与技能包进文件系统、密钥进 vault（库里只存引用）。 这页讲实体、字段与事实源归属；进程目录内部结构见 进程目录与状态，镜像/沙箱见 沙箱与镜像。

三处存储的分界

为什么这样分：沙箱里的 harness 只有文件工具、没有 SQL，且跨引擎中立要求「文件即状态」；而用户/归属/进程索引是要查询的关系事实，归数据库。密钥不进任何快照。

🗄 SQLite 关系管理 User / AuthSession / Skill / SkillVersion / Installation / CapabilityGrant / Process / Snapshot。可查询，存指针 + 关系事实，不存重内容。
📁 文件系统 权威内容 /var/lib/aprog/skills/<name>/<ver>/（版本化技能包）、/var/lib/aprog/proc/<pid>/（进程目录快照）、事件流。
🔐 Vault 密钥 capability 用的 token/secret 值。DB 只存引用，spawn 时按授权注入沙箱，永不烘进镜像/快照。
DB 行里的 state/phase 是快查投影；进程目录的 meta.yml 才是 resume 权威（冲突以目录为准）。见 三层数据。

实体（SQLite）

User

人类用户，现阶段由管理员 out-of-band 创建（无 self-signup）。

字段类型说明
idstring唯一标识
usernamestring唯一，登录用
password_hashstringargon2 / bcrypt
created_at / last_login_attimestamp

AuthSession

登录创建、登出失效。一个 User 可同时持有多个（多设备各一个）。

字段类型说明
id (= token)string客户端持有的不透明串
user_idstringUser.id
created_at / last_used_at / expires_attimestamp默认 TTL 30 天
revoked_attimestamp?登出时填

验证：revoked_at IS NULL AND expires_at > now()，否则 401。

Skill 程序 / 库

一个技能包的身份。程序(application) 和库(library) 都是 Skill，靠 kind 区分——对底层 harness 它们都是 skill，「程序/库」是平台层角色（见 运行全景）。

字段类型说明
name (= id)string全局唯一的技能名，如 design。即 harness 里的 skill 名，必须唯一——故直接作 id，不加命名空间。publish 时强制唯一
kindenumapplication（可 spawn）/ library（只被依赖）
sourceenumbuiltin（源在 repo）/ user（用户上传）。同形，仅来源不同
owner_idstring?作者；builtin 为平台
latest_versionsemver指向最新的 SkillVersion

SkillVersion 不可变

一个技能的某个版本，发布后不可变。进程钉死的就是它。描述/依赖随版本走。

字段类型说明
skill_name + versionstring + semver主键，如 design@0.4.0
descriptionstring简介（界面 ? 与程序商店展示）
depends_lockjson依赖闭包锁：解析后的库版本，如 {state@0.3.0, live-annotate@0.4.0}。保证闭包可复现
capabilitiesjson声明需要的 capability + scope（spawn 时据此授权）
store_pathpath/var/lib/aprog/skills/<name>/<version>/（只读副本，publish 时拷入）
源 → publish → 副本 repo 的 programs/ 是编写源；一次 publish 把目录拷成版本化只读副本并写 SkillVersion 行。 Control Plane 只从副本注入，不读可编辑的源——所以改程序发新版不影响钉旧版的进程。

Installation 安装 / 归属

用户把某 Skill 装到自己桌面。安装 = 桌面整理：装则上桌面/任务栏，卸载只删此行、进程与数据全留，重装即恢复。库不单独安装，随程序依赖闭包透明带入。

字段类型说明
user_id + skill_namestring主键
pinned_versionsemver?留空=跟 latest；填了=锁定某版本启动
installed_attimestamp

Capability & CapabilityGrant

capability = 给 harness 调的工具（CLI / MCP），全量烘在沙箱镜像里；按 program 差异化在 spawn 时控制暴露面。详见 沙箱与镜像。

实体 / 字段说明
Capability（name, kind=cli|mcp, version）能力目录；binary 在镜像 /opt/aprog/bin
CapabilityGrant（user_id + skill_name → capability + scopes, secret_ref）谁的哪个程序获准用哪个 capability、什么 scope、用 vault 里哪个 secret。spawn 时据此注 PATH/MCP/token/egress

Process

一次 program 执行的实例，主 noun。版本在 spawn 时钉死，一生不变（升级是显式迁移）。

字段类型说明
pidstringuuid 取前几位，如 a3f9c2d1
user_idstringowner
skill_namestring钉死的程序名，如 design
skill_versionsemver钉死的版本；依赖闭包由该 SkillVersion 的 depends_lock 推出（不同进程可钉不同版本）
stateenumrunning（有沙箱）/ hibernating（无沙箱）
providerenum沙箱厂商：daytona / e2b …（见 provider 抽象）
sandbox_idstring?当前沙箱 ID，无沙箱时 null
checkpoint_refstring?最新检查点的引用（控制平面侧存储路径）；唤醒时灌回新沙箱（见 检查点）
created_at / last_active_attimestamp

事实源：DB（关系元数据）+ snapshot_ref / 沙箱里的进程目录（内容，resume 权威）。

Snapshot 镜像注册

沙箱镜像（capability + 引擎 + driver 烘成）。一个镜像起 N 个沙箱。Daytona 等无内置不可变 tag，故自算 content-hash 作名。

字段说明
id (= content_hash)aprog-sandbox:<sha12>，capability 集合的内容哈希
provider + provider_ref厂商及其侧的 snapshot 名/id
resourcescpu/mem/disk（Daytona 烘进 snapshot）
statusbuilding / active / failed

关系一览

User──< Installation >──Skill桌面装了哪些程序
Skill──< has >──SkillVersion不可变版本 + depends_lock
User×Skill──< CapabilityGrant >──Capability每程序的能力授权 + scope
User──< runs >──Process── 钉 →SkillVersion
Process── 跑在 →Snapshot某镜像起的沙箱

两个状态

Process 只有两个状态，由「有没有沙箱」驱动：

running 运行 sandbox_id 非空，有 live 对话
hibernate ⇄ wake⇄
hibernating 休眠 sandbox_id=null，留 snapshot_ref
休眠态：交互区收起、目录仍可浏览（见 工作窗口）。状态永不删除——程序跑完也只是处于休眠、随时可再唤醒。

检查点与事实源

进程状态两种形态：沙箱里的工作副本、控制平面上的检查点。同一时刻只有一份权威—— 沙箱在跑时是工作副本；沙箱不在时（休眠=destroy）是控制平面检查点（运行中经 DriverChannel 持续流回）。状态穿越沙箱生死，永不删除。

不变量 引擎私有的 session.jsonl 不进快照、不权威，resume 靠进程目录的状态文件而非对话。 详见 三层数据 与 唤醒靠状态不靠对话。
