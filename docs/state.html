状态契约

进程的权威状态不依赖对话上下文，而记于进程目录——一套由 state 程序库定义的、有 schema 的状态契约。 本篇定义进程目录的承载形态（git 仓库）、三层数据的权威边界、execution 状态契约的文件结构与 resume 语义、 检查点的捕获时机与一致性职责，以及分支、rewind 与数据库投影。 运行编排见 05 · 运行生命周期，程序分发见 02 · 程序与发版。

标注约定 全文以 ✅ 已实现 / 🚧 部分实现 / ❌ 未实现 标注各环节状态。须区分两类断言：协议 / 契约设计（已定）与运行时落地（多在 driver 侧待接入），后文如实标注。

1 · 进程目录与权威状态

每个进程在 per-user 根目录 ~/.aprog/ 下拥有一个进程目录 ~/.aprog/<id>/（类比 /proc/<pid>/）。 它就是底层 harness 的工作目录（cwd），也是平台唯一关心、用户界面所见的进程视图。 核心断言：进程目录就是进程的权威状态——沙箱可销毁、对话可丢弃，唯独这个目录是事实源。 为支持分支与 rewind，进程目录以一个 git 仓库承载：一个进程 = 一个仓库，一个检查点 = 一次 commit。

维度约定说明
根目录~/.aprog/per-user；其下每个子目录是一个进程
进程目录~/.aprog/<id>/权威状态；同时是 harness 的 cwd 与 git 工作树
标识<yyMMdd>-<nanoid>自动生成、实践无碰撞、可按 ls -lt 排序；创建时必须呈现给用户，否则无法日后 resume
承载git 仓库进程目录即一个被版本化的仓库；检查点 = commit，分支 = branch，rewind = checkout
耐久git push / git clone不挂持久卷，活在沙箱临时盘；耐久靠检查点 push 到远端，唤醒靠 clone 灌回
内容meta.yml / input.md / execution-state/由 state 库定义的状态契约，见 §3
两套标识，各司其职 进程目录的 id 由 state 协议定义为 <yyMMdd>-<nanoid>（per-execution）。控制平面的 PCB 另以整型 pid 索引进程（投影侧，见 §6）；二者分属契约层与关系投影层，互不替代。

须区分两类实体并行存在：程序为共享、不可变、按版本分发，经 OCI 制品拉取（见 02）； 进程态为每进程独占、可变、可分支，以 git 仓库承载。二者在唤醒时并行下载落入沙箱，职责互不重叠。

2 · 三层数据及权威性

同一段“执行”涉及三类数据，职责严格分开，不得混为一谈：

层内容 / 载体权威性承载 / 分发参与 resume
① 程序 Programskill 目录（解包至 skills/）共享、不可变、按版本OCI 制品按版本拉取提供执行逻辑（非状态）
② 进程态 Process statemeta.yml / input.md / execution-state/每进程独占、可变、可分支 · 唯一权威git 仓库（commit / push）✅ 权威
③ 会话事件流 Session eventssession.jsonl（harness 产生的逐字事件）仅展示 / 审计，非权威事件流（投影）❌
唤醒靠状态，绝不反向重建对话 休眠进程被唤醒时分到全新沙箱、引擎冷启动、空上下文。它靠 resume 协议读状态文件判断“做到哪了”， 绝不把历史对话翻译回引擎重新喂入来“恢复记忆”。所有要紧的东西（阶段、游标、决策、产物）都已被 state 捕获进目录——对话因此可丢弃。 这也是换引擎（Claude → 其他）不丢工作的原因：状态在目录里，新引擎冷启动后照样从状态接着干。
引擎私有缓存不在三层之内 引擎 SDK 自身的会话缓存（沙箱内的私有上下文）是沙箱生命周期内的临时物，永不权威、不进检查点，沙箱一销毁即没。它既非进程态，也不是 aprog 拥有的会话事件流。

3 · execution 状态契约

state 程序库定义 execution 的文件契约与 resume 语义。一个 execution 是“一次运行”的命名作用域： 程序引用 + 已收输入 + 累积的 execution-state + 生命周期元数据——它不是运行的结果，而是运行发生于其中、可经 ID 寻址、跨中断持久的工作区。目录三件：

路径角色OS 类比
meta.yml协议标准字段 + state_schema 快照PCB（标准化外部接口）
input.mdappend-only 用户输入流输入队列
execution-state/程序自管的工作状态私有进程内存（程序管理）

meta.yml · 标准字段

协议标准字段在前；程序可在其下追加自有字段，只懂协议的工具读标准字段、忽略其余。

字段取值说明
id<yyMMdd>-<nanoid>execution 标识
program名 / 路径被执行物（典型为一份 skill 文档）
program_version版本复制程序文档的版本字段；无则省略
statusrunning / completed / failed生命周期状态
phase自由文本程序自定义 FSM 的当前阶段；必须写于此，使 resume 不必先读 execution-state/
created_at / last_active_atISO8601时间戳
input_cursorinput-NNN最近一条已完全处理的输入 ID；无则省略——驱动 resume
state_schema有序列表execution 自有的 schema 快照，见 §3 · state_schema

input.md · 追加式输入流

单文件、append-only，原始历史绝不重写。它只存用户对程序说的话，是驱动 FSM 的权威意图流；程序内部的问答、决策、中间产物一律进 execution-state/。每条输入的结构：

元素格式约定
条目 IDinput-NNN有序递增，是 input_cursor 指向的对象
时间戳ISO8601每条必带
标题可选短标题—
分隔符---条目间以 Markdown 水平线分隔；不识别其他分隔符
正文Markdown仅含用户输入
input.md ≠ 会话事件流 input.md 只存用户输入，驱动执行；会话事件流是含思考 / 工具 / 回复的完整转录，驱动界面。用户消息在两处都出现，但服务对象不同。

execution-state/ · 有 schema 的执行状态

程序为本次 execution 累积的工作状态。一个 YAML 索引文件统领，大块内容经 by-ref 指针外置为 blob：

组成内容
state.yaml唯一 KV 索引：每个 schema 条目对应 YAML 树的一个叶子路径（点号分隔，如 vision.tagline）。读一次即知程序当前所有持有
<子路径> blob经 { $file: <相对路径> } 从 state.yaml 引用的大块内容；必须留在 execution-state/ 内，禁止 ../ 逃逸

每个叶子的值有三种存储模式：

storage值的表示附加字段
inlineYAML 标量 / 列表 / 映射，直存 state.yaml—
by-ref{ $file: <path> }，内容在 blobcontent_type（markdown / yaml / json / html / text / binary / …）
map内层键遵循 key_pattern 的映射，每个内层值再按内层存储约定key_pattern + value_storage（+ value_content_type）
原子性与生命周期 state.yaml 写入必须原子（写临时文件再 rename() 覆盖），读者永不见半成品；blob 同此模式（大二进制流式写入时以 .partial 暂存、完成再 rename）。 从 state.yaml 删除某键时，必须同时删除其引用的 blob（不留孤儿）。

state_schema · 声明的形状

state_schema 是 meta.yml 下的有序列表，声明 execution-state/ 的形状。它是 execution 的自有副本： 创建时从程序 SKILL.md 的基线 schema 快照复制而来，此后冻结——程序日后即便发布新基线，本 execution 仍沿用其冻结副本（默认不迁移运行中的 execution）。 基线由程序自身的声明与各 depends_on 库的 schema 合并而成（同名键来自不同来源 = 创建期硬错误，须改名再试）。 运行期可向列表追加新条目以扩展；强制：state.yaml 中每个叶子都须对应一个 schema 条目（“先声明、再写入”）。

字段必填含义
key是state.yaml 树的点号路径（如 vision.tagline）；段内含字面点须引号包裹
storage是inline / by-ref / map
content_typeby-ref 时被引 blob 的格式
key_patternmap 时内层键模式（如 <mod-id>，仅文档、不强制正则）
value_storagemap 时内层值的存储：inline / by-ref
value_content_typemap 且内层 by-ref 时内层 blob 的格式
description是此键的用途

resume · 游标与续跑

resume 读状态而非对话。续跑流程严格定义如下：

  • 读 meta.yml 的 phase 与 input_cursor；
  • 取 input.md 中位于游标之后的条目，即未处理输入；
  • 读 execution-state/state.yaml 及其 by-ref blob，恢复工作状态；
  • 继续当前阶段。仅在一条输入被完全消费后（已反映进 execution-state/）才推进游标。
at-least-once，不是 exactly-once 协议保证 at-least-once 投递：若 agent 在作用于某输入后、推进游标前死亡，下次 resume 会再见同一条输入。 重处理安全（幂等）是程序自身的责任，非协议责任——程序应在 execution-state/ 留痕，借以识别“已开始未完成”并跳过已应用项。

4 · 检查点

沙箱临时盘随时可能消失（厂商回收、OOM、崩溃、网络分区）。让状态扛住，靠检查点——把进程目录的 state 子集 commit 进 git 并推到远端。 但“什么时候算一个能安全 resume 的点”，平台判断不了，只有上层程序知道。职责据此三分：

关注点归谁具体
状态的内容与结构程序 / state 库meta / input / execution-state 如何组织、自身逻辑如何 resume
状态的一致性（何时算干净）程序在语义一致点主动调检查点工具 mark 一个 commit；平台无从判断
状态的耐久性平台 / driver如何传、如何恢复、多久真同步一次（debounce）；对内容无知

两种「一致」

把“一致”拆开，才看清为何必须由程序裁定：

层级含义谁能判断
无撕裂（tear-free）未在写盘中途、盘面原子（写临时文件再 rename）平台 / driver 可观测（agent 空闲那刻）
语义一致（coherent）execution-state/ 此刻是“进展到哪”的完整忠实描述，从此恢复能正确续只有程序知道
turn.end 只是无撕裂，不等于语义一致 程序的 state 更新常跨多个 turn（一个逻辑操作写到一半就让出回合）。仅以 turn.end 为时机的快照虽无撕裂，却可能落在逻辑中途，是语义半成品。所以判定一致的权威必须是程序，而非平台的猜测。

mark 武装 + turn.end 捕获

检查点被做成一个工具（平台赋予 harness 的一种操作，类比数据库 COMMIT）：程序在语义一致处主动调一次 = “我现在干净，这是合法续点”。 但调用 ≠ 检查点：此刻引擎还在跑、盘还在被写，去拍快照会与正在进行的写竞态、撕裂。所以这个调用只是 mark / 意向，driver 把它挂起， 等 harness 把本轮回复说完（turn.end、转入等待输入）、盘面 settled，才真正捕获——这才同时拿到语义一致（程序 mark）+ 无撕裂（回合结束）。 

sequenceDiagram
    autonumber
    participant P as 程序（经引擎）
    participant D as driver
    participant G as git 远端
    P->>D: 语义干净点 → 调检查点工具（mark：武装 / 意向）
    Note over D: 挂起 mark，盯住事件流
    P->>D: turn.end（回复说完、转等输入），盘面 settled
    alt mark 自武装以来未被写脏
        D->>D: git add -A && git commit（新 HEAD = 最新检查点）
        D->>G: 按 debounce → git push（耐久）
    else mark 后又调过修改性工具（安全网）
        Note over D: 干净点被改脏 → 本次不捕获，等下一个干净 mark
    end
安全网：mark 后又乱写则作废 driver 盯 mark 到 turn.end 之间：若 agent 又调了修改性工具（write / edit / 写盘的 bash，区别于只读的 read / search / list），说明被标记的干净点之后又被改脏，这次不算检查点，等下一个干净 mark。 这逼出程序纪律：写在前、mark 在后、紧接着让出回合。

debounce 是一条耐久性 ↔ 同步成本的滑杆——mark 只决定“何时算一致”，多久真 push 由 driver 按策略定：

策略同步成本崩溃丢失窗口
每个 mark 都 push高最小（最多回退一个 mark）
每 N 个 mark / 每 T 分钟中中
仅休眠时 push最低最大（整段运行）
liveness 兜底 程序若从不 mark，崩溃就全丢回 bootstrap。基线是程序责任（state 库在 loop 纪律里要求周期性 mark）；平台可选在太久没干净 mark 时，于某个 turn.end 拍一张无撕裂快照作粗兜底，但干净点永远来自程序的 mark。
休眠 = 强制落最近 mark + 让出沙箱 进程空闲等人，恰好也是最该让出沙箱的时刻。休眠在机制上即“确保最新 mark 落地 + destroy 沙箱”；控制平面也可在分享 / 迁移前主动强制一次检查点。
耐久传输：git push / clone 一个检查点 = 一次 commit，最新检查点 = 仓库 HEAD（不在 PCB 存指针）。耐久靠 git push 到控制平面托管的远端（bare 仓库），唤醒靠 git clone --depth=1（默认检出 HEAD）灌回新沙箱——天然跨厂商。 前端只读已提交的一致快照（经 CP 代理按 ref 读 git），尚未提交的撕裂态不给用户看；驱动 push / clone 的是 CP 现签、钉死单库、短时有效、最小权限的令牌，真凭证不出控制平面。

外部副作用边界

检查点能把进程倒带到上一个 mark，但倒带不了已逃出进程目录的外部副作用——典型是向远端项目 git push、发邮件、调外部 API：它们立刻、不可逆地耐久了，平台 rewind 不了。 这个双写窗口结构上消不掉（换顺序只把缝挪到另一边），平台层没有更好的办法。真正的一致由程序语义兜，两条纪律：

  • 记游标：副作用完成后把其标识（如远端 commit hash）记进 execution-state/，随检查点原子持久化——这就是 git 版的 input_cursor。
  • 恢复签出记录的不可变 hash，而非分支 tip：分支 tip 是可变外部引用（可能被逃逸的 push 或协作者推动）；签出记录的不可变 hash，工作树与进程状态就都落在同一点、彼此自洽，堵死“工作树静默超前于我的记忆”这种最坏腐坏。
远端孤儿无害——只关心内部是否自洽 只要程序独占该分支（agent 自己的 WIP 分支），恢复后重做出的新 commit 经 push --force 盖掉逃逸的孤儿，孤儿变 unreferenced 被 GC。正确性底线是恢复后内部自洽（记游标 + 签出 hash 已保证）；“恢复即对账、跳过重做”只是省重复劳动的可选优化。唯一会真咬人的是共享分支且他人已 pull，超出独占 WIP 分支的场景。

于是恢复 = 纯 resume，没有回退语义要设计：harness 启动后就是普通 resume，不知道也不关心“丢了一段活”。 mark 到崩溃前那几条输入由控制平面 eager 持久化（输入很小、一到就存，是耐久脊梁），恢复后当普通新输入重投给正在等输入的 harness——无重放、无补偿、无回退事件。 两类状态两种节奏互补：输入（轻、脊梁）发出即存；执行状态 / 会话（重）在程序 mark 处按 debounce 检查点。状态永不删除，退出只是没有沙箱关联，目录与历史完整保留。

5 · 分支与 rewind

检查点是一串带名字的存档点（mark 时可带 label，像 git commit -m）。于是进程目录天然就是控制平面上一个被版本化的仓库： 从某个旧存档点岔一条分支去试别的方向，进程历史就从“线”变成“树”。底层即给进程目录做 git 版本管理，UI 上不暴露任何 git 术语。

进程概念git 对应
一个检查点 mark一次 commit（label = commit message）
分支（从存档点岔开试新方向）branch
倒带到某存档点checkout / reset
唤醒checkout 分支 tip 灌进沙箱
运行中沙箱里的活态dirty working tree（崩溃丢这截、回到上一 commit）
 
flowchart LR
    A["commit · mark①"] --> B["commit · mark②"]
    B --> M[("主线 tip（唤醒检出此处）")]
    A -. "从 mark① 分支（试新走法）" .-> X["commit · 支线"]
    X --> N[("支线 tip（独立可运行 / 可并行）")]
    B -. "rewind = checkout 旧 mark" .-> A
两棵耦合的版本树：superproject ↔ submodule 注意有两棵独立的 git：① 进程目录——平台在控制平面上版本化（外层）；② 项目代码——程序在项目自己的远端版本化（内层）。 二者靠记录的 commit hash 游标钉在一起：外层每个 commit 里存着一个指向内层的 SHA——正是 superproject 钉住 submodule 的形状。 于是平台 fork 外层（branch 进程目录）免费、纯在 CP 上；要让内层也分叉，得靠程序在 resume 时把项目 repo 从那个 SHA 另开一条分支（平台对 git 无知，只传一个“你是 fork”信号）。

运行单位是“分支”，不是进程：列表里每条分支是一条独立可 attach / 可运行 / 可分享的执行线，可并行；“进程”退化成把同家族分支兜在一起的分组。 主分支没有自己的名字——它就是这个进程；只有支线在创建时起名。

只 branch，不 merge 外层机械上能 merge，但“合并两条 agent 跑出来的状态 + 两条代码分支”语义上不成立。分支用于分支尝试（同一起点试不同走法），不用于最终汇合。

6 · 数据库：投影

内容真相在 OCI（程序）与 git（进程态）；控制平面数据库是可查询的投影与平台独有的关系态，非权威内容来源：

表定位真相在哪
processes（PCB）进程控制块：pid / owner / 钉死 program@版本 / state / sandbox_id / repo_url …进程态在 git；PCB 是关系投影
programs / program_versions程序身份 + 版本索引（digest、镜像依赖）OCI digest + git 源
installations用户 × 程序的安装关系DB 权威（平台独有关系态）
哪些内容态不入 PCB
  • meta.yml 的 phase / status / input_cursor 是程序内部 FSM 态，resume 权威在进程目录，不进 PCB（冲突以目录为准）。
  • “最新检查点是哪个 commit” = 仓库 HEAD，可推导，不入 PCB；spawned / hibernating 的区分由 state 列承载。
  • 进程跑在哪个镜像，由钉死的 program@版本 join program_versions 派生，不入 PCB。

7 · 实现状态
