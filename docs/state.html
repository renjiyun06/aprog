进程目录与状态

一个核心断言：进程的状态不依赖对话上下文。进程“做到哪了”记录在它的目录里—— 一套由 state 程序库定义的、有 schema 的状态文件。沙箱可销毁、对话可丢，唯独这个目录是事实源。

进程目录 = 真正的状态

每个进程是 ~/.aprog/<pid>/ 下的一个目录（类比 /proc/<pid>/）。它就是底层 harness 的工作目录（cwd）—— 这点很重要：平台只关心这个目录里的东西，用户在界面上看到的进程目录也正是它。（harness 并不被禁止访问别处，只是别处平台不管。） 它不挂持久卷，活在沙箱本地盘上；耐久性靠检查点经 B 平面流回控制平面，而非靠卷。 目录结构由 state 程序库统一定义：身份、输入流、有 schema 的执行状态、以及 resume 语义。

唤醒靠状态，不靠对话

休眠进程被唤醒时，会分到一个全新沙箱、引擎冷启动、空上下文。它怎么知道“做到哪了”？ 靠 resume 协议读状态文件：读 meta.yml 的 phase 和 input_cursor、读游标之后的 input.md 输入、 读 execution-state/state.yaml 及其 by-ref blob，然后重跑当前阶段的 on-entry。

绝不反向重建引擎上下文 aprog 不会把历史对话翻译回引擎、重新喂进去来“恢复记忆”。因为进程状态根本不依赖对话—— 所有要紧的东西（阶段、游标、决策、产物）都已被 state 捕获进目录。对话因此是可丢弃的。

这也是为什么换引擎（Claude → Codex）不丢工作：会话归 aprog 所有、状态在目录里，新引擎冷启动后照样能从状态接着干。 代价只是它读不到上一个引擎的私有对话缓存——而这本来就不需要。

三层数据：谁权威、谁进快照

同一段“执行”涉及三类数据，职责严格分开，别混为一谈：

层文件谁写进快照参与 resume用途
进程状态meta.yml / input.md / execution-state/程序（经引擎）✅✅ 权威让进程接着干
会话记录session.aprog.jsonlDriver✅（在进程目录里）❌前端历史 + 审计
引擎缓存session.jsonl（SDK 私有）引擎❌❌沙箱内临时上下文，可丢
  • 进程状态是唯一权威，驱动 resume；它和会话记录都在进程目录里，跟着进程持久化。
  • 会话记录是 engine-neutral 的逐字转录，只为展示和审计，不参与 resume。
  • 引擎缓存（session.jsonl）是沙箱生命周期内的临时物，沙箱一销毁就没了，永不权威、不进 aprog 快照。
input.md ≠ 会话记录 input.md 只存用户输入，是 state 协议里驱动 FSM 的权威意图流（input_cursor 标记处理到哪）； 会话记录是含思考/工具/回复的完整转录。用户消息在两处都出现，但服务的对象不同——前者驱动执行，后者驱动界面。

检查点：耐久性是平台的事，不是程序的事

沙箱本地盘随时可能没（厂商回收、OOM、崩溃、网络分区）。让状态扛住这个，靠的是检查点——把进程目录的 state 子集 经 B 平面流回控制平面。关键在职责分层：

关注点归谁具体
状态的内容与结构程序 / state 库meta / input / execution-state 怎么组织、程序自己逻辑怎么 resume
状态的耐久性平台 / driver何时快照、怎么传、怎么恢复。对内容无知
检查点不归程序控制 上层程序无法、也不该决定何时做检查点。它是平台机制，由 driver 用 harness-specific 的手段实现—— 比如 Claude Agent SDK 在回复结束时的 hook，driver 接住后按配置决定要不要同步（这只是一种方式）。程序不参与，也不需要「为检查点而 flush」。

检查点落在 quiescent 点

自然的保存点是 quiescent 点：harness 已回复完、正在等用户下一条输入。此刻 agent 不在写盘，加上 state 库原子落盘（写临时文件再 rename）， 盘面自然是个一致、可 resume 的点。它是「每个回合结束」的子集（多了「之后转入空闲」这个条件），因此比每回合都存稀疏。

quiescent 点 = 自然的休眠点 进程空闲等人，恰好也是最该让出沙箱的时刻。于是「做检查点」和「hibernate-prepare」在同一时刻发生—— 休眠在机制上就是「在 quiescent 点做检查点 + 顺手让出沙箱」，两者天然合一。

节奏是配置项

检查点频率是一条耐久性 ↔ 同步成本的滑杆，平台配置：

配置同步成本崩溃丢失窗口
每个 quiescent 点高最小（最多一组对话的算力）
每 N 个 quiescent / 每 T 分钟中中
仅显式休眠时最低最大（整段运行的算力）

恢复 = 纯 resume，没有「回退语义」

沙箱 / driver 死了，我们不抢救——直接把进程恢复到上一个健康检查点（新沙箱热恢复 / 经 B 灌回检查点）。这里没有任何回退语义要设计：

  • 底层 harness 启动后就是普通 resume，它不知道、也不关心「丢了一段活」。
  • 检查点之后到崩溃前那几条输入，控制平面 eager 持久化着（输入很小、一到就存，是耐久脊梁）；恢复后当普通新输入重投给那个正在等输入的 harness，它当第一次收到来处理。没有重放逻辑、没有补偿、没有回退事件。
  • 唯一代价：丢失那段活若产生过系统外副作用（发邮件、写外部库），重跑会重做。这明确划在范围外——aprog 只管系统内状态，不管系统外。

于是两类状态用两种节奏，互补：输入（轻、是脊梁）发出即存；执行状态 / 会话（重）在 quiescent 点按配置检查点。 状态永不删除——退出只是没有沙箱关联，目录与历史完整保留。具体的 spawn / hibernate / wake / exit 调用链见 生命周期流程。
