进程目录与状态

一个核心断言：进程的状态不依赖对话上下文。进程“做到哪了”记录在它的目录里—— 一套由 state 程序库定义的、有 schema 的状态文件。沙箱可销毁、对话可丢，唯独这个目录是事实源。

进程目录 = 真正的状态

每个进程是 ~/.aprog/<pid>/ 下的一个目录（类比 /proc/<pid>/）。它就是底层 harness 的工作目录（cwd）—— 这点很重要：平台只关心这个目录里的东西，用户在界面上看到的进程目录也正是它。（harness 并不被禁止访问别处，只是别处平台不管。） 它不挂持久卷，活在沙箱本地盘上；耐久性靠检查点经 B 平面流回控制平面，而非靠卷。 目录结构由 state 程序库统一定义：身份、输入流、有 schema 的执行状态、以及 resume 语义。

唤醒靠状态，不靠对话

休眠进程被唤醒时，会分到一个全新沙箱、引擎冷启动、空上下文。它怎么知道“做到哪了”？ 靠 resume 协议读状态文件：读 meta.yml 的 phase 和 input_cursor、读游标之后的 input.md 输入、 读 execution-state/state.yaml 及其 by-ref blob，然后重跑当前阶段的 on-entry。

绝不反向重建引擎上下文 aprog 不会把历史对话翻译回引擎、重新喂进去来“恢复记忆”。因为进程状态根本不依赖对话—— 所有要紧的东西（阶段、游标、决策、产物）都已被 state 捕获进目录。对话因此是可丢弃的。

这也是为什么换引擎（Claude → Codex）不丢工作：会话归 aprog 所有、状态在目录里，新引擎冷启动后照样能从状态接着干。 代价只是它读不到上一个引擎的私有对话缓存——而这本来就不需要。

三层数据：谁权威、谁进快照

同一段“执行”涉及三类数据，职责严格分开，别混为一谈：

层文件谁写进快照参与 resume用途
进程状态meta.yml / input.md / execution-state/程序（经引擎）✅✅ 权威让进程接着干
会话记录session.aprog.jsonlDriver✅（在进程目录里）❌前端历史 + 审计
引擎缓存session.jsonl（SDK 私有）引擎❌❌沙箱内临时上下文，可丢
  • 进程状态是唯一权威，驱动 resume；它和会话记录都在进程目录里，跟着进程持久化。
  • 会话记录是 engine-neutral 的逐字转录，只为展示和审计，不参与 resume。
  • 引擎缓存（session.jsonl）是沙箱生命周期内的临时物，沙箱一销毁就没了，永不权威、不进 aprog 快照。
input.md ≠ 会话记录 input.md 只存用户输入，是 state 协议里驱动 FSM 的权威意图流（input_cursor 标记处理到哪）； 会话记录是含思考/工具/回复的完整转录。用户消息在两处都出现，但服务的对象不同——前者驱动执行，后者驱动界面。

检查点：耐久性归平台，一致性归程序

沙箱本地盘随时可能没（厂商回收、OOM、崩溃、网络分区）。让状态扛住这个，靠检查点——把进程目录的 state 子集 经 B 平面流回控制平面。但「什么时候算一个能安全 resume 的点」，平台判断不了，只有上层程序知道。所以职责这么分：

关注点归谁具体
状态的内容与结构程序 / state 库meta / input / execution-state 怎么组织、自己逻辑怎么 resume
状态的一致性（何时算干净）程序在语义一致的点主动调检查点工具标记一个 commit。平台无从判断这个
状态的耐久性平台 / driver怎么传、怎么恢复、多久才真同步一次（debounce）。对内容无知
平台只提供两件基础设施 这一节的总纲：进程一致性主要是上层程序的事，平台只给两样最小的底座——① 持久化进程目录（把 state 子集搬到控制平面、崩溃后灌回）， ② 一个检查点工具，供程序在它认为一致的时刻调用。其余（什么算一致、外部副作用怎么对齐、resume 怎么续）全是程序的语义责任。

两种「一致」：无撕裂 vs 语义一致

把「一致」拆开，才看清为什么非得程序来定：

层级含义谁能判断
无撕裂（tear-free）没在写盘中途、盘面原子（写临时文件再 rename）平台 / driver 可观测（agent 空闲那刻）
语义一致（coherent）state/ 此刻是「我进展到哪」的完整忠实描述，从这恢复能正确续只有程序知道
turn-end 只是无撕裂，不等于语义一致 曾经的设计拿「harness 回复完、转入空闲」这个 quiescent 点当检查点时机——它确实无撕裂，但程序的 state 更新常跨多个 turn （一个逻辑操作写到一半就让出回合）。turn-end 落在中间，快照恢复出来不会损坏、却是个语义半成品。所以判断一致的权威必须是程序，不是平台的猜测。

检查点是给程序的工具，平台只做 debounce

于是检查点反过来了：平台把它做成一个工具（就像别的平台能力——赋予 harness 的一种操作，类似 CLI 命令）， 程序在语义一致处主动调一次 = 「我现在干净，这是个合法续点」。类比数据库的 COMMIT——程序说「一致了」，存储层自己决定何时真正落盘。两个决定就此分开：

决定归谁怎么做
此刻一致吗（commit mark）程序调检查点工具喊一声。可能落在 turn 边界，但判据是语义、不是空闲
何时真捕获 + 真传平台 / drivermark 只武装；真正捕获推迟到 harness 结束本回合回复（turn.end）、盘面静止时。再按 debounce 决定多久真推回 CP
关键 · 调用 ≠ 检查点：mark 武装，turn.end 才捕获 程序调检查点工具时引擎还在跑、盘还在被写——此刻去拍快照会和正在进行的写竞态、撕裂。所以这个调用只是个 mark / 意向， driver 把它挂起，等 harness 把这轮回复说完（turn.end、转入等待输入）、盘面 settled，才捕获——这才同时拿到语义一致（程序 mark）+ 无撕裂（回合结束）。 driver 是 harness 父进程、看着事件流，turn.end 对它是个明确信号。
安全网（兜底）：driver 还要盯 mark 之后到 turn.end 之间——若 agent 又调了修改性工具（write/edit/写盘的 bash，区别于只读的 read/search/list）， 说明被 mark 标记的干净点之后又被改脏，这次不算检查点，等下一个干净的 mark。这逼出程序纪律：写在前、mark 在后、紧接着让出回合。正常程序本就该如此，安全网只抓「mark 完又乱写」。

debounce 是一条耐久性 ↔ 同步成本的滑杆：

策略同步成本崩溃丢失窗口
每个 mark 都同步高最小（最多回退一个 mark）
每 N 个 mark / 每 T 分钟中中
仅休眠时同步最低最大（整段运行）
liveness 兜底 程序若从不调工具，崩溃就全丢回 bootstrap。基线是程序责任（state 技能在 loop 纪律里要求周期性 mark）；平台可选在太久没干净 mark 时，于某个 turn.end 拍一张无撕裂快照当粗兜底，但干净的点永远来自程序的 mark。
休眠 = 一次 mark + 让出沙箱 进程空闲等人，恰好也是最该让出沙箱的时刻。hibernate-prepare 会强制把最近的 mark 同步掉再 destroy—— 所以休眠在机制上就是「确保最新 mark 落地 + 顺手让出沙箱」。CP 也可在分享 / 迁移前主动 checkpoint-now 强制一次。

系统外副作用：程序记游标，平台不回退

检查点能把进程倒带到上一个 mark，但倒带不了已经逃出进程目录的外部副作用——典型是 git push：推上远端的 commit 立刻、不可逆地耐久了，平台 rewind 不了它。

场景：程序开发一个 git 项目，某轮 ① 改码 ② commit + push（远端到 C）③ 进度记进 state/ ④ 检查点同步。崩溃若卡在 ②④ 之间—— push 活下来、记录随沙箱死，恢复回到上一个 mark（还在 B），进程记忆与远端世界岔开。这个双写窗口结构上消不掉（换顺序只把缝挪到另一边）， 平台层没有更好的办法。真正的一致由程序语义兜，两条纪律：

  • 记游标：push 后把 commit hash 记进 state/（跟着检查点原子持久化）——这就是 git 版的 input_cursor。
  • 恢复签出记录的 hash，而非分支：分支 tip 是可变外部引用（可能被那次逃逸的 push 或协作者推动）；签出记录的不可变 hash，工作树与进程状态就都落在 B、彼此自洽，堵死「工作树静默超前于我的记忆」这种最坏的腐坏。
远端孤儿无害——只关心内部是否自洽 恢复后签出 B、重做出 C′，远端那个孤儿 C「挂着就挂着」：只要程序独占这条分支（agent 自己的 WIP 分支），push --force 用 C′ 盖掉 C，孤儿变 unreferenced 被 GC。 正确性底线是恢复后内部自洽——记游标 + 签出 hash 已经保证。「恢复即对账」（认领 C、跳过重做）只是省重复劳动的可选优化，非必需。 唯一会真咬人的是共享分支且他人已 pull（改写已发布历史），超出独占 WIP 分支的场景。

恢复 = 纯 resume，没有「回退语义」

沙箱 / driver 死了，我们不抢救——直接把进程恢复到上一个健康 mark（新沙箱热恢复 / 经 B 灌回检查点）。系统内状态没有任何回退语义要设计：

  • 底层 harness 启动后就是普通 resume，它不知道、也不关心「丢了一段活」。
  • 上个 mark 到崩溃前那几条输入，控制平面 eager 持久化着（输入很小、一到就存，是耐久脊梁）；恢复后当普通新输入重投给正在等输入的 harness，它当第一次收到来处理。没有重放、没有补偿、没有回退事件。
  • 系统外副作用（git push、发邮件、调外部 API）不靠平台回退，靠程序的游标 + 幂等纪律对齐，见 上一节。协议保证仍是 at-least-once，不是 exactly-once。

于是两类状态两种节奏，互补：输入（轻、是脊梁）发出即存；执行状态 / 会话（重）在程序 mark 处按 debounce 检查点。 状态永不删除——退出只是没有沙箱关联，目录与历史完整保留。具体的 spawn / hibernate / wake / exit 调用链见 生命周期流程。

版本化进程目录 → 分支

既然检查点是一串带名字的存档点（程序 mark 时可带 label，像 git commit -m），那进程目录天然就是控制平面上一个被版本化的仓库。 从某个旧存档点岔一条分支去试别的方向，进程历史就从「线」变成「树」。底层就是给进程目录做版本管理，UI 上不暴露任何 git 术语（见 工作窗口 · 分支与进度树）。

进程概念版本化对应
一个检查点 mark一次 commit（label = commit message）
分支（从存档点岔开试新方向）branch
倒带到某存档点checkout / reset
唤醒checkout 分支 tip 灌进沙箱
运行中沙箱里的活态dirty working tree（崩溃丢这截、回到上一 commit）
两棵耦合的版本树：superproject ↔ submodule 注意有两棵独立的 git：① 进程目录——平台在控制平面上版本化（外层）；② 项目代码——程序在项目自己的远端版本化（内层）。 二者靠 记录的 commit hash 游标钉在一起：外层每个 commit 里存着一个指向内层的 SHA——正是 superproject 钉住 submodule 的形状。 于是平台 fork 外层（branch 进程目录）免费、纯在 CP 上；但要让内层也分叉，得靠程序在 resume 时把项目 repo 从那个 SHA 另开一条 git 分支（平台对 git 无知，只传一个「你是 fork」信号）。

运行单位是「分支」，不是进程：列表里每条分支是一条独立可 attach / 可运行 / 可分享的执行线，可并行；「进程」退化成把同家族分支兜在一起的分组。 主分支没有自己的名字——它就是这个进程；只有支线在创建时起名。

没有 merge 外层机械上能 merge，但「合并两条 agent 跑出来的状态 + 两条代码分支」语义上不成立——所以只 branch、不 merge。分支用来做分支尝试（同一起点试不同走法），不是用来最终汇合。
