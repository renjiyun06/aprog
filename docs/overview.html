aprog 设计文档 · 01

总览

本篇定义 aprog 平台的核心抽象、组成模块，以及一次进程唤醒的控制流。 它是设计文档体系的主干；各专题在此基础上展开：程序与发版（02）、镜像与沙箱（03）、通道与协议（04）、运行生命周期（05）、状态契约（06）。

1 · 核心抽象

aprog 是运行 AI 程序的虚拟操作系统。其核心抽象与传统操作系统逐一对应：

操作系统aprog定义
可执行程序程序 Program以目录形态存在的一项能力（skill）；对平台为可版本化发布的包（OCI 制品）。静态实体。
进程进程 Process程序的一次运行，可休眠与唤醒，拥有独立的 PID 与进程目录。
/proc/<pid>/进程目录~/.aprog/<pid>/，进程的权威状态，以 git 仓库承载（以支持分支）。
CPU引擎 Engine底层 agent 循环，当前由 Claude Agent SDK 启动的原生引擎承担，可替换。
内存与运行环境沙箱 Sandbox每进程独立的容器，提供临时算力，由 PPIO 供给，于唤醒时创建。

须区分两类实体：程序为共享、不可变、按版本分发，经 OCI 制品发布； 进程态为每进程独占、可变、可分支，以 git 仓库承载。 二者在唤醒时并行落入沙箱，职责互不重叠。

2 · 体系结构：两个平面

系统交互沿两条正交的轴展开，分由两个子系统承载，不应合并：

维度A · 资源平面B · 数据平面
职责沙箱的创建与销毁（算力）进程的事件、输入与控制（数据）
承载者SandboxProvider（PPIO）driver 与控制平面之间的通道
形态调用厂商 SDK 创建容器由 driver 主动建立的一条 WebSocket
消息格式—统一帧 Frame<t,p>，可扩展

依赖方向受约束：driver 不依赖控制平面的实现代码，仅依赖协议契约 @aprog/protocol； 引擎对平台为黑盒，由 driver 适配。控制平面对外区分两个面：北面向 Web 提供 REST 与 SSE，南面向 driver 提供 WebSocket 通道（/v1/driver/channel）。

4 · 进程唤醒控制流

下图为用户触发唤醒至引擎就绪的控制流时序。唤醒经资源平面创建沙箱、数据平面建立通道，driver 在沙箱内拉起运行环境后回报就绪：

sequenceDiagram
    autonumber
    actor U as 用户与前端
    participant CP as 控制平面
    participant PM as ProcessManager
    participant SP as SandboxProvider
    participant D as driver
    participant E as 引擎
    U->>CP: POST /proc/:pid/wake
    CP->>PM: wake(pid)
    PM->>SP: create(pid, program@version)
    Note over SP: 生成 bindToken；先登记后启动
    SP->>D: 推送 driver 并注入 env，后台启动
    PM-->>U: 返回 PCB（state=waking）
    D->>CP: WebSocket Hello（bindToken）
    CP-->>D: Welcome（pid, mode）
    CP-->>D: 下发种子：程序引用 / registry / 凭证
    D->>D: 拉取 OCI 闭包，untar 至 skills/
    D->>D: git clone 进程态
    D->>E: 经 SDK 启动引擎
    E-->>D: ready
    D->>CP: ready 帧
    CP-->>U: 经 SSE 或轮询转为 running

唤醒为异步：wake 立即返回 waking 态，前端经轮询或 SSE 等待； driver 完成握手、拉取程序闭包与进程态、启动引擎并回报 ready 后，进程转为 running。

5 · 程序分发

程序以目录形态开发，发布为 OCI 制品并推送至 GHCR（工具 aprog-publish）。 依赖声明于程序目录的 aprog.json（{"dependencies":{"state":"0.1.0"}}），不写入 SKILL.md。 发布期解析传递闭包，将每个依赖按 sha256 digest 钉死，拍平写入 OCI config（即 lockfile）。 driver 拉取时不解析依赖，仅取该已钉死的扁平闭包，解包并排部署至 skills/，并借内容寻址实现去重。 详见 02 · 程序与发版。

6 · 状态模型

程序（共享、不可变）经 OCI 按版本拉取；进程态（每进程、可变、可分支）以 git 承载。 execution 内部的状态契约（meta.yml、input.md、execution-state/、cursor 与 resume）详见 06 · 状态契约。控制平面的数据库为上述真相的投影，非权威来源。
