Harness 适配

事件流协议是 engine-neutral 的——它不长得像任何一家引擎。每款底层 harness 由一个 <name>-driver 把它私有的消息翻译成 aprog 事件流。本页给出协议要求 Driver 做的事， 以及 Claude Agent SDK 和 Codex 两套具体映射。换 harness 只换 Driver，协议与前端不动。

Driver 要做的四件事

① 翻译把引擎私有消息 → aprog 事件（turn.* / item.* / user / error）
② 定序保证一次运行内事件的局部顺序（两家 CLI 都不提供序号）；全局 seq 由 control-plane 落库时盖，Driver 不盖（见 沙箱交互界面 · seq）
③ 映射 id把引擎的 item / call 标识归一成稳定 id，让 delta 能折叠、重连能按 id 替换
④ 兜策略自动应答审批（不进流）、回显 user 事件、内存持有“在飞项”供恢复
目标事件种类见 事件流协议 · 事件种类。下面两张表是 ① 的具体落法。

Driver 的运行时拓扑：它 host 一个引擎进程

上面四件事都发生在一个常驻沙箱的进程里。先把这个进程长什么样讲清楚，后面的映射才有落点： driver 自己不调模型——它 import Claude Agent SDK（一个库）， 由 SDK spawn 一个原生 Claude Code 引擎二进制当子进程；agent 循环（内置工具、skill 机制、上下文管理、子代理）真正跑在那个引擎里， driver 只读它的事件流、翻译、拨回控制平面。

aprog-driver 运行时推入 · node bundlecreate 时由 provider 经 files.write 推入、后台 node 启动，持注入的 bindToken 拨回 CP（见 信任模型）。翻译层 + 生命线，不含 agent 循环本体。
└ import · Claude Agent SDK 库编程控制面：query() 取事件、注入输入、interrupt()、多会话。这是 driver「编程式驱动程序」要的自由度——CLI（PTY 单会话）给不了。
　└ spawn · Claude Code 引擎 原生二进制SDK 作 per-platform optional dep 自带（如 @anthropic-ai/claude-agent-sdk-linux-x64）。agent 循环真正在此，走 stream-json 跟 SDK 通信。人在终端跑的 claude CLI = 同一引擎的 PTY 前端。
　　└ → 模型 API引擎二进制直连 Anthropic。driver / SDK 都不碰这一跳。
所以「换 harness 只换 driver」是在 driver↔引擎 这条缝上换：Claude 这条用 SDK 驱原生引擎；Codex 另一套（见 下文）。
为什么 driver 一定依赖 SDK，而不是自己调模型 API 因为 agent 循环本身（内置工具 Bash/Read/Edit、SKILL.md 机制、compaction、子代理）就是 Claude Code 引擎。 aprog 不重写这套循环——它的价值在编排（沙箱 / 生命周期 / 持久化 / 多租），不在造引擎。 所以 driver = 翻译层 + 生命线，真正「思考和动手」的是引擎；SDK 是 driver 编程驱动那个引擎的库，不是纯 in-process API 客户端——query() 底下 spawn 的是引擎进程。

对应到打包产物，运行时这条链是一份烘入的原生引擎 + 一份运行时推入的 node driver：

材料怎么来运行时角色
aprog-driverbun build --target=node 打成 .mjs bundle，create 时 files.write 推入跑在 base 自带的 node v20翻译层 / 生命线，稳定
Claude Code 引擎烘进镜像（npm 装 @anthropic-ai/claude-code 或 SDK 自带二进制）原生（无需 node）引擎本体，随镜像版本钉

引擎是原生 linux/amd64（glibc，PPIO base 为 debian 12）；driver 是 node bundle，跑在 base 自带的 node v20 上（故 base 必须带 node）。driver 不烘进镜像 → 可独立于镜像迭代。 烘进镜像哪一层、怎么烘，见 沙箱与镜像 · 分层 与 烘焙。

引擎与 SDK 版本成对锁 Claude Agent SDK 与 Claude Code 引擎严格版本对齐（每个 SDK 版本对应一个引擎版本）。 driver bundle 内嵌 SDK（运行时推入）、镜像烘引擎二进制 → 这两者必须是匹配的一对、一起升，由 image@版本 钉成一对，不能各自独立 bump。
镜像里怎么把 SDK 指到引擎 driver 启动时用 SDK 的 spawnClaudeCodeProcess（官方原话：给「VM / 容器 / 远程」用）或 pathToClaudeCodeExecutable 指向烘好的引擎路径。 driver 现在是 node 上跑的 .mjs（非 --compile 单文件），不存在「$bunfs 虚拟文件系统里 require.resolve 找不到自带引擎」那类坑—— 引擎当独立文件烘到已知路径、直接 pathToClaudeCodeExecutable 指过去即可。也因此 HarnessSupervisor 对 Claude 这条退化为 SDK 的 spawn 钩子（进程拉起交给 SDK，不再手搓）——与现有 supervisor.ts 待对齐。

转换逻辑放在哪：纯核 + I/O 壳

① 这件「翻译」的事，归属是明确的：EngineAdapter——driver 里唯一为每款引擎重写的模块，跑在沙箱内 （packages/driver/src/engine.ts 定契约，engines/claude.ts / engines/codex.ts 各一份实现）。 所有引擎特定知识被封死在这一个模块里，这正是「换 harness 只换 Driver、前端/control-plane 只认 @aprog/protocol」能成立的原因。

但 adapter 内部又刻意拆成两半，因为「原生→aprog」这段映射会随引擎版本悄悄漂移，必须能离线测：

层是什么纯/副作用签名
纯转换核 transduce全部映射逻辑：kind/value 判定、stop_reason 归一、severity、parent、coalesce纯函数（不读网络、不发消息、不依赖时钟）(native, state) → { events, state }
I/O 壳 start()读 supervisor.stdout 字节、分行 JSON.parse、折转换核、盖 ts有副作用（流 + 时钟）(ctx) → AsyncIterable<Omit<Event,'seq'>>
为什么纯核不造 seq 也不造 ts 转换核吐的是草稿事件（连 ts、seq 都还没有）。ts 由 I/O 壳盖（它有时钟）； 全局 seq 由 control-plane 落库时盖（只有常驻的它看得全跨沙箱/休眠的连续序）。 纯核唯一的「记忆」是显式传进传出的 state——比如每个 id 攒到的文本，用来在 item.end 给出合并全量（coalesce）。 这就是「语义归 adapter、位置归 Sequencer/CP」的切分。
金标测试（golden）守住「双向覆盖」 纯核可测之后：录一段真实引擎原生流当 fixture → 喂转换核 → 把产物钉成「金标」。以后每次改 adapter / 引擎升级， 自动比对产物是否一字不差，退化立刻飘红，而不是等用户在界面上发现「工具结果不显示了」。 已落地：engines/claude.test.ts（thinking/工具/reply 一轮 + stop_reason 归一）、 engines/codex.test.ts（命令三段 + error 分级 + turn.aborted 归一）。共享的累积/合并在 engines/coalesce.ts，I/O 壳在 engines/shell.ts。

跨 harness 的难点与裁决

问题ClaudeCodexaprog 裁决
全局顺序号无CLI 无（Responses API 有）Driver 统一合成 seq——这正是适配层存在的核心理由
用户回显不回显原生回显一律由 Driver 发 user 事件（echo-from-stream），去重引擎自带回显
工具 / 命令tool_use + 单独 tool_result 消息命令三段 + 结构化结果统一为 aprog tool/command 项：start → (args/output delta) → end(result)
思考 / 推理单流 thinking + 签名并行摘要分段 + 原始 CoTthinking 项；需要时 delta 带 part 序保持不交错
停止原因stop_reason + result.subtypeTurnComplete / TurnAborted{Interrupted/Replaced/BudgetLimited}归一成「用户反应」小集 completed/interrupted/limit/refused/error；引擎原词存 raw_stop_reason
错误分级(多为 fatal)Error / StreamError(重连) / Warningerror.severity = fatal/transient/warning，让 UI 别把「重连中」报成「挂了」
引擎多余的丰富度—限流/budget/guardian/hook/rate_limit…用户不据此行动的 → 不进 typed，原样进 raw / usage.extra（保真兜底，不污染抽象）
审批（默认放行）审批往返(需回 Op)自动批准：Driver 自动应答，审批不进流（设计上 harness 全控沙箱）
引擎私有会话~/.claude/.../<id>.jsonl + resumerollout JSONL + codex resume都是引擎私有缓存：不权威、不进快照（见 三层数据）

所以 aprog 协议覆盖多家 harness 的关键，就是把它们的差异全部吸收进 Driver：顺序号合成、身份归一、用户回显、审批自动、引擎私有会话忽略。 前端只认 aprog 事件流，永远不知道下面是 Claude 还是 Codex。
