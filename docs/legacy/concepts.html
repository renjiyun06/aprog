核心概念

aprog 只有四个一级名词：程序、进程、引擎、沙箱。理解它们之间的关系， 就理解了整个平台。它们刻意互相解耦——谁都不知道对方的具体实现。

程序 Program

一段以 skill 形态存在的能力，载体是一份 SKILL.md（外加它带的资源、脚本）。 程序是静态实体：它只声明“要做什么、分几个阶段、读写哪些状态”，不关心由谁来执行、跑在哪台机器上。

在界面上，每个程序是桌面/任务栏上的一个图标——“UI 设计”“需求分析”“代码库分析”都是程序。 打开它，看到的是这个程序的若干个进程。

进程 Process

程序的一次运行，完全类比操作系统进程：有 PID，有自己的工作目录，可被休眠（hibernate）和唤醒（wake）。 同一个程序可以并行跑多个进程——“UI 设计”可以同时有“官网重设计”和“后台改版”两个互不相干的进程。

进程的真正状态落在它的目录 ~/.aprog/<pid>/ 里（类比 /proc/<pid>/，也是 harness 的 cwd）。 它不挂持久卷，靠检查点持久化——一个进程就是一个 git 仓库，进程目录是它的工作树，一个检查点 = 一次 commit，耐久靠 git push 到控制平面托管的仓库。 这个目录就是进程，存储模型详见 进程状态存储（Git 模型），目录布局详见 进程目录与状态。

两种状态 进程对用户只有两态：运行（当前关联着沙箱）与 休眠（没有关联沙箱）。休眠进程没有正在进行的对话，但它的目录与状态完整保留，随时可唤醒。

引擎 Engine

底层的执行引擎，类比 CPU。它提供 agent loop —— 给模型喂上下文、跑工具调用、产出思考与回复。 当前实现是 Claude Agent SDK；未来可替换为 Codex SDK 或其他 agent loop。

程序不知道引擎是谁。引擎的差异封装在一层薄薄的 Driver 里 （见 组件清单）。这让同一段程序能换引擎跑，性价比和能力差异留给运维决定。

沙箱 Sandbox

per-process 的容器：装着工具（git、curl、运行时…），给引擎一个文件系统、进程空间和网络去跑。 当前由 PPIO（派欧云，E2B 同构托管）提供 lifecycle 与 exec 能力。

沙箱是临时算力，本身无权威状态——它随时可以销毁重建，磁盘是临时盘、不挂持久卷。进程休眠就是释放沙箱；唤醒就是重新分配一个沙箱、git clone 进程仓库到它的工作树里（见 Git 模型）。 所有“要紧的东西”都已提交进 git、推到控制平面，不在沙箱里。

Program Engine Sandbox

这种解耦是平台价值所在：aprog 管的是“程序的生老病死”，而不是绑定某一个 LLM 厂商或某一种容器。 详细的架构展开见 三层抽象。

一次执行的生命周期

从用户在界面里新建一个进程，到它跑完归档，会经历这样一条链：

spawn 休眠 创建进程目录（meta.yml / input.md），尚未分配沙箱
↓
attach / wake 运行 分配沙箱、注入进程目录、启动 Driver + Engine
↓
▶ 运行中：对话循环 用户输入 → 引擎跑 agent loop → 思考流 / 工具调用 / 回复 → 对话区。 事件同时被 Driver 翻译、落进进程目录的 会话记录。
↓
detach 用户离开；沙箱可保留，引擎继续跑
↓
hibernate 休眠 释放沙箱；状态已 commit 并 push 到控制平面托管的进程仓库，完整保留
↺ 可反复 wake / hibernate↓
wake 运行 重新分配沙箱，git clone 进程仓库到工作树，引擎冷启动，靠状态文件重建工作记忆——不重放对话
↓
exit 已退出 最后一次 commit + push、销毁沙箱。进程仓库永不删除，新输入可再 attach 回来

关键一点：唤醒不依赖对话上下文。引擎冷启动后，是靠读进程目录里的状态文件 （阶段、输入游标、execution-state）来知道“做到哪了”，而不是把历史对话重新喂回去。 这条设计贯穿全文，详见 进程目录与状态。
