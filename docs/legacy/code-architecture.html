代码架构

这是 monorepo 的工程视角：仓库里有哪些 package、各自是什么、源码怎么摆、谁依赖谁、怎么跑。 与「组件清单」（运行时有哪些活物）、「三层抽象」（概念模型）互补—— 这一页只回答「代码在哪、怎么组织」。逐字对齐各包 package.json 与 src/ 树。

全景：6 运行时 package · tools/ · images/ · programs/

仓库是一个 Bun workspace（根 package.json 的 workspaces: ["packages/*", "tools/*"]）。packages/* 是运行时库（被 import）；tools/* 是构建期 / 操作员 CLI（bun link 上系统 PATH、不被 import）。images/ 与 programs/ 是仓库顶层内容目录、都不是 package（无人 import）——它们是 git 跟踪的源/管理面，各有一条「源目录 → 发版工具 → 外部产物」的对称管线：

aprog/
├ packages/                                  运行时库（被 import）
│ ├ control-plane/   @aprog/control-plane  — 北向 API + 进程编排 + 流中枢（host service）
│ ├ sandbox/         @aprog/sandbox        — A 平面 provider 运行时库（create/destroy，被 CP import）
│ ├ driver/          @aprog/driver         — 沙箱内常驻 driver（烘进镜像，非源码依赖）
│ ├ protocol/        @aprog/protocol       — engine-neutral 事件 + 信封（叶子，无依赖）
│ ├ log/             @aprog/log            — 共享结构化日志（叶子，无依赖）
│ └ web/             @aprog/web            — Solid.js「OS 桌面」前端
├ tools/                                     构建期 / 操作员 CLI（bun link 上 PATH，一工具一包）
│ └ bake/            @aprog/bake           — aprog-bake：锚仓库根跑 images/<名>/<版本>/<供应商>/bake.ts
├ images/            （非 package）镜像源：<名>/<版本>/<供应商>/bake.ts —— aprog-bake → 厂商 snapshot
├ programs/          （非 package）程序源：<类>/<名>/[<版本>/]SKILL.md —— aprog-publish → OCI 包
└ docs/              本文档站
包出处跑在哪入口 / 产物

依赖图：单向无环

区分两类边：源码依赖（import，编译期）与镜像内容依赖（driver 跟着镜像一起烘，非 import）。

web┄ HTTP/SSE ►control-plane运行时,非源码依赖(走 /cp-api 契约)
control-plane───►protocol·sandbox
sandbox───►log·ppio-sandbox外部(延迟 require)
driver───►protocol·engine SDK外部,在镜像里
control-plane┈┈►driver镜像内容依赖,非源码
driver┄ OCI pull ►程序包(OCI)spawn 时拉 programs/ 发布的包,非源码 import
───► 源码依赖　·　┈┈►/┄► 运行时 / 镜像内容 / OCI 内容依赖。protocol 与 log 是叶子（零依赖）；images/ 与 programs/ 是内容——被消费/拉取，不被 import。
两条「非源码」边是有意的 ① web 不 import 任何 @aprog/*——前后端只靠 HTTP/SSE 契约耦合（见 控制平面 API），前端可独立演化。 ② control-plane 不源码依赖 driver——driver 跟着沙箱镜像一起烘，通过固定 protocol + DriverChannel 交互，故能独立换引擎、独立发版。

@aprog/control-plane

平台中枢，跑在 aprog VM 上的长跑 service。源码依赖 @aprog/protocol + @aprog/sandbox（外加 nodemailer 发信）。src/ 按子系统分目录：

src/
  index.ts            入口:Bun.serve(:8099) 组装并起服务
  config.ts           配置(GitHub / SMTP / dataDir…)
  db/index.ts         SQLite schema + applyMigrations(建表)
  api/                北向 HTTP/SSE —— router · context(Deps 接缝) · http(组装根)
    middleware/auth   · respond · errors · sse(流式端点)
    routes/           auth · programs · proc · installations · shares · notifications
  auth/               codes · email · tokens · users · validate
  catalog/            programs(身份+版本目录,播种两表) · installs
  process/            manager(PCB 编排) · repo-gateway(建进程 git 仓库) · sandbox-gateway(A 平面收口,当前 mock)
  driver-channel/     B 平面流中枢(受理 driver 拨入 / 下发 input)
  persistence/        checkpoint(检查点)
  stream/             hub · store · resync(事件盖全局 seq + 落库 + 扇出 + 重放)

实现分层（翻译膜 / 请求生命周期 / Deps 接缝）见 控制平面 API 实现；数据表见 数据模型。

@aprog/sandbox

A 平面：把沙箱当「被托管的资源」。纯运行时 provider 库（被 control-plane import），不含任何 bin。源码依赖 @aprog/log，外部 ppio-sandbox（延迟 require，mock 路径不扛）。

src/
  provider.ts         SandboxProvider (create / destroy) + CreateOptions(bindToken / onProvisioned)
  providers/ppio.ts   PPIOProvider(driver 运行时推入 + 后台 node;AgentBay 暂下线,留扩展点)
  types.ts            ImageRef / Resources / SandboxHandle / ProviderId('ppio')
  errors.ts           SandboxError 体系(provider 内 wrap() 归一厂商异常)
  index.ts            单一出口(只导运行时那半)
打镜像不在这个包 早先的 ImageBaker / ImageBuilder 策略接缝（过度设计）已拆除。镜像 = 仓库顶层 images/ 目录，每版本下每供应商一个 bake.ts 直调该厂商 SDK；跑 bake.ts 的 runner 是 tools/bake（aprog-bake），已搬出本包——它与 sandbox 运行时代码零耦合，只认 images/ 约定。见 沙箱与镜像。 control-plane 对本包的耦合面只是几个 import（PPIOProvider + ImageRef / SandboxHandle 等类型）。

tools/ —— 构建期 / 操作员 CLI

第三个物种：既不是被 import 的运行时库（packages/*），也不是被跑的内容（images/*），而是命令行工具。tools/* 是 workspace 包，一工具一包（与 images/<名>/<版本>/ 同构：自包含目录、自带依赖、ls 即枚举），靠 bun link 软链进 ~/.bun/bin 上系统 PATH。

tools/
  bake/   @aprog/bake   bin: aprog-bake   deps: citty(help/arg)
    src/bake.ts   findRepoRoot(import.meta.dir 已解析软链) → images/<名>/<版本>/<供应商>/bake.ts
路径靠两层约定，与 cwd 无关 L1 工具→仓库根：锚定 import.meta.dir（Bun 把 ~/.bun/bin 的软链解析成真实仓库内位置）向上找标记 = name==="aprog" 的工作区根 package.json，再经 APROG_REPO_ROOT 传给 bake.ts（免得 bake.ts 自己数上溯层数）。L2 参数→目标：images/<名>/<版本>/<供应商>/bake.ts（文件系统即 registry）。由此 aprog-bake 永远操作「它被 bun link 的那个仓库」的 images/，在任何目录敲都对。 依赖三层：共享基建（citty）各包自 dep、Bun 跨包去重；工具专属重依赖归各自包；被 images/ 下 bake.ts（内容、非包）调用的厂商 CLI（ppio-sandbox-cli）按需安装 / 上 PATH。

@aprog/driver

沙箱内常驻进程，create 时由 provider 运行时推入（files.write driver.mjs）、后台 node 启动，持注入的 bindToken 拨回 CP。源码依赖 @aprog/protocol；引擎 SDK 是运行时（在镜像里）。control-plane 不源码依赖它。

src/
  driver.ts           主:自启拨回 + 编排
  supervisor.ts       harness 父进程 / 子进程监管
  engine.ts + engines/   claude · codex · shell(引擎适配) + coalesce(事件合流)
  channel.ts          DriverChannel 客户端(事件上行 + input 下行)
  sequencer.ts        局部定序(全局 seq 由 CP 落库时盖)
  fs.ts · bundle.ts   进程目录 / 检查点(git)
  index.ts

「SDK 私有消息 → engine-neutral aprog 事件」的翻译在沙箱内做——每款引擎一个 <name>-driver，对外协议固定。详见 组件清单 · Driver、Harness 适配。

@aprog/protocol

叶子包，零依赖。control-plane 与 driver 共用的线缆契约：envelope.ts（信封 / 定序）+ events.ts（engine-neutral 事件）+ index.ts。它是「平台 ↔ 引擎解耦」的支点——见 事件流协议。

@aprog/log

叶子包，零依赖。单文件 src/index.ts：createLogger(scope) 结构化日志。当前被 @aprog/sandbox 使用；其它包按需接入。

@aprog/web

Solid.js + Vite 的「OS 桌面」前端。不 import 任何 @aprog/*——只通过 /cp-api（HTTP/SSE，vite 代理到 :8099）与后端按契约通信。

src/
  index.tsx · App.tsx     挂载 + 桌面壳
  components/             Desktop · Window · Taskbar · Login/Register · ProgramShell …
  programs/               各智能程序(design / codereviewer / settings / store) + registry + shell-program
  stores/                 响应式状态:auth · catalog · processes · windows · installed · notifications …
  lib/                    api(fetch 封装) · highlight
  styles/app.css

窗口/桌面交互见 工作窗口；前端消费的接口见 控制平面 API。

images/（非 package）

仓库顶层的镜像目录：images/<名>/<版本>/<供应商>/bake.ts。版本目录 = 逻辑镜像（中立），其下按供应商分子目录各自打包。不是 workspace 成员、无人 import——它是构建期内容，由 aprog-bake <名> <版本> <供应商> 跑对应 bake.ts（直调厂商 CLI ppio-sandbox-cli template build）。当前有 images/base/0.1.0/ppio/（烘 claude-code 引擎 + 非密 GLM 路由）。 目录模型、版本=目录名、工具=材料等见 沙箱与镜像；程序如何依赖镜像见 data-model · program_versions。

programs/（非 package）

仓库顶层的程序源目录——平台内置、随平台一起暴露给用户的那批智能程序。和 images/ 一样不是 workspace 成员、无人 import：它是 git 跟踪的源 / 管理面，发版产物在外部 registry。按 SKILL.md frontmatter 的 kind 两分： 

programs/
├ applications/   kind: application — 有执行模型(FSM/阶段/产物)，被运行     design · shape · skin · grasp
│ └ <名>/[<版本>/]SKILL.md
└ libraries/      kind: library     — 封装协议/机制，被引用而非被执行       state · live-annotate
  └ <名>/[<版本>/]SKILL.md

依赖单向：application → library（如 design 依赖 state 状态协议 + live-annotate 预览/批注）。这是「三公民」（Library / Program / Capability）里的 Program + Library 两层；Capability（给底层 harness 调的 CLI / MCP 工具）另有其家。

发版管线与镜像逐字对称——源目录在 git、产物在外部 registry：

programs/<类>/<名>/<版本>/SKILL.md   ──aprog-publish──►  OCI artifact (GHCR)  → digest
images/<名>/<版本>/<供应商>/bake.ts   ──aprog-bake────►  厂商 snapshot        → ImageRef

「git 是真相，registry 只存产物」。OCI 包形态、依赖闭包解析、spawn 拉取、库表降为投影等见 程序与发版。

现状要补一层 磁盘上现在是 programs/<类>/<名>/SKILL.md，还缺 /<版本>/ 这层；catalog 也还是 control-plane/src/catalog/programs.ts 里一个写死的 const。目标：补成 programs/<类>/<名>/<版本>/，并由 aprog-publish 从 git 源 + OCI digest 投影出库表、取代 const（见 program-package · 本地管理 与 data-model · program_versions）。
