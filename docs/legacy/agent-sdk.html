Claude Agent SDK 事件模型

事件流协议是 engine-neutral 的抽象；这页是它落在 Claude Agent SDK 上的地面真相——SDK 到底吐什么、字段叫什么、哪些是稳定标识。Driver 正是把这套原生事件归一成 aprog 事件流的那一层（映射汇总见 Harness 适配 · Claude，这里给出它依赖的全部底层细节）。

两层 API,别混 Agent SDK(TS 包 @anthropic-ai/claude-agent-sdk)的 query() 在外层吐 SDKMessage;开了逐字流式后,它把原生 Messages API 流事件(message_start/content_block_delta…) 原样裹在 stream_event 里透出来。aprog 的 item.* 增量几乎全来自后者。本页字段以 TS SDK 为准(见 版本)。

aprog 固定用 TypeScript 版 Agent SDK。一句话锁定:

query() 与 SDKMessage 顶层信封

入口是 query({ prompt, options }),返回一个异步迭代器 AsyncGenerator<SDKMessage>。每条消息按 type 字段判别:

type何时出现字段(名 : 类型)
system 会话开头一次(subtype:"init");上下文压实时(subtype:"compact_boundary") type、subtype:"init"|"compact_boundary"、session_id:string、uuid:string。init 另带本次会话元信息:model、cwd、tools:string[]、permissionMode、mcp_servers、slash_commands 等;compact_boundary 另带 compact_metadata:{ trigger:"manual"|"auto", pre_tokens:number }
assistant 每条助手消息完成后(coalesced 全量) type、message:BetaMessage(内含 id:string、role:"assistant"、content:ContentBlock[]、model:string、stop_reason、usage)、parent_tool_use_id:string|null、session_id:string、uuid:string
user 用户输入,以及工具结果回灌(SDK 把 tool_result 当 user 消息送回) type、message(role:"user"、content 可含 tool_result 块)、parent_tool_use_id:string|null、session_id:string、uuid:string
stream_event 逐字增量,仅在 includePartialMessages 开启时 type、event:BetaRawMessageStreamEvent(原生流事件,见 内层)、parent_tool_use_id:string|null、session_id:string、uuid:string
result 整次 query() 收尾一次 type、subtype:"success"|"error_max_turns"|"error_during_execution"、is_error:boolean、num_turns:number、duration_ms:number、total_cost_usd:number、usage:{ input_tokens, output_tokens, cache_creation_input_tokens?, cache_read_input_tokens? }、result:string(仅 success)、session_id:string、uuid:string
TS 类型名:SDKSystemMessage / SDKAssistantMessage / SDKUserMessage / SDKPartialAssistantMessage(即 stream_event)/ SDKResultMessage; 压实边界另有独立的 SDKCompactBoundaryMessage(type:"compact_boundary")。

逐字流式:includePartialMessages 与 stream_event

不开 partial,query() 只在每条助手消息完成后吐一条 assistant——没有"正在打字"。 开了之后,SDK 额外吐 stream_event,其 .event 就是原生 Anthropic 流事件本体。两者并存:

  • 实时取自 stream_event 流(细到 token)——喂 aprog 的 item.delta;
  • 权威全量取自随后的 assistant 消息(已 coalesced)——喂 aprog 的 item.end,不靠拼 delta。
// TS:同一次 query 里两类消息交替到达
for await (const m of query({ prompt, options: { includePartialMessages: true } })) {
  if (m.type === "stream_event") handleRaw(m.event);   // 逐字增量
  else if (m.type === "assistant") finalize(m.message); // 权威定稿
  else if (m.type === "result")    closeTurn(m);        // 用量收尾
}

原生 Messages API 流事件序列

stream_event.event 严格遵循 Anthropic Messages API 的 SSE 顺序。一条助手消息的事件骨架:

message_start            → { message: { id:"msg_…", role, content:[], usage } }
  content_block_start    → { index:0, content_block:{ type:"text"|"thinking"|"tool_use", … } }
  content_block_delta*   → { index:0, delta:{ type:"text_delta", text:"…" } }
  content_block_stop     → { index:0 }
  content_block_start    → { index:1, content_block:{ type:"tool_use", id:"toolu_…", name, input:{} } }
  content_block_delta*   → { index:1, delta:{ type:"input_json_delta", partial_json:"{\"q\":" } }
  content_block_stop     → { index:1 }
message_delta            → { delta:{ stop_reason }, usage:{ output_tokens } }   // output_tokens 是累计
message_stop             → {}
还可能穿插 ping(保活,忽略)与 error(如 overloaded_error)。完成边界: content_block_stop 收一个块、message_stop 收整条消息——直接对应 aprog 的 item 结束 / turn 结束。

逐个原生事件的字段:

事件字段
message_starttype、message:{ id:string, type:"message", role:"assistant", model:string, content:[], stop_reason:null, stop_sequence:null, usage:{ input_tokens, output_tokens, … } }
content_block_starttype、index:number、content_block(按类型分三种,见下表)
content_block_deltatype、index:number、delta(四种之一,见 四种 delta)
content_block_stoptype、index:number
message_deltatype、delta:{ stop_reason:string, stop_sequence:string|null }、usage:{ output_tokens:number }(累计值)
message_stoptype(无其它字段)
ping / errorping:仅 type;error:type、error:{ type:string, message:string }

content_block_start.content_block 按 type 分三类(这是 aprog item_type 的来源):

content_block.type字段对应 aprog 项
texttype、text:""(起始为空,靠 text_delta 填)reply
thinkingtype、thinking:""、signature:""thinking
tool_usetype、id:"toolu_…"、name:string、input:{}(起始为空,靠 input_json_delta 填)tool——Driver 再据 name 细分:Bash→command、Edit/Write→file_change

四种 delta 载荷

每个 content_block_delta 按 delta.type 判别,载荷各不相同:

delta.type载荷字段用途 / 对应 aprog patch
text_deltatext: string回复正文增量 → reply 项的 {kind:"text"}
thinking_deltathinking: string思考增量 → thinking 项的 {kind:"text"}
input_json_deltapartial_json: string工具入参的 JSON 分片(非对象!需累计到 content_block_stop 再 parse) → tool 项的 {kind:"tool_args"}
signature_deltasignature: string思考块的完整性签名,在思考块 stop 前给出;非正文,随定稿保留即可
thinking 默认可能为空 较新 Opus 默认 thinking.display:"omitted"——块结构照旧出现,但 thinking_delta 的文本为空,只剩 signature_delta。 要实时展示推理,Driver 侧需把 display 设为 "summarized"(见 资料来源)。

三个标识:把 delta 折成一条的锚

aprog 的 id(逻辑分组)直接取自 SDK 的稳定标识。三者各管一段:

SDK 标识出处性质 / aprog 用法
message.idmessage_start.message.id(msg_…),在 partials 与最终 assistant 间稳定把整条消息的事件锚到一起;是 delta↔定稿的身份桥
index每个 content_block_* 上的 0 基序号,同一块的所有 delta 不变区分一条消息里的多个块;aprog 用 message.id:index 作 thinking/reply 项的 id
tool_use.idcontent_block_start.content_block.id(toolu_…)工具调用的稳定引用;aprog tool 项直接用它作 id,并据此把后续 tool_result 回挂
seq 不来自 SDK SDK 只保证一次运行内的到达顺序,没有跨生命周期的全局序号。aprog 的 seq 由 control-plane 落库时盖 (单调、跨沙箱/休眠连续)——这正是"流归进程不归沙箱"在适配层的体现,详见 事件流协议 与 沙箱交互界面 · seq。

工具生命周期与 canUseTool 审批回调

一次工具调用在流上分两段,跨越两条 SDK 消息:

① 模型发起(assistant 流内)content_block_start type=tool_use(带 id/name)→ input_json_delta* 累计入参 → content_block_stop
↓
② 结果回灌(下一条 user 消息)user 消息里的 tool_result 块,按 tool_use_id 对应回 ① 的调用。内置工具(Bash/Read/Edit)同理
所以 aprog tool 项的 result 不来自模型流,而来自这条回灌的 user 消息——Driver 按 tool_use_id 挂回同一项。

执行前,SDK 可走 permission 回调 canUseTool 征询是否放行: 

// TS 签名
async function canUseTool(
  toolName: string,
  input: Record<string, any>,
  options: { signal: AbortSignal; suggestions?: PermissionUpdate[] }
): Promise<PermissionResult>

type PermissionResult =
  | { behavior: "allow"; updatedInput: Record<string, any> }   // 可改写入参
  | { behavior: "deny";  message: string }                       // 拒因回灌给模型
aprog 自动放行 产品形态自动批准一切操作:Driver 的 canUseTool 恒回 allow,审批不进 aprog 流 (协议里没有 allow/deny 事件)。这与 Codex 那种"审批往返"的差异,统一被 Driver 吸收——见 跨 harness 裁决 · 审批。

回合收尾:result 与 usage

一次 query() 以一条 result 收尾(SDKResultMessage),aprog 映射成 turn.end:

字段含义
subtype"success" / "error_max_turns" / "error_during_execution" … → 连同 message stop_reason 归一成 turn.end.stop_reason
usage{ input_tokens, output_tokens, cache_creation_input_tokens?, cache_read_input_tokens? } — 全程总账
total_cost_usd整次运行累计成本
num_turns / duration_ms / is_error回合数 / 墙钟耗时 / 是否出错
用量有两处:流中的 message_delta.usage.output_tokens 是累计(可做实时进度); 最终账目在 result.usage。aprog turn.end.usage 取后者。

会话与 resume

每条 SDK 消息带 session_id;SDK 自身支持续接(options.resume: sessionId 或 options.continue), 会话落在 ~/.claude/projects/<cwd>/<session-id>.jsonl。

引擎私有,不权威 这个 .jsonl 是引擎私有缓存,不是 aprog 的数据源——aprog 的唯一真相是 control-plane 上那条带全局 seq 的流 (见 实时与持久是同一条流 与 三层数据)。SDK 的 resume 只在引擎层面用; aprog 的跨沙箱续命靠进程目录回灌,不依赖它。

归一回 aprog

以上原生事件,Driver 全部吸收成 engine-neutral 的 aprog 事件——前端永远只认 aprog 流,不知底下是 Claude 还是 Codex。 完整对照表(含 Codex 并列)见 Harness 适配。一句话总结 Driver 在 Claude 这条线上做的归一:

  • 顺序号合成:SDK 无全局 seq → control-plane 落库盖。
  • 身份归一:message.id:index / tool_use.id → aprog id。
  • 用户回显:SDK 流式输入不回显用户消息 → Driver 自发 user 事件(echo-from-stream)。
  • 审批自动:canUseTool 恒 allow,不进流。
  • 子代理归属:parent_tool_use_id → aprog 信封的可选 parent(前端据此折子树,见 协议 · parent)。
  • 停止原因:message stop_reason + result.subtype → turn.end.stop_reason。
  • 上下文压缩:compact_boundary → aprog compaction 事件(流不删减,仅如实记录)。
双向覆盖已闭合 补上 parent / stop_reason / compaction 后,协议对 Claude SDK 双向自洽:aprog 每个事件都能从 SDK 产出;SDK 每个事件要么被覆盖,要么是配置(init 元信息归 meta.yml)/保活(ping)/审批(自动放行)等故意不进流的项。
