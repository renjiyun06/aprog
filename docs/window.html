工作窗口

每个程序打开是一个窗口，所有程序的窗口骨架都一样——它本质是“执行一个智能程序”的视图， 而不是某个业务领域的专用界面。窗口分三区：进程列表、对话区、目录区。

进程列表（左）

当前程序的所有进程。每行显示状态指示圆点 + 进程名。顶部是“进程”标题、进程数和一个 ＋（新建进程）。 点某个进程即 attach 到它，对话区和目录区随之整体切换（不混合）。

新建进程：点 ＋ 弹出一个表单，目前只填一个名称（CPU/内存/沙箱配置等后续再加）。 每个新进程是 ~/.aprog/<pid>/ 下的一个独立目录。

进程两态：运行与休眠

对用户而言进程只有两种状态，由“是否关联沙箱”决定：

状态含义对话区目录区
运行当前关联着沙箱，正在执行有：会话流 + 输入框有
休眠未关联沙箱无：此刻没有进行中的对话，只显示“唤醒”入口有

休眠进程没有交互区——它压根没有正在运行的对话。但它的目录区依旧在，文件可只读浏览。 界面提供显式的状态切换：运行中的进程有“休眠”按钮（释放沙箱），休眠中的进程有“唤醒”按钮（重新关联沙箱）。

唤醒后的连续性 唤醒后对话区会铺出该进程完整的历史对话（读自进程目录的会话记录），但底层引擎是冷启动、不重放对话—— 它靠状态文件知道做到哪了。原因见 唤醒靠状态，不靠对话。

对话区（中）

窗口主区就是对话区，没有阶段步进器、没有授权弹窗。它呈现用户与底层 harness 的对话， harness 产出的东西归为三类：思考流、工具调用、回复，加上用户消息。 这些事件如何从引擎流到这里，见 事件协议与会话。

  • 输入框：Enter 发送，Ctrl+Enter 换行，无发送按钮、无提示语。
  • 打开文件时主区顶部才出现标签栏：第一个标签是“对话”，其余是只读文件标签。

目录区（右）

当前进程的目录树（~/.aprog/<pid>/），真实的多级树：meta.yml / input.md / execution-state/ / session.aprog.jsonl / artifacts/…。默认收起， 点标题栏的目录按钮向右弹出——整个窗口加宽来容纳它，不挤压对话区。会话区与目录区之间是一条可拖动的分割线。

文件一律只读：用户能查看，但修改只由程序和底层 harness 进行。文件按类型显示不同图标（VS Code 风格）， 支持语法高亮；子目录默认只展开一级，有缩进引导线，文件夹开/合图标不同。
