工作窗口

每个程序打开是一个窗口，所有程序的窗口骨架都一样——它本质是“执行一个智能程序”的视图， 而不是某个业务领域的专用界面。窗口分三区：进程列表、对话区、目录区。

进程列表（左）

列表的单位是「分支」（执行线），不是进程。一个进程可以有多条并行分支（见 分支与进度树），所以「进程」在这里只是把同家族的分支兜在一起的分组。 每行显示状态圆点 + 名字。点某条分支即 attach 到它，对话区和目录区随之整体切换（不混合）。

渐进式展开，不分叉就不啰嗦：

  • 没有支线（绝大多数情况）：一个进程就一行，直接显示进程名——主分支不另起名字，它就是这个进程。
  • 开了支线后：这一行下方缩进挂出支线行（带 ⤷ 标记），显示创建时起的支线名。一层为止，不做无限树。

列表只保留圆点 + 名字，存档点 / 进度树 / 开支线都不挤进来，移到对话区标题栏。 新建进程：点顶部 ＋ 填个名称，是 ~/.aprog/<pid>/ 下一个独立目录。

列表里也包含别人分享给你的分支（分享下沉到分支），可加小标记区分「我创建的 / 共享给我的」与角色（可写 / 只读）。

进程两态：运行与休眠

对用户而言进程只有两种状态，由“是否关联沙箱”决定：

状态含义对话区目录区
运行当前关联着沙箱，正在执行有：会话流 + 输入框有
休眠未关联沙箱无：此刻没有进行中的对话，只显示“唤醒”入口有

休眠进程没有交互区——它压根没有正在运行的对话。但它的目录区依旧在，文件可只读浏览。 界面提供显式的状态切换：运行中的进程有“休眠”按钮（释放沙箱），休眠中的进程有“唤醒”按钮（重新关联沙箱）。

唤醒后的连续性 唤醒后对话区会铺出该进程完整的历史对话（读自进程目录的会话记录），但底层引擎是冷启动、不重放对话—— 它靠状态文件知道做到哪了。原因见 唤醒靠状态，不靠对话。

对话区（中）

窗口主区就是对话区，没有阶段步进器、没有授权弹窗。它呈现用户与底层 harness 的对话， harness 产出的东西归为三类：思考流、工具调用、回复，加上用户消息。 这些事件如何从引擎流到这里，见 事件协议与会话。

标题栏承载这条分支的元信息：状态点 + 名字 + 程序版本 + 当前存档点 chip + 进度树按钮 + 休眠/唤醒 + 协作者 + 分享。其中：

  • 输入框：Enter 发送，Ctrl+Enter 换行，无发送按钮、无提示语。
  • 打开文件时主区顶部才出现标签栏：第一个标签是“对话”，其余是只读文件标签。
  • 当前存档点 chip（⬤ 某存档点 · 进行中）：点开是这条分支的存档点切换器（只读跳转查看 / 开支线 / 看进度树）。
  • 进度树按钮（标准 git-branch 字形）：一键直达整棵进度树，看全部分支。
  • 分享按钮（仅 owner 可见）：分享的是这条分支，见 分享面板。
  • 只读成员：以 viewer 身份进来的人，输入框、[允许]/[拒绝]、打断都禁用，只能看。

目录区（右）

当前进程的目录树（~/.aprog/<pid>/），真实的多级树：meta.yml / input.md / execution-state/ / session.aprog.jsonl / artifacts/…。默认收起， 点标题栏的目录按钮向右弹出——整个窗口加宽来容纳它，不挤压对话区。会话区与目录区之间是一条可拖动的分割线。

文件一律只读：用户能查看，但修改只由程序和底层 harness 进行。文件按类型显示不同图标（VS Code 风格）， 支持语法高亮；子目录默认只展开一级，有缩进引导线，文件夹开/合图标不同。

分支与进度树

进程历史不是一条线，而是一棵树：在任意存档点可以开一条支线去试别的走法，主线不受影响。底层是给进程目录做版本化， 但界面全程不出现 git 术语——只说 存档点 / 主线 / 支线 / 进行中 / 进度树。

  • 存档点：程序在语义一致处自己 mark 的点（可带 label），是合法续点。一条分支有自己线性的存档点历史。
  • 开支线：从某存档点岔出一条独立执行线，创建时起名（主分支不起名、就是进程名）。新支线即出现在列表缩进处。
  • 倒带到某存档点（回到此处）：毁灭性，放弃其后进度，需二次确认。

进度树用 gitgraph 形态呈现：竖向时间轴，每条分支一条彩色泳道，存档点是圆点，支线从父存档点曲线岔出，存档点按时间跨泳道交错。 无外框——分支再多也清爽。顶部一排分支名 chip（gitgraph legend）：点某条分支名，只高亮它及其祖先、其余淡出，即按分支筛选查看各存档点。

从任何分支都能看全貌 进度树属于进程（整棵版本树），不属于某条分支——不管你站在哪条分支，标题栏的进度树按钮打开的都是同一份全貌，只是「你在这」高亮在当前分支。 唯一例外：被分享进来的分支（你是 sharee）只看到它那条线 + 岔开前的共享祖先，看不到没分享给你的兄弟支线。

分享：下沉到分支

分享的原子单位是「分支」，不是整个进程——你真正协作的是一条带活对话的执行线。「分享进程」会有歧义（哪条分支？）。 分享一条分支 = 给对方这条执行线的对话 + 状态 + 它自己的存档点历史（含岔开前的共享祖先），但看不到兄弟支线。分两档权限：

角色能做什么
owner创建者。读写 + 分享 / 改权 / 撤回。
editor可读 + 可发消息；不能再分享。
viewer只读：看对话与目录，不能发消息。
  • 分享面板（点对话区 ⤴ 分享）：列出这条分支已分享给谁、各是什么角色，可加人（选 editor/viewer）、切换角色、移除——owner 在这里统一管理。同进程的其它分支不受影响。
  • 对方无需接收：分享后这条分支直接出现在对方的列表里，不用同意；但对方会在消息中心收到一条「X 把分支分享给你」。
  • 随时改 / 撤：owner 可把 editor 降成 viewer、或移除；被移除后对方列表里这条立即消失。
  • 被共享方不能再分享：只有 owner 看得到分享按钮。
待定 · sharee 开支线归谁 被分享方（editor）若从这条分支再开支线，新支线落在同一个进程（版本树）里——但它归谁所有、能不能反过来分享，是分享 × 分支交叉出来的开放问题（倾向：新支线继承进程 owner，sharee 只在其上有 editor 权）。尚未定。
底层为什么简单 所有用户的输入都先到控制平面再下发沙箱，底层 harness 不区分谁发的（都算「用户」）。所以多人同写不需要复杂协调，详见 控制平面 API · 共享与权限。

消息中心

仿 Windows 11：窗口底部状态栏右下角一个铃铛（带未读角标），点开是通知面板；有新事件时还会在右下角弹出 toast。

  • 会收到什么：有人把进程分享给你；你参与的进程有了新回复（且你没在看它）。
  • toast：右下角短暂弹出，点一下直接跳到对应进程（attach）。正在看的那个进程的新回复不再弹（你已经在看了）。
  • 面板：点铃铛展开通知列表，可标记已读 / 全部已读；角标显示未读数。
对应端点见 控制平面 API · 消息中心。
