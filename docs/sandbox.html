沙箱与镜像

沙箱是进程的临时算力——可销毁、可替换。aprog 当前用 Daytona，但从第一行代码起就把它关在一个 provider 抽象后面，因为之后还要接别的厂商。这页讲清楚两件事：怎么把厂商差异隔离干净， 以及一个 aprog 沙箱镜像里到底烘了什么、怎么烘。

沙箱是可替换层：两半抽象

调研多家沙箱厂商后有一个清晰结论——运行时控制面各家几乎一样，唯独「烘镜像」四种模型互不兼容。 所以抽象的正确切法是把它劈成两半，中间只用一个不透明的 ImageRef 衔接：

SandboxProvider 运行时 create / exec / openPty / inject·extractDir / hibernate·wake / destroy。各厂商高度一致，差异由 caps 能力位吸收。
ImageRef←
ImageBaker 烘镜像 BakeSpec → ImageRef。厂商间最不能统一的一步，单独成接口、按策略分流（见下）。
运行时那半统一；烘镜像这半分流。SandboxProvider 不管镜像怎么来的，只认 ImageRef—— 这就是厂商隔离的边界。上层（进程编排）只从 sandbox/ 的单一出口取，永不直接 import 某个厂商。
为什么这样切 如果只抽象「运行时」而把烘镜像也塞进同一个接口，Daytona 的声明式 addLocalDir 和 Morph 的「跑起来再存快照」 根本无法共用一个方法签名。把烘焙拎出去、用 ImageRef 解耦，运行时接口才能对几乎所有厂商保持干净。

能力位与分档

运行时虽相似，但几个关键点必须按能力声明、不能假设。每个 provider 暴露一组 caps，上层据此分流：

能力位含义谁有上层据此做什么
pty交互式 PTY——harness 桥接靠它Daytona ✓ · E2B ✓；Modal/Cloudflare 缺能不能直接桥接交互式引擎
memorySnapshot内存级挂起恢复E2B ✓ · Fly ✓ · Morph ✓；Daytona ✗决定 hibernate 走哪种形态（见下）
egressAllowlist出站白名单多数支持capability 访问企业系统时按 scope 放行

能力位把厂商映射成几条隔离档位，一套抽象多档并存：

档位provider隔离为什么是它
默认档Daytona（现状）容器级 + gVisor烘镜像最全（声明式 addLocalDir）、PTY + TS SDK 齐，冷启动快
强隔离档E2BFirecracker microVM硬件级隔离 + 内存快照 ~1s 唤醒；与 Daytona API 形状最像，是验证抽象的「第二家靶子」

两态休眠

进程休眠时要让出沙箱。怎么让出，取决于 caps.memorySnapshot—— hibernate() 因此是两态的，产物对调用方不透明，wake() 按同形态还原：

extracted 无内存快照 Daytona 这类：tar-out 进程目录成平台侧快照 → 销毁沙箱。wake 时起新沙箱、tar-in 注回。靠 state 协议 resume。
memory-snapshot 有内存快照 E2B/Fly/Morph 这类：内存级 pause，连进程/变量一起冻住，~1s 唤醒。可省去 tar-out。
同一个 hibernate/wake 接口、两种形态。注意：即便有内存快照，进程的权威状态仍在进程目录里 （见 进程目录与状态）——内存快照只是省了一次导出，不改变「状态不依赖对话」这条铁律。

镜像里烘了什么

一个 aprog 沙箱镜像（snapshot）服务所有进程——Daytona 的模型是「一个 snapshot 起 N 个 sandbox」，没有 per-process 镜像。 所以镜像内容要按变化频率分三层，per-process 的东西绝不烘进去：

① 基础镜像（烘进去）OS + Bun + 引擎运行时（Claude Agent SDK / Codex）+ capability 依赖。慢变，跟平台 release 走。
② driver（烘进去）@aprog/driver 编成单文件可执行，落 /opt/aprog/bin/。改得勤——prod 钉版本烘入，dev 用 bind-mount 盖掉免重烘。
③ 每进程注入（spawn 时，不烘）进程目录三件套 meta/input/state + auth 凭据 + program(skill) 文件。每进程不同。
引擎运行时一定烘（重、稳，还能暖 layer cache 让冷启动快）；per-process 的一定注入（它们私有于进程/用户）。

烘焙：四模型与我们的主路径

「怎么把东西弄进镜像」是厂商间最不能统一的地方——四种互不兼容的模型。ImageBaker 因此只定一个统一输入 BakeSpec 和不透明产物 ImageRef，具体交给策略实现：

策略代表厂商形态binary 注入怎么落
declarativeDaytona · E2B 2.0 · Modal链式 SDK builderaddLocalDir(staging/bin → /opt/aprog/bin)，最贴
dockerfileFly · Northflank · Cloudflare · Kata 自托管生成 Dockerfile + 推 registry自己写 COPY
snapshotMorph（Infinibranch）跑起来再存快照boot → exec 装 → save，没有 build-time 文件 API（最异类）

aprog 主路径走 declarative（Daytona 首个实现）：镜像定义就是仓库里一段 TS、和 control-plane 同栈同类型， 不用自建 registry，命名 snapshot 永久缓存。代价是多语言工具的编译必须在镜像外先做完——这恰好把 Go/Rust/Python 的 toolchain 挡在镜像之外，更干净。流程：

capabilities/*多语言工具源码 + 清单
编成 amd64→
staging/bin/语言无关的「成品架」
addLocalDir→
snapshot/opt/aprog/bin 全在 PATH
addLocalDir 把本地 staging/ 作为构建上下文上传到 Daytona 云端 runner 构建（无需本地 Docker）， 所以产物必须构建时已在本地——这正是 declarative 策略要求「编译在外」的原因。 capability 怎么组织见能力清单（编写中）。

约束与版本

下面几条来自 Daytona 文档实证，必须内化进设计（不是假设）：

  • amd64 only——所有 capability / driver 必须编成 linux/amd64，写进构建约定。
  • tag 不能用 latest，且 Daytona 没有内置 content-hash / 不可变 tag——所以「capability 集合 content-hash → tag」要我们自己算，用作 snapshot 名（aprog-sandbox:<sha12>）。命名 snapshot 永久缓存正好承接。
  • 资源（cpu/mem/disk）烘进 snapshot——属于镜像定义的一部分，从命名 snapshot 起 sandbox 时不再逐个改。
  • 镜像大小 / 注入 context 上限 Daytona 未公布——capability binary 多了镜像会变大，得早测。
dev / prod 对称 dev 期 daytona.create({ image }) on-the-fly 构建（24h 缓存），改完工具重跑即生效、不注册 snapshot； prod 才 snapshot.create 按 content-hash 命名注册永久 snapshot，可复现、冷启动稳。这和 driver 的 dev/prod 烘法完全对称。
版本握手 driver 烘在镜像里、control-plane 在 VM 上，两者共用 protocol。协议升版后可能错配—— 所以 attach 时要做一次版本握手：driver 上报 protocol 版本，control-plane 校验兼容、不兼容就拒绝并提示「镜像该更新了」。
