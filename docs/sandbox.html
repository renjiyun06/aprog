沙箱与镜像

沙箱是进程的临时算力——可销毁、可替换。aprog 当前用 Daytona，但从第一行代码起就把它关在一个 provider 抽象后面，因为之后还要接别的厂商。整层独立成包 @aprog/sandbox， 与 control-plane / driver / protocol 平级。这页讲清楚三件事：怎么把厂商差异隔离干净、 一个 aprog 沙箱镜像里到底烘了什么怎么烘，以及「打镜像」为什么是构建期的事、不归运行时服务器。

沙箱是可替换层：两半抽象

调研多家沙箱厂商后有一个清晰结论——运行时控制面各家几乎一样，唯独「烘镜像」四种模型互不兼容。 所以抽象的正确切法是把它劈成两半，中间只用一个不透明的 ImageRef 衔接：

SandboxProvider 运行时 create / destroy，仅此。纯生命周期、不碰文件——文件全走 DriverChannel。各厂商高度一致。
ImageRef←
ImageBaker 烘镜像 BakeSpec → ImageRef。厂商间最不能统一的一步，单独成接口、按策略分流（见下）。
运行时那半统一；烘镜像这半分流。SandboxProvider 不管镜像怎么来的，只认 ImageRef—— 这就是厂商隔离的边界。上层只从 @aprog/sandbox 的单一出口取，永不直接 import 某个厂商。
为什么这样切 如果只抽象「运行时」而把烘镜像也塞进同一个接口，Daytona 的声明式 addLocalDir 和 Morph 的「跑起来再存快照」 根本无法共用一个方法签名。把烘焙拎出去、用 ImageRef 解耦，运行时接口才能对几乎所有厂商保持干净。

包边界：谁依赖、烘焙在哪一侧

@aprog/sandbox 是 A 平面的全部，独立成包。它管四件事，其中前两件是运行时用的、后两件是构建期用的：

打镜像是构建期的事，不归运行时服务器 运行中的 control-plane 在 spawn/wake 热路径上只消费一个已烘好的命名 ImageRef，从不现场烘镜像。 prod 下镜像由 CI 调 aprog-bake 预烘成命名 snapshot；dev 下才 on-the-fly 现烘。 bake 与 provider 共用厂商 SDK 与 ImageRef，所以同包内聚；但由 CLI / CI 触发、而非服务器在请求里调用—— 这正是把它从 control-plane 抽出来的理由。

厂商分档

运行时接口本身极简（见上：create / destroy），各厂商干净覆盖。真正的差异在 隔离强度、烘镜像模型、冷启动速度——据此把厂商分成几档，一套抽象多档并存，按场景选：

档位provider隔离为什么是它
默认档Daytona（现状）容器级 + gVisor烘镜像最全（声明式 addLocalDir）、TS SDK 齐，冷启动快
强隔离档E2BFirecracker microVM硬件级隔离；与 Daytona API 形状最像，是验证抽象的「第二家靶子」
自托管 / on-premNorthflank（Kata / Cloud Hypervisor）可选 microVM / gVisor成熟 BYOC，企业私有部署；也避开 Daytona 自托管的 AGPL 传染
分支探索档Morph（Infinibranch）microVM~250ms 全 VM 状态分支，适合并行 agent / tree-of-thought
不再有 caps 能力位 早期接口带一组 caps（pty / memorySnapshot / egressAllowlist）做能力分流。现已全部移除—— 因为 hibernate 统一走「检查点 + destroy」（不依赖内存快照）、harness 桥接是 driver 内部的事（不依赖 provider pty）、 出站白名单等 capability 体系尚未落地。需要时再按需加回（YAGNI）。

休眠 = 让出全部资源 = destroy

这里说的休眠是进程的休眠，目的就是把沙箱资源全部让出。 所以它直接 destroy 沙箱——而不用厂商的 stop：stop 虽免 compute，却仍按 disk 计费， 而进程休眠常是用户隔很久才回来，留着盘只是白烧钱。

hibernate确保末次 commit 已 git push 到 CP remote → provider.destroy。厂商成本归零。
检查点在 CP→
wakeprovider.create → git clone --depth=1 + checkout <ref> → harness resume。
耐久真相是 CP 托管的 bare git repo（运行中持续 git push），不依赖任何厂商热令牌。 天然跨厂商——迁移就是「在厂商 Y 起沙箱后 git clone 同一个 remote」。完整模型见 进程状态存储（Git 模型），语义见 进程目录与状态。
费用：running 计 compute+disk；destroy 后归零。stop（免 compute、仍计 disk）/ archive（冷盘更便宜）对我们没意义——耐久不靠厂商留盘，靠 git remote。

镜像里烘了什么

一个 aprog 沙箱镜像（snapshot）服务所有进程——Daytona 的模型是「一个 snapshot 起 N 个 sandbox」，没有 per-process 镜像。 所以镜像内容要按变化频率分三层，per-process 的东西绝不烘进去：

① 基础镜像（烘进去）OS + Bun + 引擎运行时（Claude Agent SDK / Codex）+ capability 依赖。慢变，跟平台 release 走。
② driver（烘进去）@aprog/driver 编成单文件可执行，落 /opt/aprog/bin/。改得勤——prod 钉版本烘入，dev 用 bind-mount 盖掉免重烘。
③ 每进程注入（spawn 时，不烘）进程目录三件套 meta/input/state + auth 凭据 + program(skill) 文件。每进程不同。
引擎运行时一定烘（重、稳，还能暖 layer cache 让冷启动快）；per-process 的一定注入（它们私有于进程/用户）。

烘焙：四模型与我们的主路径

「怎么把东西弄进镜像」是厂商间最不能统一的地方——四种互不兼容的模型。ImageBaker 因此只定一个统一输入 BakeSpec 和不透明产物 ImageRef，具体交给策略实现：

策略代表厂商形态binary 注入怎么落
declarativeDaytona · E2B 2.0 · Modal链式 SDK builderaddLocalDir(staging/bin → /opt/aprog/bin)，最贴
dockerfileFly · Northflank · Cloudflare · Kata 自托管生成 Dockerfile + 推 registry自己写 COPY
snapshotMorph（Infinibranch）跑起来再存快照boot → exec 装 → save，没有 build-time 文件 API（最异类）

aprog 主路径走 declarative（Daytona 首个实现）：镜像定义就是包里一段 TS、同栈同类型， 不用自建 registry，命名 snapshot 永久缓存。BakeSpec 由政策层 assembleSpec 组装、aprog-bake CLI 驱动， provider 无关的那部分（base/engines/localBins/env）逐字段映射到各厂商 builder（Daytona addLocalDir / E2B copy）。 代价是多语言工具的编译必须在镜像外先做完——这恰好把 Go/Rust/Python 的 toolchain 挡在镜像之外，更干净。流程：

capabilities/*多语言工具源码 + 清单
编成 amd64→
staging/bin/语言无关的「成品架」
addLocalDir→
snapshot/opt/aprog/bin 全在 PATH
addLocalDir 把本地 staging/ 作为构建上下文上传到 Daytona 云端 runner 构建（无需本地 Docker）， 所以产物必须构建时已在本地——这正是 declarative 策略要求「编译在外」的原因。 capability 怎么组织见能力清单（编写中）。

厂商分叉：策略持骨架，ImageBuilder 填方言

baker 按构建模型分文件，不按厂商分——declarative 一个策略服务 Daytona / E2B 2.0 / Modal。 那同属 declarative 的几家，差异（addLocalDir vs copy、snapshot.create vs build）在哪里收？ 答案是再下一层的 ImageBuilder 接缝：DeclarativeBaker 自己不认识任何厂商， 只懂 declarative 该按什么顺序拼、注入到哪个约定目录、怎么命名； 真正碰厂商 SDK 的，只有各自的 ImageBuilder 实现。

DeclarativeBaker一份步骤骨架，厂商无关
注入 newBuilder()→
DaytonaImageBuilderaddLocalDir · snapshot.create → id=snapshot 名
E2BImageBuildercopy · build({alias}) → id=template alias
策略横向少数几个（按模型）；厂商在每个模型内部靠「注入不同 ImageBuilder」纵向分叉。共性沉在骨架里复用一次。

接缝就是一张厂商无关的「动词表」，外加一个产出不透明 ImageRef 的 finalize：

// ① 分叉接缝：厂商无关的动词表
interface ImageBuilder {
  base(image: string): void;
  run(cmd: string): void;
  copyDir(local: string, dest: string): void;
  env(vars: Record<string, string>): void;
  finalize(name: string | undefined, res: Resources): Promise<ImageRef>;
}

// ② 策略：只懂 declarative 的步骤顺序，不碰任何 SDK
class DeclarativeBaker implements ImageBaker {
  constructor(private newBuilder: () => ImageBuilder) {}   // ← 注入，不 new 具体厂商
  async bake(spec: BakeSpec) {
    const b = this.newBuilder();
    b.base(spec.base);
    for (const e of spec.engines)   b.run(e);
    for (const d of spec.localBins) b.copyDir(d, '/opt/aprog/bin');  // 注入约定
    for (const c of spec.commands)  b.run(c);
    if (spec.env) b.env(spec.env);
    return b.finalize(spec.name, spec.resources);            // 共享骨架到此为止
  }
}

两家各填一个 ImageBuilder，跟自己的 provider 同文件（都碰同一套厂商 SDK）。动词逐字对应：

ImageBuilder 动词Daytona（@daytonaio/sdk）E2B（e2b 2.0）
baseImage.base(x)Template().fromImage(x)
run.runCommands(cmd).runCmd(cmd)
copyDir.addLocalDir(local, dest).copy(local, dest)
env.env(vars).setEnvs(vars)
finalizesnapshot.create({name, image, resources}) → id=snapshot 名build(t, {alias:name}) → id=template alias

政策层 pickBaker 用一行把 provider 映射到「策略 + 对应 builder 工厂」：

function pickBaker(provider: ProviderId): ImageBaker {
  switch (provider) {
    case 'daytona': return new DeclarativeBaker(() => new DaytonaImageBuilder());
    case 'e2b':     return new DeclarativeBaker(() => new E2BImageBuilder());
    case 'morph':   return new SnapshotBaker(/* … */);
  }
}
加一家厂商有多便宜 同模型新厂商（如 Modal，也是 declarative）= 只写一个 ModalImageBuilder（填那 5 个动词）+ pickBaker 加一行， DeclarativeBaker 一字不改。不同模型的厂商（只收 Dockerfile）= 走 DockerfileBaker 那条线，与 declarative 无关。 这就是 bakers/ 永远只有 3~4 个文件、不随接入厂商数膨胀的原因。
落地状态：Daytona 这条已实现 ImageBuilder 接缝 + DeclarativeBaker + DaytonaImageBuilder 已落代码、真实 Daytona 端到端验证通过（烘出 active snapshot 后自清理；见集成测试 daytona-builder.integration.test.ts）。 DockerfileBaker / SnapshotBaker / E2BImageBuilder 仍是桩。当初列的两处不对称，实现时已拍板： ① on-the-fly → 改为 ImageRef-always——Daytona 的 snapshot.create 的 name 必填、无匿名镜像，所以 bake 恒产出命名 ImageRef，provider.create 契约两家一致；dev/prod 之分退化为缓存/可变性策略。 ② resources 归属 → 归 bake——随 snapshot.create({resources}) 烘进 snapshot（贴 Daytona 模型）。

代码组织：按「耦合对象」放实现者

@aprog/sandbox 的文件布局遵循一条原则：接口放包根（厂商无关契约），实现者按「它绑了谁」落目录。 决定一个文件难不难改、能不能脱离网络单测的，是它耦合了哪家厂商的 SDK——所以「跟你耦合的东西放一起」。 

src/
  ── 接口/契约（包根，厂商无关）──
  provider.ts      SandboxProvider          (create/destroy)
  baker.ts         ImageBaker + ImageBuilder
  types.ts         ImageRef / Resources / SandboxHandle / ProviderId
  errors.ts        SandboxError 体系 + mapDaytonaError
  config.ts        DaytonaConfig 等 provider 入参契约

  ── 实现者（按耦合对象分目录）──
  providers/       ← 按厂商分（碰各家 SDK）
    daytona.ts          DaytonaProvider       implements SandboxProvider
    daytona-builder.ts  DaytonaImageBuilder   implements ImageBuilder   ← 注意在这
    e2b.ts              E2BProvider           (桩)
  bakers/          ← 按构建模型分（厂商无关策略）
    declarative.ts      DeclarativeBaker      implements ImageBaker
    dockerfile.ts       DockerfileBaker       (桩)
    snapshot.ts         SnapshotBaker         (桩)

  ── 装配 / 出口 ──
  bake.ts          assembleSpec / contentHash / pickBaker / bake   （缝合两轴）
  cli.ts           aprog-bake 入口
  index.ts         单一公共出口

providers/ 与 bakers/ 是两条不同的分类轴，原因前文已述：运行时是 per-厂商，烘焙策略是 per-构建模型。接口 → 实现者 → 落点：

接口实现者落在为什么
SandboxProviderDaytonaProvider / E2BProviderproviders/<厂商>.ts运行时 per-厂商，各自抱自己的 SDK
ImageBakerDeclarativeBaker / DockerfileBaker / SnapshotBakerbakers/<模型>.ts策略 per-构建模型、厂商无关；一个 declarative 服务多家
ImageBuilderDaytonaImageBuilderproviders/ daytona-builder.ts名义属烘焙，但绑 Daytona SDK → 跟厂商放，不跟策略放
最能说明问题的一个：DaytonaImageBuilder 按名字它是 builder、属烘焙，似乎该进 bakers/；按耦合它绑死 Daytona SDK，所以进 providers/。 我们按耦合放——于是 bakers/declarative.ts 保持一行厂商 SDK 都不碰（纯策略骨架），Daytona 的方言全锁进 providers/daytona*.ts。 ImageBuilder 这个接口留在 baker.ts（它是烘焙契约），但它的 Daytona 实现跟 Daytona 待一起。

两轴在装配层 bake.ts 的 pickBaker 交汇：case 'daytona' 取 bakers/ 的策略、注入 providers/ 的 builder。 收益：任一厂商的 SDK 改动只震荡 providers/<它>，策略与契约纹丝不动。

约束与版本

下面几条来自 Daytona 文档实证，必须内化进设计（不是假设）：

  • amd64 only——所有 capability / driver 必须编成 linux/amd64，写进构建约定。
  • tag 不能用 latest，且 Daytona 没有内置 content-hash / 不可变 tag——所以「capability 集合 content-hash → tag」要我们自己算，用作 snapshot 名（aprog-sandbox:<sha12>）。命名 snapshot 永久缓存正好承接。
  • 资源（cpu/mem/disk）烘进 snapshot——属于镜像定义的一部分，从命名 snapshot 起 sandbox 时不再逐个改。
  • 镜像大小 / 注入 context 上限 Daytona 未公布——capability binary 多了镜像会变大，得早测。
命名：ImageRef-always BakeSpec.name 恒有值（aprog-sandbox:<sha12>，由 contentHash 自算）——Daytona 无匿名镜像，且让 provider.create 的 ImageRef 契约统一。 snapshot.create 据此注册命名 snapshot，可复现、冷启动稳。dev/prod 不在「有没有 name」，而在缓存/可变性策略：同一 content-hash 重烘即幂等覆盖，prod 视作不可变。
版本握手 driver 烘在镜像里、control-plane 在 VM 上，两者共用 protocol。协议升版后可能错配—— 所以 attach 时要做一次版本握手：driver 上报 protocol 版本，control-plane 校验兼容、不兼容就拒绝并提示「镜像该更新了」。
