沙箱与镜像

沙箱是进程的临时算力——可销毁、可替换,藏在一个 provider 抽象后面(当前 Daytona,之后还要接别家)。 镜像则不是一层抽象,而是仓库里一个目录:按「名 / 版本」组织、每个版本自带打镜像脚本与材料。 而程序依赖某个版本的镜像。这页讲三件事:provider 怎么把厂商差异关干净、镜像怎么在仓库里组织与版本化、程序如何依赖镜像。

沙箱:唯一要抽象的是「运行时」

调研多家沙箱厂商后有个清晰结论——运行时控制面各家几乎一样(起一个、销毁一个,文件全走 DriverChannel)。 所以只需把运行时抽象掉、关进 provider 后面;镜像怎么来的,运行时只认一个不透明 ImageRef,不关心。

SandboxProvider 唯一的抽象 create / destroy,仅此。纯生命周期、不碰文件。各厂商高度一致 → 抽象薄、成本低,这正是「运行时换厂商」可行的地方。
吃一个 ImageRef←
镜像 不是抽象,是目录 仓库里 images/<名>/<版本>/ 一个目录,自带 bake.ts。烘出来在厂商侧注册成 snapshot,运行时拿它的 ImageRef 起沙箱(见 下文)。
SandboxProvider 是上层唯一从 @aprog/sandbox 取的东西——这就是厂商隔离的边界,永不直接 import 某个厂商 SDK。 「镜像怎么烘」不在这个抽象里:它是仓库里的目录 + 脚本,构建期才跑,运行时不参与。
为什么镜像不做成抽象 早期版本曾把「烘镜像」也抽象成一套 ImageBaker × ImageBuilder 的策略/方言接缝,想一份定义烘到任意厂商。 但现实是只用 Daytona、其余全是桩——那是从 0 个真实第二案例提前抽象,典型的过度设计,已拆除。 镜像回归成「一个目录 + 一段直调厂商 SDK 的 bake.ts」。等真出现第二家厂商 / 第二种构建模型,再从两个具体脚本里提炼骨架(见 烘焙)。

包边界:精瘦运行时库,镜像不在包里

@aprog/sandbox 是一个被别的模块 import 的库,只剩运行时那一半:

职责内容
provider 运行时SandboxProvider(create/destroy)+ 各厂商实现(DaytonaProvider / E2BProvider)
入参契约 / 句柄DaytonaConfig 等 provider 配置、SandboxHandle、不透明 ImageRef
错误归一SandboxError 体系 + mapDaytonaError(把厂商 SDK 异常归一)
@daytonaio/sdk · @aprog/log厂商 SDK(下沉到此,不污染 control-plane)+ 日志
依赖→
@aprog/sandbox只剩运行时 provider 库
被依赖←
@aprog/control-plane仅 import 两个 type:DaytonaConfig(config.ts)、SandboxHandle(driver-channel.ts)
耦合面就两处 import type,且不再直接抱厂商 SDK。
镜像材料为什么不能放进这个包 这个包是host 侧库代码,跑在 control-plane 上、被别的模块 import。而镜像里的工具是沙箱侧材料—— 打进镜像后在沙箱里被 harness 调,没有任何 host 模块会 import 它。两者是不同物种: 把材料塞进被 import 的库,等于让 control-plane 的依赖闭包背上一堆它永不使用的工具源码。 所以镜像目录 images/ 放在仓库顶层、不是 package(见 代码组织)。

厂商分档

运行时接口本身极简(create / destroy),各厂商干净覆盖。差异在 隔离强度、冷启动速度、烘镜像方式——据此分档,一套抽象多档并存,按场景选:

档位provider隔离为什么是它
默认档Daytona(现状)容器级 + gVisor声明式构建最全(addLocalDir)、TS SDK 齐、冷启动快
强隔离档E2BFirecracker microVM硬件级隔离;与 Daytona API 形状最像,是验证抽象的「第二家靶子」
自托管 / on-premNorthflank(Kata / Cloud Hypervisor)可选 microVM / gVisor成熟 BYOC,企业私有部署;避开 Daytona 自托管的 AGPL 传染
分支探索档Morph(Infinibranch)microVM~250ms 全 VM 状态分支,适合并行 agent / tree-of-thought
没有 caps 能力位 早期接口带一组 caps(pty / memorySnapshot / egressAllowlist)做能力分流,现已全部移除—— hibernate 统一走「检查点 + destroy」(不依赖内存快照)、harness 桥接是 driver 内部的事、出站白名单等体系尚未落地。需要时再按需加回(YAGNI)。

休眠 = 让出全部资源 = destroy

休眠是进程的休眠,目的就是把沙箱资源全部让出。 所以它直接 destroy 沙箱——而不用厂商的 stop:stop 虽免 compute,却仍按 disk 计费, 而进程休眠常是用户隔很久才回来,留着盘只是白烧钱。

hibernate确保末次 commit 已 git push 到 CP remote → provider.destroy。厂商成本归零。
检查点在 CP→
wakeprovider.create(按程序依赖的镜像)→ git clone --depth=1 → harness resume。
耐久真相是 CP 托管的 bare git repo(运行中持续 git push),不依赖任何厂商热令牌。 天然跨厂商——迁移就是「在厂商 Y 起沙箱后 git clone 同一个 remote」。完整模型见 进程状态存储(Git 模型)。
费用:running 计 compute+disk;destroy 后归零。stop / archive 对我们没意义——耐久不靠厂商留盘,靠 git remote。

镜像 = 仓库里一个目录

在我们这里,「镜像」就是一个目录;它下面是各个版本,每个版本也是一个目录,装着这个镜像需要的一切 + 打镜像脚本: 

images/
  <镜像名>/              ← 一个镜像 = 一个目录;名字就是目录名
    <版本>/              ← 一个版本 = 一个自包含子目录
      bake.ts            ← 这个版本怎么烘(直接调 @daytonaio/sdk)
      …这个镜像需要的材料
    <另一个版本>/
      …
文件系统即 registry不在代码里枚举镜像名、不要 registry 模块。发现 = ls images/<名>/;加一个镜像 = 新建目录,不在任何地方注册。
版本 = 目录名人定、可读、可排序——正是「每种镜像都有版本」要的。一个版本发布后不可变(进程钉的就是它)。
一个镜像起 N 个沙箱Daytona 模型是「一个 snapshot 起 N 个 sandbox」,没有 per-process 镜像。但也不是「一个全宇宙镜像装下所有工具」——镜像有区分,程序按需依赖。

镜像内容按变化频率分层,per-process 的东西绝不烘进去:

① 基础 + 工具(烘进去)OS + Bun + 引擎运行时(Claude Agent SDK / Codex)+ 这个镜像要带的工具。慢变,随镜像版本走。
② driver(烘进去)@aprog/driver 编成单文件可执行,落 /opt/aprog/bin/。prod 钉版本烘入,dev 用 bind-mount 盖掉免重烘。
③ 每进程注入(spawn / wake 时,不烘)进程目录、auth 凭据、program(skill) 文件。每进程不同,经 DriverChannel / git clone 进来。

烘焙:跑那个版本目录里的 bake.ts

打镜像就是跑某版本目录里的 bake.ts——它直接调 @daytonaio/sdk 把 base + 引擎 + 工具拼成 Image、 snapshot.create 注册成命名 snapshot,返回 ImageRef。一个薄 CLI 把它包成命令: 

aprog-bake <镜像名> <版本>     # = 进 images/<镜像名>/<版本>/ 跑 bake.ts
打镜像是构建期 / CLI 的事,不归运行时服务器 运行中的 control-plane 在 spawn/wake 热路径上只消费一个已烘好的命名 ImageRef,从不现场烘镜像。 prod 下镜像由 CI 调 aprog-bake 预烘;aprog-bake 跑完即退,产物(snapshot)寄存在厂商侧,服务器之后只按名字引用。
为什么没有「策略 / 厂商方言」那一层 每个 bake.ts 都写死 Daytona,不绕任何 ImageBaker / ImageBuilder 接缝。 代价是「一份定义烘到任意厂商」这个投机选项没了——但镜像产物本来就不跨厂商、换厂商得重烘,且我们只用 Daytona,所以没真损失。 真要支持第二家厂商时,会从两个具体 bake.ts 里抽公共骨架(从 2 个真实案例提炼),而不是现在从 0 个猜。

工具 = 镜像的材料,各自独立

一个镜像带的额外工具——飞书 CLI、pandoc、某个领域专用命令——是这个镜像的材料。它们不共享、不集中编目:

工具可能本身就是一个项目复杂到有自己的 repo / 构建 / 生命周期。把它矮化成「共享目录下一个版本子目录」是低估了它,所以不做工具中心目录。
由 bake.ts 自己弄进来工具活在自己的 repo,bake.ts 负责引用 / 构建它进镜像(落 /opt/aprog/bin)。怎么编、怎么拉,是工具自己的事。
我们这层只管镜像不认识、不编目、不命名工具——「镜像管理」对我们就是 images/ 这个目录 + 各版本的 bake.ts,没有第二套管理系统。
多语言工具的编译在镜像外先做完(这把 Go/Rust/Python 的 toolchain 挡在镜像之外,更干净);bake.ts 只注入编好的 amd64 产物。

程序依赖镜像

程序不能跑在任意镜像上——它需要带着它那套工具的镜像。用 OS 比喻:program 是「二进制」,image 是它要跑的「运行时 / 用户态」。 所以是一条版本精确的依赖边:program@版本 → image@版本。

processPCB 钉死 program 版本
钉→
program@版本声明依赖 image: 名@版本
声明依赖→
image@版本→ 解析成 ImageRef
进程用哪个镜像不是独立决策,而是透过 program 版本派生的。PCB 已钉死 program_version,程序版本不可变 → 它的镜像依赖也不可变 → 长休眠进程 wake 时永远解析出同一个镜像,没有漂移。
镜像共享、程序声明依赖 镜像在共享的 images/ 目录里(一个 base 镜像可被多个程序复用);程序在自己的 meta 里声明 image: 名@版本 指过去。 依赖落库在 program_versions(image_name + image_version 两列)。 「程序跟镜像一起发布」是工作流(加新镜像版本 → 把程序依赖指过去,常在一次改动里),不是结构绑定。
两个独立版本化的东西,靠校验保持一致
  • 边显式声明 + 版本不可变 → 这条边是稳定的。
  • 发布期校验:发一个 program 版本时,CI 检查它依赖的 image@版本 在 images/ 里确实存在 / 已烘 → 悬空依赖是发布错误,不是运行时惊吓。
  • GC:一个镜像版本可退役 ⟺ 没有任何「被某进程依赖的 program 版本」还指着它(透过这条边传递)。

约束与版本

  • amd64 only——所有工具 / driver 必须编成 linux/amd64,写进构建约定。
  • 版本 = 目录名(人定、不可变)。content-hash 当初是为补「没有可读版本」而自算的;现在目录名即版本即身份,content-hash 退化为可选的完整性校验,不再是镜像的名字。
  • 资源(cpu/mem/disk)随 snapshot.create 烘进 snapshot——属镜像定义的一部分,起 sandbox 时不再逐个改。
  • 镜像大小上限 Daytona 未公布——工具多了镜像会变大,得早测。
版本握手 driver 烘在镜像里、control-plane 在 VM 上,两者共用 protocol。协议升版后可能错配—— 所以 attach 时做一次版本握手:driver 上报 protocol 版本,control-plane 校验兼容、不兼容就拒绝并提示「镜像该更新了」。

代码组织

packages/sandbox/          @aprog/sandbox —— 精瘦运行时库(被 control-plane import)
  src/
    provider.ts            SandboxProvider (create/destroy)
    providers/daytona.ts   DaytonaProvider
    types.ts               ImageRef / Resources / SandboxHandle / ProviderId
    errors.ts              SandboxError 体系 + mapDaytonaError
    config.ts              DaytonaConfig 等 provider 入参契约
    cli.ts                 aprog-bake:按「名+版本」找到目录跑 bake.ts
    index.ts               单一公共出口(只导出运行时那半)

images/                    ← 仓库顶层:镜像目录(不是 package,无人 import)
  <镜像名>/<版本>/bake.ts  + 材料

(工具各自活在自己的 repo,由 bake.ts 引用 / 构建进来)
为什么这样切 packages/sandbox 是 host 侧、被 import 的库;images/ 是沙箱侧材料、构建期才跑、无人 import——物种不同故分家(见 包边界)。 aprog-bake 留在 packages/sandbox 是因为它和 @daytonaio/sdk 依赖同包,bake.ts 直接 import 零接线。
