三层抽象

aprog 的运行时围绕三个互不知情的抽象组织：Program / Engine / Sandbox。 本页讲它们的分层、部署拓扑、三条通信链路，以及锁定这套架构的几条关键决策（ADR）。 各组件的具体职责见 组件清单。

概念分层

skill 接口 ▼↓
⚙️ Engine CPU · 可替换 底层执行引擎，Claude Agent SDK，未来可换 Codex 等
跑在 ▼↓
🧊 Sandbox PPIO 环境：装工具 + 给 Engine 跑的容器
互不知道对方实现，通过薄适配 Engine Driver 粘合。

三者互不知道对方的具体实现：Program 用统一的 skill 接口表达自己；Engine 提供 agent loop + 工具调用； Sandbox 提供文件系统 + 进程 + 网络。它们通过一层薄适配 Engine Driver 粘合。

  • 沙箱不对外暴露入站端口；所有交互靠沙箱内 driver 出站拨回 CP（B 平面），生命周期起停经 provider SDK（A 平面）。

三条通信链路

链路 1 · 前端 ↔ Control Plane

一般命令走 HTTP/JSON；事件流订阅走 SSE（from=<seq>），行分隔 JSON。鉴权用一套 token 每次请求带上。 下行是 turn.* / item.* / user / error 等 engine-neutral 事件，上行是用户输入。事件类型是 aprog 协议层的术语，不是某个 SDK 的术语。

链路 2 · Control Plane ↔ Sandbox（两平面）

见 沙箱交互界面。分两个正交平面：

  1. A · 资源平面：CP 经 provider SDK 起停沙箱（create/destroy/hibernate/wake）。不碰文件。
  2. B · 数据平面：沙箱内 driver 自启后持运行时注入的 bindToken 经 HTTPS 拨回 CP，建一条活满沙箱生命周期的常驻 DriverChannel——只剩事件流上行与输入下行两件实时事。检查点耐久与 wake 恢复改走 git（git push / clone 到 CP 托管 bare repo，见 进程状态存储），不再经此平面搬运文件。
  3. detach 只是摘掉一个订阅者，DriverChannel 不随之关；reattach 用游标 from=<seq> 补回间隙。

链路 3 · Driver ↔ Engine

进程内。Driver 把 Engine 作为子进程拉起：import { query } from '@anthropic-ai/claude-agent-sdk'，用 streaming-input 模式跑： prompt 是 CP 经 DriverChannel 下发的用户输入 async iterable，cwd 为进程目录， settingSources: ["project"] 加载进程目录里的 skill。消息从 iterator 出来，Driver 翻成 aprog 事件（局部定序，CP 落库盖 seq）。

关键决策（ADR）

ADR-1 · Control Plane 代理沙箱流，不让前端直连

所有前端到沙箱的数据经 Control Plane 中转。理由：单一鉴权面、provider 抽象不外漏、沙箱不暴露公网端口、攻击面小。 代价是 Control Plane 在每条流的热路径上，规模化后需要反代水平扩展，但架构不变。

ADR-2 · 沙箱内常驻 Driver，不裸跑 Engine

沙箱内常驻 claude-driver，自启后拨回 CP 建一条 per-sandbox 的 DriverChannel，跨 attach/detach 持续活、兜住 Engine 输出。 detach 后前端走了但 DriverChannel 不关、Engine 还在跑，产出得有东西兜并继续落库；reattach 要能补上间隙； “SDK 消息 → aprog 事件”的翻译在沙箱内做，让平台与引擎解耦。

ADR-3 · Engine 可替换，Program 不知道 Engine

Program 只表达“做什么”，不调用任何 engine-specific API；差异封在 Driver。每款引擎一个 <name>-driver 包， 对 aprog 协议的 wire format 固定。当前 MVP 只实现 claude-driver，codex-driver 等留 stub、接口先对齐。

ADR-4 · 会话归 aprog，引擎 session 不权威 已修订

本条已修订 早期 ADR-4 曾规定“引擎的 session.jsonl 进快照、为了 resume 接着对话”。该前提已被推翻。

现行决策：进程状态不依赖对话上下文。进程靠 state 程序库定义的状态文件 resume， 引擎冷启动后读状态即可接着干，绝不反向重建引擎上下文。因此：

  • aprog 用自己的协议自己存储会话（落进程目录，engine-neutral），见 事件流协议。
  • 引擎私有的 session.jsonl 降级为沙箱生命周期内的临时缓存，不进 aprog 快照、永不权威。
  • 换引擎不丢工作，因为会话归 aprog、状态在目录里。
