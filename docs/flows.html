生命周期流程

把组件串起来看一次进程的生老病死：spawn → attach → detach → hibernate → wake → 完成/kill（回到 hibernating）。 每一步谁调用谁、沙箱与检查点如何变化。耐久（检查点）走 git push、读文件走 git（见 进程状态存储（Git 模型）），DriverChannel（B 平面）只剩事件上行 + input 下行，A 平面只起停沙箱。

spawn · 新建进程

用户在窗口里点 ＋ 填个名字。Control Plane 只登记进程、存下初始输入，不写骨架、不起沙箱：

前端POST /proc {program, name}
↓
🧭 Control Plane
  • 验 token → User
  • 生成 pid（uuid 取前几位）
  • DB 插入 Process(state=spawned, sandbox_id=null)，钉死 skill_name + skill_version
  • eager 存下用户初始输入（耐久脊梁）
  • 返回 pid

此时进程休眠（没有沙箱）。状态：spawned。 注意：进程目录的骨架（meta.yml / input.md / execution-state/）由 harness 按 state 技能自己写， 平台不代笔——它只在首次跑起来时由 harness 落地（见 attach）。

attach · 关联沙箱并连上对话

第一次 attach（或从休眠 wake）会创建沙箱，driver 自启拨入、灌入 bootstrap、起 harness，事件随即经 DriverChannel 上行：

前端POST /proc/<pid>/attach
↓
🧭 Control Plane · 起沙箱（A 平面）
  • 验 token + owner
  • sandbox_id 为 null → provider.create 起新 沙箱（钉死版本的镜像）
  • 写 sandbox_id + sandbox_started_at（state→running）
driver 自启 · 持烘入凭证拨出 ▼↓
🔌 driver 拨入 · 建 DriverChannel
  • driver 拨向 CP + create-time 绑定 + 版本握手
  • CP 经握手下发 skill 闭包 + per-process auth + git deploy 凭证；driver 持凭证 git clone --depth=1 + checkout 铺进程目录（首跑则空 / 初始 input）。fs 与检查点不经 DriverChannel
  • 工作树由 git clone --depth=1 拉回最新检查点（默认检出 HEAD；首跑则是空 / 初始 input，见 耐久传输）
  • driver 拉起 harness；首次由 harness 按 state 技能写出骨架
  • harness 产出 → driver 局部定序 → CP 盖全局 seq 落库 → 扇出前端

前端的 attach 就是向 CP 订阅事件流（from=<seq>，SSE/WS 连的是 CP、不是沙箱）。 之后用户每输入一句：前端 → CP → DriverChannel → driver → 喂给引擎，引擎跑 agent loop， 消息回流 → driver 翻成 aprog 事件 → CP 落库 + 扇出渲染。详见 事件协议与会话。

detach · 离开但进程照跑

用户离开会话（关窗口/断网）。前端只是退订，沙箱与 DriverChannel 不受影响：

前端断开订阅
↓
🧭 Control Plane 摘掉一个订阅者
  • driver + harness 照常跑；DriverChannel 是 per-sandbox 常驻，不随 detach 关
  • 事件继续经 DriverChannel 上行、CP 继续落库
  • 更新 Process.last_attached_at

沙箱还在，引擎可能还在跑。重新连上就是 reattach：跳过沙箱创建，from=<seq> 用上次游标补回间隙事件。

hibernate · 释放沙箱

用户在进程行点“休眠”按钮，显式释放算力。state 早已随检查点 git push 到 CP remote，所以休眠很轻：

前端POST /proc/<pid>/hibernate
↓
🧭 Control Plane 让出算力
  • control('hibernate-prepare') → harness 在 quiescent 点落盘 + 末次检查点；确保末次 commit 已 git push 到 CP remote
  • provider.destroy → 释放全部资源（不用 stop：stop 仍计 disk 费）
  • sandbox_id = null → 语义上的 休眠

耐久真相是 CP 托管的 bare repo（已 git push 的检查点），destroy 后厂商成本归零。状态：休眠。引擎私有的 session.jsonl 随沙箱消失——无所谓，它不权威。

wake · 重新关联沙箱

用户在休眠进程点“唤醒”按钮。等同一次 attach：provider.create 起新沙箱、driver 拨入，工作树由 git clone --depth=1 拉回最新检查点（默认检出 HEAD，见 耐久传输）。

冷启动靠状态 新沙箱里引擎是空上下文。它读进程目录的状态文件（input_cursor / input.md / execution-state）重建工作记忆， 不重放历史对话。前端则照常铺出完整历史对话（读会话记录），对用户连续。原因见 唤醒靠状态，不靠对话。

完成 / kill · 回到休眠

程序执行完毕（或被用户 kill），Driver 收到 SessionEnd。没有终止态——跑完 / 被杀都只是回到 休眠，处理同 hibernate：

DriverSessionEnd
↓
通知 🧭 Control Plane 让出算力
  • 确保末次 commit 已 git push 到 CP remote（见 耐久传输）
  • provider.destroy 沙箱
  • sandbox_id = null, state = hibernating, last_active_at = now

回到 hibernating、状态永不删除——只是没有沙箱关联。目录、产物、会话记录都还在，新输入到来还能重新 attach 回 running。 状态机只有 spawned / running / hibernating、无终止态，快照/检查点生命周期的事实源归属见 数据模型。
