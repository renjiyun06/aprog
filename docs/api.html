控制平面 API

沙箱交互界面讲的是控制平面朝南跟沙箱说话；这页讲控制平面朝北跟 客户端说话。当前客户端只有一个：Web 前端（暂不做纯 CLI 实现）。一句话定调： 命令走 REST，实时事件流走 SSE，输入与交互回写也走 REST。 这页是这套接口的设计稿——只定契约、不写实现；拿不准的地方我直接标成「待确认」块，你在文档里逐处评论即可。

这页 vs 旧的 reference.html 参考 · HTTP 端点里那份速查是早于本轮模型收敛写的，已和现状冲突两处：① 它写实时流是 WebSocket、事件形如 agent.thinking；现状是 SSE + 协议里的 item.*/turn.* 事件并集。② 它有 POST /proc/<pid>/attach「关联沙箱」，但收敛后 driver 自启拨出、 一条 DriverChannel 活满沙箱整生命周期，attach 不再关联沙箱（见 待确认 · attach）。 本页确定后，reference.html 的速查应据此重生。

北面 / 南面

控制平面是两套接口的交汇点，方向相反、抽象也相反：

客户端 本页Web 前端。REST + SSE。多用户、带 token。
北 · REST 命令 + SSE 事件流↓
control-plane（VM 上）鉴权 · 进程编排 · 事件流中枢（盖全局 seq / resync / 扇出）· 检查点持久化。两面的中枢。
南 · SandboxProvider（A）+ DriverChannel（B）↓
沙箱 / driver 见 interaction.htmlA 平面起停沙箱；B 平面带内承载事件 / 输入 / 控制 / fs / bundle。
北面对客户端隐藏一切南面细节：客户端只跟 pid 与事件流打交道，看不到沙箱、provider、bindToken、检查点。 控制平面在运行时是透明中继（事件流 / fs 实时穿透到沙箱内的 driver），休眠时是持久化数据源（回放已落库的流、按检查点服务 fs）。

传输与通用形态

整套 API 的横切约定，逐条单列一行，便于你就某一条单独评论：

维度约定
命令 / 写REST（JSON over HTTP）。每个端点对应一个平台操作：ps / spawn / hibernate / wake / cat / ls。
实时事件流SSE：一条长连 GET …/stream，服务端推 协议 事件（见 事件流端点）。单向（server→client）。
反向（输入 / 中断）不复用 SSE，走独立的小 POST（输入与交互）。SSE 单向、这些是低频指令，拆开比塞进双工连接更简单。
鉴权每个请求带 token（header）；SSE 同样带 token 起连。失败 401。进程按成员授权（一对多，成员都可读写；见 Q9），非成员 403。
标识pid 由控制平面分配，全局唯一、跨休眠/唤醒不变（沙箱可换、pid 不换）。客户端只认 pid。
错误信封非 2xx 统一 { "error": { "code": "...", "message": "...", "retryable": bool } }。code 取自一套稳定枚举（待定，见 待确认 · 错误码）。
幂等create 类（spawn）支持 Idempotency-Key header，重复提交返回同一 pid，不重复起进程（见 待确认 · 幂等）。
版本路径前缀带版本，建议 /v1/…。本页端点表省略前缀。

生命周期命令语义

三个命令对应 process/lifecycle.ts 的 FSM。关键收敛：spawn 纯控制平面（只建记录、不碰沙箱）， 起沙箱的活全在 wake——首次运行与休眠后唤醒走完全相同的路径，唯一差别是灌入的 bundle（首次没有检查点，用 bootstrap）：

命令动作链终态
spawn分配 pid + 建进程目录（绑定 program + 初始 input）。纯控制平面，不碰沙箱。未运行
wakeprovider.create → driver 自启拨入 → pushBundle 灌入（首次=bootstrap：program+初始 input；之后=最新检查点）→ driver 拉起 harness（首次运行 / resume）running
hibernatecontrol('hibernate-prepare') + 末次检查点（pullBundle）→ provider.destroy未运行
前端「新建进程」= spawn + wake 点 ＋ 新建并提交，前端连发两个请求：POST /proc（spawn，落进程记录）→ POST /proc/:pid/wake（起沙箱跑起来）。 spawn 之所以单拆出来，是因为它只是控制平面里记一笔，瞬时完成；而 wake 才是那条会起沙箱、要等几秒的重操作。这也让「新建」与「休眠后再跑」共用同一个 wake。
待确认 Q2 · wake 同步还是异步？ spawn 只写控制平面记录，瞬时同步返回，无悬念。wake 含 provider.create（Daytona 实测约数秒），两种返回风格： (a) 同步——阻塞到 running 才返回；(b) 异步——立刻 202 返回过渡态，前端订阅 …/stream 或轮询 GET /proc/:pid 观察转 running。 我倾向 (b)：起沙箱的耗时不该占着 HTTP 连接，且前端本来就要连 stream。
待确认 Q3 · 进程状态集怎么定？ process/manager.ts 现在只有 running | hibernating。新模型里，spawn 后「已建未运行」和 hibernate 后「休眠」 对平台是不是同一个状态（都=无沙箱、可 wake，差别只在有没有检查点）？若是，状态集就两态：running / 未运行； 若否，需要区分 新建未运行 与 休眠。另外 Q2 选异步还要不要 waking 过渡态？这块归在生命周期讨论里一起定。
待确认 Q4 · 进程怎么「终结」？（无 kill / 无删除） 已去掉 kill，也没有删除进程的操作——进程目录与历史永不删。那两个问题待定： (a) 程序真的做完了（沙箱里 harness 自己退出）时，进程转成什么终态？自动 hibernate 保留现场？还是有个 done/exited 终态？ (b) 永不删除 → 进程目录无限累积，要不要 archive / GC 机制（仅清冷数据、不算「删进程」）？

事件流端点

GET /proc/:pid/stream?from=<seq> 是客户端看进程「实时画面」的唯一入口，承接 stream/resync.ts 的契约—— 同一条订阅里先回放 seq > from 的历史，再无缝续 live：

  • 游标：from = 客户端已渲染到的最后一条事件的 seq。首连传 0。
  • 投递语义：at-least-once；客户端按事件 id 幂等去重（见 协议 · 恢复与压实）。
  • 载荷：每条 SSE message 是一个 @aprog/protocol 的 Event（turn.* / item.start|delta|end / user / error）。不是旧的 agent.thinking 形。
  • running 时：控制平面把 driver 上行的事件盖全局 seq、落库、扇出（透明中继 + 多端共看）。
  • 未运行时：没有 live，只回放已落库的流到 head 后挂起；要恢复 live 须显式 wake。
为什么 resync 与 live 必须同一条连接 别拆成「HTTP 拿全量历史 + 另开连接收增量」——两条连接的交接点会漏/重。resyncThenLive 在服务端原子交接，客户端只管按 id 去重。

输入与交互

两个反向指令，都是小 POST，落到 DriverChannel 的对应方法：

端点映射语义
POST …/inputsendInput(item)用户消息或文件，交给 driver。落到哪、怎么消费由上层程序决定，平台不管。输入由 driver echo 进事件流（user 事件），不在 POST 响应里回显内容。
POST …/interruptcontrol('interrupt')打断当前回合（相当于 Esc）。checkpoint-now / hibernate-prepare 是平台内部信号，不对客户端暴露。
没有审批端点 设计上 harness 全权控制整个沙箱、所有操作自动放行，所以协议里没有 requires_action 事件、北面也没有 respond(allow/deny) 端点——审批这件事在引擎启动姿势里就解决了，不在运行期往返（见 跨 harness 裁决 · 审批）。
多消息怎么处理：排队 vs 打断（对齐 Claude Agent SDK） 共享进程下多人可同时发消息，但harness 不区分谁发的——对它就是「用户」（SDK 的 user 消息只有 role:"user"、无 author），所以平台也不带发送者身份。 当一条消息到达而上一回合还没回复完，只有两种处置，恰好对应 SDK 的两个原语： · 排队等待（默认）：sendInput 入队，等当前回合回复完再送给模型，一条一回合、不拼接。即 SDK 的 streaming-input 队列。 · 打断：POST …/interrupt → control('interrupt') → SDK 的 query.interrupt() 中止当前回合，下一条排队消息随即处理。 这两种来自底层 harness 能力、不是平台自造，所以并发多写在控制平面只需做到达定序，不必发明「输入锁 / 轮次归属」。
待确认 Q5 · input 的返回 + 文件怎么传？ (a) 返回：POST …/input 投递成功该返回什么？我倾向 202 + 该输入落库后的 seq（客户端可用它对齐自己在流里的回显），而非回显内容本身。认可吗？ (b) 文件输入：kind:'file' 时内容怎么传——内联 base64（小文件，简单）还是 multipart / 预签名直传（大文件）？大小阈值多少？以及它落到 sendInput 还是直接 pushBundle 进进程目录？

进程目录读

GET /proc/:pid/fs 服务 前端左下角的目录树与文件预览。只读，平台只按字节服务、不解释文件语义（meta.yml / execution-state/ / artifacts/ 这些多是上层程序自己写的）。数据源随状态切换：

  • running：实时穿透 → DriverChannel.fs.list/read，读的是沙箱内 driver 的 cwd 实时态（见 interaction · 实时目录读）。
  • hibernating：沙箱已销毁，按最新检查点服务。
已定 Q6 · 休眠时 fs = 读检查点数据，一致即真 休眠时直接读控制平面上存的检查点数据。它是 state 的某个一致快照——读到什么，进程恢复时就按什么执行，二者同源、不存在不一致。 所以不必当作「残缺视图」来报警；响应仍带 source: 'live'|'checkpoint' 只为告诉客户端「这是检查点态」。检查点不含的运行期碎产物本就不参与恢复，读不到是符合预期的。

共享与权限

进程与用户是一对多：owner（创建者）可把进程分享给别人。前面已定底层不区分发送者，这里补权限粒度——分享方能精确控制对方能干什么。

角色读流 / 读 fs发消息 / 打断分享给他人改/撤权限
owner（创建者）✓✓✓✓
editor（可读可写）✓✓✗✗
viewer（只读）✓✗（写类端点 403）✗✗
  • 两档共享权限：editor（可读 + 可发消息）/ viewer（只读）。viewer 调 input/interrupt 一律 403。
  • 被共享方不能再分享：分享、改权限、撤回只有 owner 能做（/shares* 非 owner → 403）。
  • 无需「接收」：分享后对方直接进入（出现在他的进程列表），不需要同意；但会在消息中心收到一条「X 把进程 Y 分享给你」通知。
  • owner 可随时改/撤：PATCH 改档（editor⇄viewer）、DELETE 撤回；撤回后对方进程列表移除、活跃的 stream/fs 下次请求即 403 并断流。
  • 统一管理：owner 在前端有一处「分享管理」可看某进程分享给了谁、各是什么角色，并就地改/撤（端点 GET/POST/PATCH/DELETE /proc/:pid/shares）。
待讨论 · 共享的边角 (a) 分享按 userId 直接加，还是也支持「邀请链接」（发链接、谁点谁进，可设角色/有效期）？P0 我倾向只做 userId 直加。 (b) owner 能不能转让 / 多 owner？P0 我倾向单 owner、不转让。 (c) editor 能否 hibernate/wake 这类生命周期操作，还是只 owner？我倾向生命周期归 owner，editor 只对话。这几点要不要这样定？

消息中心

用户级、跨进程的通知中枢，形态对标 Windows 11：状态栏右下角一个铃铛（带未读角标），点开是通知面板；新通知还会在右下角弹出 toast（如某进程有了新回复）。

通知类型触发带
share有人把某进程分享给你来源用户 + procId + 角色
reply你参与的某进程有了新回复（且你当前没在看它）procId
  • 拉取：GET /notifications 取列表（分页、可只看未读）；实时：GET /notifications/stream（SSE）推未读数与新通知，驱动角标 + toast。
  • 已读：POST /notifications/read（带 ids 或全部）。
  • 不重复打扰：你正在 attach 的那个进程的 reply 不再 toast（你已经在看了）。
待讨论 · 通知触发 (a) reply 通知只在「我没在看该进程」时发，对吗？还是只要不在前台就发？ (b) P0 通知类型就 share / reply 这两类，够不够？要不要加进程完成 / 出错 / 被撤回等？ (c) 通知是否需要长期留存与跨设备同步（落库），还是只在线推、读了即弃？我倾向落库（消息中心要能回看）。
