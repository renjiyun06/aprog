沙箱交互界面

沙箱与镜像讲了沙箱怎么烘、怎么起；这页讲沙箱起来之后， 控制平面跟它之间到底发生哪些交互、这些交互该抽象成什么接口。结论一句话：交互分属两个正交的平面—— 一个把沙箱当托管资源管（厂商相关），一个跟烘在沙箱里的 driver 对话（厂商无关）。 所有文件搬运都走后者，于是前者瘦成纯生命周期。

两个平面

把「控制平面 ↔ 沙箱」之间所有往来摊开，它们天然落在两类，区别不在数据多少，而在对谁说话：

control-plane（VM 上）进程编排 · 会话持久化 · 检查点 · 向 web 扇出。两个平面的发起方。
A · 资源平面（带外 / 厂商 API）↓
SandboxProvider 厂商相关纯生命周期：create / destroy（就这俩）。把沙箱当被托管的资源。各家 API 各异。不碰文件、不碰协议。
B · 数据平面（带内 / 我们的协议）↓
DriverChannel 厂商无关带内全包：协议事件流、输入、控制、实时 fs 读、大块传输（bootstrap / 检查点 / 恢复）。同一个 driver、同一套协议。
沙箱（临时、可销毁）
厂商 daemonA 平面的对端：响应厂商 API。
driverB 平面的对端：自启、持凭证拨出、桥接 harness ↔ 协议。
harness + enginedriver 拉起的子进程，cwd 在进程目录。
两个平面的厂商相关性恰好相反：A 平面每家 API 不同（SandboxProvider 吸收）； B 平面对所有厂商都一样（说话的是我们自己的 driver、跑我们自己的协议）。厂商在 B 平面只负责一件事—— 给我们一根可达 driver 的字节管道。所有文件搬运（启动注入、运行检查点、唤醒恢复、UI 浏览）都在 B 平面，A 平面碰都不碰。
为什么文件全在 B、A 一个文件接口都不留 曾经的直觉是「整目录注入/导出该走厂商原生 fs（A 平面）」，理由是 ① 信任种子没法用还没建立的 B 通道送；② 耐久备份要扛住坏掉的 driver。 这两条在收敛后的模型里都不成立：① 信任种子烘进镜像即可（镜像本身就是带外通道，见信任与启动序）； ② 我们从不从坏掉的 driver 里抢救——driver 死 = 沙箱死 = 进程恢复到上一个检查点（见 检查点），不需要事后导出。 于是 injectDir/extractDir 被彻底删除，SandboxProvider 的厂商面更小，也更 provider-neutral——而这正是当初做这层抽象的目标。

A · 资源平面：SandboxProvider

把沙箱当托管资源管，仅此而已。当前需求下只需要两个方法：

方法方向干什么
create(image, res)→用烘好的 ImageRef 起沙箱。driver 随镜像 entrypoint 自启并拨出（见 下文）。
destroy(h)→销毁、释放全部资源（即进程 hibernate 的落地）。状态早已经 B 平面落到控制平面。

就这两个。没有 hibernate/wake/exec/openPty、没有 caps、没有任何文件接口——进程休眠就是 destroy（见 下文）， 文件全是 B 平面的事。这条纪律让 SandboxProvider 的厂商面收敛到最小：接一家新厂商，只要能 create/destroy + 让 driver 拨得出来，就够了。 需要快速唤醒（厂商热令牌）、运维 shell、出站白名单时再按需加（YAGNI）。

B · 数据平面：DriverChannel

控制平面跟 driver 的双工对话。driver 拨上来、通道建立后，下面这套接口在所有厂商上都一样：

方法方向干什么
subscribe(fromSeq)←可续订的 AsyncIterable<Envelope> 事件流。重连 / 唤醒时用 fromSeq 续上（见 seq 归属）。
sendInput(item)→注入一条输入：用户消息 / 文件。落到 harness 的 input.md 流。
respond(actionId, allow|deny)→对 requires_action 的应答（放行 / 拒绝某次工具调用）。
control(signal)→带内控制信号：中断当前回合、checkpoint-now、hibernate-prepare 等。
fs.list(path) / fs.read(path)⇆UI 目录浏览的实时读：driver 在沙箱内 ls/cat 自己的 cwd（见 目录读取）。
pushBundle / pullBundle⇅大块传输：启动时灌进进程目录 + auth；检查点 / 唤醒时整段 state 子集进出。
已定 · 一条连接，按 stream 多路复用 一个 driver（= 一个沙箱 = 一个运行中的进程）跟控制平面之间只建一条物理连接（HTTP/2 双工，driver 拨出）， 内部按 stream 多路复用：一条常驻 control-stream 跑全部交互消息，外加按需开的 bulk-stream 跑大块传输—— 几十 MB 的 bundle 单开一条 stream，靠 HTTP/2 帧交错 + 每流 flow-control，不头部阻塞实时事件流。 不拆成两条物理连接：两条连接换来的内核级隔离只省掉性能抖动，却引入「控制连接活、数据连接死」的部分失败与身份二次绑定—— 拿可容忍的性能风险换不可容忍的状态一致性风险，不划算。所有 stream 与连接同生死：连接断 = 全断， 重连一次、按 seq 续上即可，没有「半通」中间态。落到线上的完整 schema 见 交互 schema。

信任与启动序

既然文件全走 B，那 driver 必须能自己起来、自己拨出、且一拨上来就被信任。信任种子烘进镜像解决——镜像本身就是一条带外通道：

① create沙箱起来。镜像 entrypoint 自启 driver；driver 知道控制平面地址（烘入 / create 时给 env）。
② driver 拨出 + 握手driver 持烘入镜像的凭证拨向控制平面 → B 通道建立 + protocol 版本握手。
③ create-time 绑定控制平面把这条拨入连接钉死到它刚 create 的那个沙箱/待命进程上。
④ 灌 bootstrap控制平面经 B pushBundle：skill 闭包 + per-process auth（用户 OAuth，不能烘）+（首次）用户初始输入 /（唤醒）最新检查点。
⑤ driver 拉起 harnesscwd = 进程目录。首次由 harness 按 state 技能自己写出骨架（平台不代笔）；产出经 driver 翻成信封流回来。
凭证为何能烘进去：bake 发生在 driver 说第一句话之前，等于把 B 通道的引导凭证安全地预置好了——鸡生蛋解除。 它是整张镜像所有沙箱共享的静态秘密，只够证明「我是正版 aprog 沙箱」；要证明「我是进程 X」靠第 ③ 步的 create-time 绑定补齐。 per-process auth 敏感且因人而异、不能烘——但第 ④ 步通道已是 TLS + 凭证鉴权过的，经 B 送 auth 跟走任何厂商 API 一样安全。
driver 是 harness 的父进程 注意第 ⑤ 步：driver 先起、harness 后起，且是被 driver 作为子进程拉起来的。driver 当父进程才能掌控 harness 的生命周期 （启动、喂输入、中断、kill、探退出、休眠前确保落盘）——这也是 driver「harness-specific」的由来：它的全部职责就是包住一个 harness、把它的 IO 双向翻译成协议。

seq 归属：driver 定序，control-plane 盖章

事件流要能跨休眠 / 重连续上，就必须有一个单调递增的 seq。拆成两步最干净：

driver：定序保证一次运行内事件的局部顺序（它最贴近 harness 的产出时序）。
envelope 流→
control-plane：盖全局 seq落库时盖上跨生命周期单调的 seq 并持久化。重连 / 唤醒后 subscribe(fromSeq) 据此续。
为什么不让 driver 直接定全局 seq：driver 随沙箱生灭，休眠后换一个沙箱、换一个 driver，它无从知道「上一世」发到哪了。 全局连续性是跨沙箱的事实，只有常驻的 control-plane 看得全。

控制平面持久化的是抽象后的会话记录（engine-neutral，见 三层数据），不是底层 harness 的原始事件格式—— 省空间、且不把某个 harness 的格式焊进我们的存储。原始事件只在沙箱活着时留在 driver 侧。

交互 schema：一条连接上的消息与流

把前面 B 平面方法落到线上，就是这条连接里跑的两类 stream、三种帧。下面先定连接/流模型与帧信封， 再按业务场景把它们串成消息序。约定：→ = control-plane→driver，← = driver→control-plane。

连接与流模型

1 driver ⇒ 1 连接HTTP/2 双工，driver 拨出。运行中进程恰好一条；休眠进程零连接。连接是唯一的故障域。
control-stream常驻 1 条双向帧流，跑 hello / event / 各类 request+reply。小、低延迟、绝不阻塞。
bulk-stream按需 0..N 条每次 bundle 传输单开一条，传完即关。大、吞吐型。与事件流帧交错，互不头阻。
两类 stream 共享连接命运：连接断则全断 → 重连一次、hello 带水位续事件、未完成的 bundle 整段重传。 没有「control 活 / bulk 死」的半通态——这正是不拆两条物理连接换来的简单性。

帧信封

control-stream 上每个帧是 Request / Reply / Event 三者之一；bulk-stream 上是一段 Bundle* 序列。

帧方向字段说明
Request→ / ← id string(ULID)·op enum·body op ∈ hello(←) / input / respond / control / fs.list / fs.read（除 hello 外均 →）。id 供 reply 回指。
Reply← / → reId string·ok bool·result?·error? reId 回指对应 Request.id，方向相反。error 复用 错误信封 {code,message,retryable}。
Event← eventId string·localSeq int·payload driver 单向推、不需 reply。payload 是 engine-neutral 协议事件；CP 落库时盖 global seq。eventId 供 delta 折叠。
BundleBegin→ / ← bundleId·kind enum·byteLen·sha256·manifest? kind ∈ bootstrap(→) / restore(→) / checkpoint(←)。sha256 整体校验，manifest = 路径→大小清单。
BundleChunk→ / ← bundleId·idx int·bytes 有序分片。方向同本次 BundleBegin。
BundleEnd→ / ← bundleId·ok bool·error? 收尾。ok=false 或 sha256 不符 → 整段作废、重传（bundle 是幂等单元）。

场景 ① 拨入 · 绑定 · 注入

沙箱 create 后 driver 自启拨出，这条连接上的第一组消息。CP 在 Welcome（即 hello 的 reply）里分流「瞬时重连」与「冷唤醒」：

步方向帧说明
1←Request{op:hello, body:{driverVersion, protocol, bindCredential}}持烘入镜像的静态凭证拨入 + 版本握手。
2→Reply{ok, result:{pid, mode, resendFromLocalSeq?}}CP 据 create-time 绑定认出这是哪个待命进程，钉死身份。mode ∈ resume/restore。版本不兼容 → Reply{ok:false, error}，拒连。
3a←Event 重放（仅 mode:resume）同一 driver 还活着的网络抖动重连：driver 从 resendFromLocalSeq 重放自己缓冲区里的事件，CP 据已存 seq 去重。无需灌 bundle。
3b→BundleBegin{kind:bootstrap|restore} + chunks + BundleEnd（仅 mode:restore）冷唤醒（新沙箱/新 driver，localSeq 从 0）：CP 经 bulk 灌 skill 闭包 + per-process auth +（首次）初始输入 /（唤醒）最新检查点。
4—driver 拉起 harnesscwd = 进程目录；之后进入场景 ②。
两种「续上」别混 瞬时重连靠 driver 端缓冲 + resendFromLocalSeq 重放（局部序，便宜）；跨休眠唤醒是新 driver、localSeq 归零，靠 检查点 restore 续，连续性来自 CP 侧不断攀升的 global seq。localSeq 只在一次运行内有意义。

场景 ② 事件流 · 输入 · 审批 · 控制

进程运行期的主干。事件单向上行，其余都是 CP 发起的 Request/Reply：

子场景方向帧说明
实时事件←Event{localSeq, eventId, payload} ×Nharness 产出经 driver 翻成协议事件，持续上行。前端按 SSE 收 CP 扇出的全局序版本。
用户输入→
←		Request{op:input, body:{item}} → Reply{ok}
随后 Event{payload: user 消息}		echo-from-stream：输入被 driver 注入 input.md 流，再以一条 user 事件回灌——所有 viewer（含发起方）都从同一条事件流看到它，语义统一。
审批→
←		Request{op:respond, body:{actionId, decision}} → Reply{ok}应答先前一条 requires_action 事件，放行/拒绝某次工具调用。
控制→
←		Request{op:control, body:{signal}} → Reply{ok}signal ∈ interrupt（打断当前回合）/ checkpoint-now / hibernate-prepare。后两者会触发场景 ④ 的 bulk 回流。

场景 ③ 目录实时读

UI 浏览进程目录时（运行中），CP 当场打到 driver 的 cwd——纯请求/应答，可脏读：

方向帧说明
→Request{op:fs.list, body:{path}} → Reply{ok, result:{entries:[{name,kind,size,mtime}]}}列目录。
→Request{op:fs.read, body:{path, range?}} → Reply{ok, result:{bytes, truncated}}读文件，可带 range；大文件标 truncated。半写文件也能读（看可脏）。
休眠态走不到这条 进程休眠时没有沙箱、连接已断，fs.* 无对端。此时 UI 的目录读由 CP 直接从检查点快照应答（北向 API 内部完成），不进 B 平面。见 目录读取。

场景 ④ 检查点 · 休眠回流

耐久真相靠检查点经 bulk 上行到 CP。三种触发，同一条 bulk 路径：

触发方向帧说明
程序 mark（checkpoint 工具）←BundleBegin{kind:checkpoint} + chunks + BundleEnd主要来源：harness 内的程序在语义一致点调检查点工具——这只是 mark；driver 挂起，等 turn.end（结束回复）盘面静止才捕获，且 mark 后没再调修改性工具才算数。再据 debounce 决定是否真推（无需 CP Request，直接开 bulk-stream）。CP 收齐校验后置为新的耐久真相。
control:checkpoint-now→ 然后 ←先 Reply{ok}，driver 随即推 checkpoint bulkCP 主动要一份（如分享前、迁移前）。
control:hibernate-prepare→ 然后 ←driver 落最后一份 checkpoint bulk → Reply{ok}CP 收到 ok 后调 provider.destroy（A 平面，不在本连接上）→ 连接随沙箱销毁而断。这是连接的正常终点。
连接的两种终结 正常：hibernate-prepare → 末次 checkpoint 落地 → destroy → 断（进程完好休眠）。 异常：连接意外断 = driver/沙箱死。不抢救——进程恢复到上一个检查点，下次唤醒走场景 ①-3b。两条路都收敛到「检查点是唯一真相」。

休眠 = destroy，唤醒 = create + 灌检查点

这里说的休眠是进程的休眠，目的就是把沙箱资源全部让出。进程在 quiescent 点时 state 子集早已经 B 平面落到控制平面，所以休眠很轻——直接 destroy：

休眠 = destroy末次检查点经 B 流回 CP → provider.destroy 释放全部资源。不用厂商 stop——stop 免 compute 却仍计 disk 费，进程常隔很久才回，留盘白烧钱。厂商成本归零。
唤醒 = create + 灌回provider.create 新沙箱 → driver 拨入 → 经 B pushBundle 把检查点灌回去 → driver 拉起 harness resume。
耐久真相是控制平面侧的检查点（运行中已持续经 B 流回），不依赖任何厂商热令牌或留盘。检查点天生跨厂商—— 迁移就是「在厂商 Y 起沙箱、把检查点经 B 灌给它的 driver」。所以 SandboxProvider 不需要 hibernate/wake——它们就是 destroy / create。完整语义见 进程目录与状态。

目录读取：UI 看到的是谁的数据

用户在工作窗口里看到的进程目录是只读的。一条硬边界：UI 永远对着控制平面，从不直连沙箱 （浏览器没凭证、也没路由能摸到厂商云里的沙箱；多 tab 还要在控制平面收敛扇出）。但控制平面这份数据的新鲜度来源取决于进程状态：

浏览器 UI只跟控制平面说话
读→
控制平面运行中 = 透明中继（不吃自己的缓存）；休眠中 = 喂检查点。
运行中 → 经 B 实时拉沙箱fs.list/read 当场读 driver 的 cwd。同步按钮 / 轮询都打到这。能看到正在被写的文件。
休眠中 → 上一份检查点没沙箱可拉，给控制平面存的检查点。只读、可能略旧，UI 标「休眠快照」。
所以「UI 同步的是沙箱还是控制平面的数据」——对端永远是控制平面，但运行时它只是管道、把沙箱实时态原样转发，不是缓存。
两种读、两种节奏 同一条 B 平面 fs、同一份沙箱磁盘，被两种目的以两种节奏读： UI 实时拉可任意时刻发生（哪怕回合进行中、文件半写也无妨，它只是「看」）； 检查点必须 quiescent 点 gate（要能 resume，必须是一致点）。看可脏，存必净。

小结：三套抽象各管一段

抽象时机厂商相关性管什么
ImageBaker构建期高（四种模型）把东西烘进镜像 → ImageRef
SandboxProviderA 平面生命周期拐点中沙箱作为资源：create / destroy（hibernate=destroy、wake=create）。不碰文件
DriverChannelB 平面运行期持续无跟 driver 的协议对话：事件流、输入、控制、实时 fs、所有文件搬运

这套收敛模型对应的代码：SandboxProvider 收敛到 create/destroy（删掉 hibernate/wake/exec/openPty/caps 与 Dormant）， 新增 DriverChannel 接口承载 B 平面。检查点 / resume 的细节见 进程目录与状态。
