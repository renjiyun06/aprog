L3 · Flow detail

attach · mode 1 (首次)

用户第一次进入一个 process. 沙箱不存在, 需要起 + 注入快照 + 启 driver + 打通 IO 流. 所有后续 attach 都是这个流程的子集.

完整时序

阶段拆解

① 鉴权 + dispatch (step 1-2)

CP 收到 POST 进入 attach handler. 标准的 token middleware 验完之后, 走 sqlc 查询 GetProcessForUser (同时验 ownership). 根据 sandbox_id + state 自动判定 mode. mode 1 因为 sandbox_id 是 NULL 且 state 是 spawned.

② 起沙箱 (step 3-4)

Daytona Create 用现成的 snapshot (aprog-driver-v1). 这一步约 1.3 秒 (memory: Daytona 冷启 1.27s). 拿到 sandbox_id 后 立即 UPDATE DB —— 万一后面注入失败, CP 重启也能从 DB 找到这个孤儿沙箱来清理.

③ 注入快照 (step 5-6)

两步: 先上传 tar 文件, 再在沙箱内 tar -xf. 为什么不直接 FileSystem.UploadFiles 多文件? 因为快照文件多 (尤其 execution-state/blobs 可能几十个), 一次 tar 是一个 HTTP 请求, 文件多是 N 个. 总时间差几秒.

④ 启 driver (step 7-10)

这是 architecture 修订后的核心简化: 没有 bridge binary, 直接 ExecuteSessionCommand 起 driver. runAsync: true 让 Daytona 立即返回 command_id, driver 在后台跑.

同一个 sandbox 只创建一个 session (sid 用 aprog-<pid>), 这个 session 持续整个沙箱生命周期. 后续每次 attach 都用同一个 sid + 同一个 cmd.

⑤ 打通 IO 流 (step 11-16)

CP response 给 CLI 一个 ws_url + 一次性 5s ticket. CLI 立即升级 WS. CP 在 WS upgrade handler 里再开一条到 Daytona 的 logs WS, 然后启两个 goroutine:

  • 下行: bufio.Scanner 按行扫 Daytona WS, 一行就 send 给 CLI WS
  • 上行: 读 CLI WS, 每个 JSON 一次 HTTP POST 到 SendSessionCommandInput

⑥ 第一次用户输入 (step 17-20)

到这里整个 attach 就 "活" 了. 用户敲字符串发回 driver, driver 喂给 SDK query() 的 prompt iterable, SDK 跑 agent loop, 流式吐 message, driver translate 后 stdout, 一路回到 CLI.

错误处理

错在哪一步CP 怎么处理用户看到
step 3 Daytona 起不来UPDATE state 回 spawned (没起来不算 running); 返回 503sandbox_unavailable 提示
step 5/6 注入快照失败调 Daytona Destroy 销毁孤儿沙箱; UPDATE state 回 spawned, sandbox_id=NULL; 返回 500internal_error
step 8 driver 起不来同上, 销毁沙箱回滚internal_error
step 13 Daytona WS 连不上关 CLI WS, 返回 close code 1011"无法连接 driver stream, 重试 attach"
step 14+ driver 中途 crash下行 EOF, CP 发 {type:"fatal"}, 关 WS, UPDATE state 回 spawned, destroy 沙箱红色 fatal 错误, 提示重 attach

相关
