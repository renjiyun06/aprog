/exit 流程

用户结束一次会话, 沙箱回收, compute 归零, 状态完整落到平台磁盘. 这是 process 生命周期里 最关键的"数据汇入"一步 —— 几个小时的工作要在这里安全地从沙箱搬回平台.

核心机制

提取在前, 销毁在后. driver 进程退出 ≠ 沙箱容器销毁 —— 中间这段窗口里, 沙箱文件系统完好, CP 用 Daytona FileSystem API 把 /home/aprog 整个 tar 出来, 流式落到本地磁盘, 原子 rename 覆盖快照, 之后才调 Destroy. 数据落地这一刻沙箱必然还活着.

参与角色

User→ CLI↔ Control Plane↔ Daytona↔ Driver↔ Engine + Snapshot disk DB

完整时序

阶段拆解

PHASE 1 · slash.exit 一路下传 (step 1-3)

用户敲 /exit 跟敲普通消息走完全一样的通路: CLI WS → CP → Daytona SendStdin → driver stdin. driver 读到这一行之前不知道这是 exit 信号.

CLI 触发 slash 命令解析 · WS 发 {"type":"slash.exit"} · TUI 切到 "exiting..." 状态
CP / stream 反代 透传 收到上行 WS message · HTTP POST SendSessionCommandInput 一字节不动地塞进 driver stdin
Daytona stdin 通道 把 HTTP POST 的 body 直接写到 session command 的 stdin fd

PHASE 2 · 沙箱内优雅收尾 (step 4-9)

driver 收到 slash.exit JSON 行, 触发"generator return = end of session"这个 streaming-input 模式的核心契约. SDK 见 iterable 关了就知道用户不会再说话了, 把 in-flight 轮跑完, 然后退出 query() async iterator. driver 的 for-await 循环出来后做最后的 SessionEnd hook 收尾, 然后 process.exit.

Driver / stdio.ts 关 IO 通道 stdinIterable 的 switch 命中 slash.exit case · readline.close() · generator return
Engine (Claude Agent SDK) 收口 prompt iter EOF · 跑完当前 turn (如果有) · 发 ResultMessage · 关 session · 写完 session.jsonl 最后一行
Driver / hooks.ts 持久化 engine session id SessionEnd hook 把 session_id 写入 meta.yml.engine_session_id · fsync /home/aprog 保证全部写盘 · process.exit(0)

为什么用 SessionEnd hook 写 engine_session_id, 不让 CP 在 tar 完之后扫?

session.jsonl 文件名内嵌 session_id, CP 解析文件名当然能拿到. 但放在 meta.yml 里是为了让 driver 启动时 (mode 2) 一次性读 meta.yml 拿全部启动参数, 不用再扫 session 目录. 责任划分: session_id 是 engine 知识, 经过 driver 这一层 hook 翻译到 aprog 命名空间 (meta.yml) 里. CP 永远不解析 jsonl 文件名.

PHASE 3 · 提取数据 —— 沙箱仍活的窗口 (step 10-16)

这是整个流程的核心. 黄色警戒带之后, driver 已经 process.exit, Daytona 把 session command 状态标为 completed; 但沙箱容器本身没有被销毁 —— 它的文件系统完整保留. CP 利用这个窗口把 /home/aprog 整体打包拉回. 数据流是三段管道串起来: 沙箱内 tar → Daytona DownloadFileStream → snapshot.Overwrite untar. 中间不实例化在 CP 内存里.

CP / process 模块 检测 + 编排 轮询 GetSessionCommand 状态 (或 webhook) · 看到 completed 后驱动 phase 3 整个流程
Daytona / ExecuteCommand 沙箱内打包 在沙箱里跑 tar -cf /tmp/out.tar -C /home/aprog . · 阻塞返回 · tar 文件在沙箱临时盘里
Daytona / DownloadFileStream 数据流回 流式 HTTP GET · 在 CP 里得到 io.ReadCloser · 不缓冲在 CP 内存 · 直接管道给下游
CP / snapshot 模块 原子覆盖 Overwrite(pid, reader): untar 到 <pid>.swap 临时目录 · rename 当前 <pid> → <pid>.old · rename <pid>.swap → <pid> · rm .old

snapshot.Overwrite 原子性

同 fs 内的 rename(2) 是 POSIX 保证原子的. 三步 rename 后, 即使任意一步之间断电:

  • 断在 untar 阶段 → .swap 不完整, 但 .swap 名称不被 CP 启动时扫到, 老快照在 <pid> 完整
  • 断在 rename old 跟 rename swap 之间 → <pid> 不存在, 但 .old 完整, 启动时检测到 + rename 回 <pid>
  • 断在 rm .old 之前 → <pid> 是新快照, .old 是老的, 启动时 rm 老的

任何中断后磁盘上要么有完整的老快照要么有完整的新快照, 不存在半新半旧.

PHASE 4 · 销毁 + 落账 (step 17-22)

绿色完成带之后, CP 知道数据已经在自己磁盘上了 (不再依赖沙箱). 销毁沙箱是放心动作.

Daytona / Delete 回收 compute 销毁 sandbox 实例 · per-second 计费停止 · sandbox_id 此后不可用
DB / processes 状态机推进 UPDATE state='exited', sandbox_id=NULL, last_exited_at=NOW · 一次事务
CP / stream 反代 通知客户端 WS 发 {"type":"session.end"} · 主动 close WS (code 1000 normal)
CLI / TUI 收尾 收到 session.end 后 tea.Quit 退出 Bubble Tea 程序 · process exit 0

不变式

这个流程依赖几条强不变式, 实现时要保证:

  1. tar-out 在 Delete 之前完成 —— 顺序不能颠倒. CP 实现里这是同一个 goroutine 里的顺序代码, 不并行
  2. session.jsonl 是 tar 范围的一部分 —— 因为它在 /home/aprog/session.jsonl, tar -C /home/aprog . 自然带上. mode 2 cold resume 靠这个
  3. meta.yml.engine_session_id 在 fsync 之前已写入 —— driver SessionEnd hook 必须先写再 fsync 再 exit. 否则下次 mode 2 拿不到 session_id 没法 resume
  4. snapshot.Overwrite 原子 —— 同 fs 内三步 rename, 任何中断后磁盘有效
  5. DB UPDATE 是最后 —— 只有当 state 还是 'running' 且 sandbox_id 非空时, attach 才会被 dispatch 到 mode 3. UPDATE 之前用户尝试 attach, 会因为 sandbox 还活着走 mode 3 进入一个即将被销毁的沙箱, 但这一窗口很短 (整个 phase 4 < 1s)
