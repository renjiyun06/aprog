L3 · Wire protocol

WebSocket Protocol

attach 会话期间的双向流. 行分隔 JSON. 上下行各有自己的 message types.

握手

  1. CLI 收到 attach response 里的 ws_url (含 ?ticket=...) 和 ticket_expires_at
  2. 立即 (5 秒内) 升级到 WS
  3. CP 在 upgrade handler 里:
    • 验 ticket → user_id + pid (in-memory map)
    • 查 process 状态, 确认 attached
    • 开始打开 Daytona logs WS, 启 stream proxy goroutines
  4. 第一帧通常是 {"type":"session.ready", ...} 来自 driver init

下行: CP → CLI

每条消息单独一行 JSON. type 字段是辨别字段.

session 生命周期

{ "type": "session.ready", "session_id": "uuid", "model": "claude-opus-4-7", "tools": ["Read","Edit",...] }
{ "type": "session.compacted", "pre_tokens": 142031, "trigger": "auto" }
{ "type": "session.end" }   // /exit 后, CP 关 WS 前发

agent 事件 (来自 SDK)

{ "type": "agent.thinking",    "delta": "用户希望我先..." }
{ "type": "agent.message",     "delta": "我来分析一下..." }
{ "type": "agent.tool_use",    "id": "tu_abc", "tool": "Read", "input": { "file_path": "..." } }
{ "type": "agent.tool_result", "id": "tu_abc", "ok": true, "output": "..." }
{ "type": "agent.requires_action",
    "id": "act_xyz",
    "prompt": "可以执行 rm -rf node_modules 吗?",
    "options": [{"label":"yes","value":"allow"},{"label":"no","value":"deny"}] }
{ "type": "agent.error", "kind": "rate_limit", "message": "..." }
{ "type": "turn.end", "stop_reason": "end_turn", "usage": { "input_tokens": 1234, "output_tokens": 567 } }

program 状态

{ "type": "phase.changed", "phase": "concepts" }    // 由 driver hooks 投递

异常

{ "type": "fatal", "error": "driver crashed: ..." }   // driver 崩了, CP 紧接着关 WS

上行: CLI → CP

{ "type": "user.input",  "text": "请用 Go 实现快速排序" }
{ "type": "user.action", "id": "act_xyz", "decision": "allow" | "deny" }
{ "type": "slash.detach" }
{ "type": "slash.exit" }

处理

  • user.input / user.action: CP 透传到 driver stdin (HTTP POST 到 Daytona SendSessionCommandInput)
  • slash.detach: CP 直接关 WS, 不通知 driver. UPDATE last_attached_at. 见 /detach 流程
  • slash.exit: CP 先转发给 driver, 让 driver 走 SessionEnd hook, 然后 tar-out + destroy. 见 /exit 流程

完整 session 示例

用户输入"hello", agent 想一下, 调一次 Read 工具, 然后回话:

// CLI 升级 WS 后立刻收到:
<-- { "type":"session.ready", "session_id":"a1b2...", "model":"claude-opus-4-7", "tools":["Read","Bash"] }

// 用户输入
--> { "type":"user.input", "text":"hello" }

// driver 一路出:
<-- { "type":"agent.thinking", "delta":"用户说 hello, " }
<-- { "type":"agent.thinking", "delta":"我应该" }
<-- { "type":"agent.thinking", "delta":"看下 meta.yml 了解当前阶段..." }
<-- { "type":"agent.tool_use", "id":"tu1", "tool":"Read", "input":{"file_path":"/home/aprog/meta.yml"} }
<-- { "type":"agent.tool_result", "id":"tu1", "ok":true, "output":"id: 260521-x9k4\nphase: concepts\n..." }
<-- { "type":"agent.message", "delta":"你好! 我们当前在 concepts 阶段, " }
<-- { "type":"agent.message", "delta":"接下来需要..." }
<-- { "type":"turn.end", "stop_reason":"end_turn", "usage":{"input_tokens":1834,"output_tokens":312} }

// 用户走人
--> { "type":"slash.detach" }
// CP 关 WS, 沙箱不动

CLI 渲染策略

事件UI 表现
agent.thinking灰色斜体, 累积 delta. turn.end 后可折叠
agent.messagemarkdown 渲染, 累积 delta
agent.tool_use彩色行 ▸ tool_use: Read /path/file, 默认折叠
agent.tool_result紧跟 tool_use 之后, 截断显示前 5 行
agent.requires_action醒目红色边框, 提供 Y/N 快捷键发回 user.action
turn.end输入框切到"可输入"态; 底栏显示 usage
session.compactedtoast: "对话已压缩 (142K tokens → ...)"
phase.changed更新底栏 phase 显示
fatal红色错误页, 提示重新 attach

未决

  • cursor / offset 上报: reattach mode 3 时, CLI 是否上报最后看见的 cursor 让 driver 重放? 目前设计是 CP 自己存上次 cursor (in memory). 留口
  • backpressure: 用户输入巨快 vs SDK 处理慢, 上行 buffer 怎么处理? 现在是 WS 自然 backpressure. 后续可以加显式 queue.
  • 多 viewer: 协议不变, CP 内部要 fan-out. MVP 不开.
