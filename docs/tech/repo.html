仓库结构 · 模块职责 · 技术栈

把抽象的"模块"对到具体的目录, 把目录对到运行时位置, 把运行时位置对到选用的技术栈. 一篇文档解决 "改 X 应该动哪个文件" 和 "上线 Y 需要装什么" 两个问题.

三个二进制 · 跑在三个地方

整个 aprog 实际产出三个独立的可执行交付物, 分别住在三个位置. 别的都是它们的支撑件.

仓库目录

monorepo. 三个 binary 的源码 + 共享 protocol + 部署文件 + 文档.

技术栈选型

按"层"组织, 不按"模块". 这一栈解决"用户机 + Go 服务 + 沙箱 + 外部 SaaS" 这四类典型问题.

几个关键选型理由 (备查)

Go (CLI + CP) 单 binary · 共享 protocol 用户机零依赖, 一条 curl 装好; CLI 跟 CP 共享 internal/protocol/; 服务端 stdlib 够强不需要 framework
TypeScript (driver) SDK 一等公民 Claude Agent SDK 的 TS 版 hooks API 比 Python 版更全 (SessionStart/SessionEnd 是 TS-only); Node 20 在 sandbox 里; esbuild 出单 JS
SQLite + sqlc MVP 单 VM 够用 无独立 DB 服务, 一个文件即可; sqlc 编译时验 SQL 跟 schema 对得上; 未来换 Postgres 主要改 DDL 跟 sqlc.yaml, query 大多兼容
chi 而不是 gin/echo 最薄 就是 stdlib http.Handler + 路由 + 中间件链, 没有"框架感", 升级风险低; 中间件生态足够
coder/websocket 而不是 gorilla 现代 + 维护活跃 gorilla/websocket 已 archived; coder 是其 contributor 的 fork, API 更简单 (基于 context.Context)
Bubble Tea 而不是 tview Elm 模式 · 测试友好 Model+Update+View 纯函数, 单测容易; lipgloss 样式表达力强; charm 整套工具链一致
Daytona Go SDK 官方支持 不用手写 HTTP/WS 客户端; sandbox / process / fs 三大 API 全覆盖; 跟我们 CP 同栈
modernc.org/sqlite 而不是 mattn/go-sqlite3 纯 Go, 无 CGO cross-compile 时不要 C 工具链; CI 矩阵简单; 性能差距对我们 workload 无感

考虑过但没选的

  • CLI 用 TypeScript + Ink —— Ink TUI 也不错, 但 Node 在用户机分发链长 (要 nvm / brew install node), 比单 binary 重
  • CP 用 Python (FastAPI) —— Daytona Python SDK 是最成熟的, 但服务长跑 Go 更省心 (内存 / 部署形态 / cross-compile)
  • CP 用 TypeScript (Bun + Hono) —— 跟 driver 同语言诱人, 但 (a) Go 跟 CLI 共享 protocol 更直接 (b) Bun 在 sandbox/service 长跑场景成熟度仍不如 Go
  • Postgres 直接上 —— YAGNI. SQLite 在单 VM 下 WAL mode 处理我们的 workload 绰绰有余, 等真有 multi-VM 需求再迁
  • GORM / ORM —— 跟 sqlc 比, ORM 的"魔法" 在我们这种偏 SQL 直写的场景反而是负担
  • gorilla/websocket —— archived, 不要
  • 分布式追踪 OpenTelemetry —— MVP 不引入. 长链路里只有 CP → Daytona 一段是真跨边界, 价值不大
  • 独立 protocol 仓库 / codegen 跨语言 —— driver/claude/src/proto.ts 手抄维护 Go protocol/ 类型, MVP 量小够用; 真要 codegen 再说

部署 footprint

位置装什么占用
用户机 1 个 binary (aprog) ~10 MB · 0 运行时
aprog VM aprog-controlplane (systemd) + sqlite3 binary (备份用) + nginx + certbot ~50 MB binary + ~增长的 DB / snapshot 磁盘
Daytona snapshot 注册表 aprog-driver-vN 镜像 (node:20-bookworm + tools + 单 JS) ~250 MB per 版本
每个活跃 sandbox 从 snapshot 起一个实例 + 注入 /home/aprog 2 vCPU / 4 GiB / 8 GiB disk · per-second 计费
