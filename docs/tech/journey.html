L1.5 · End-to-end walkthrough

aprog Journey

跟着 lamarck 这个用户走一遍 aprog 的完整生命周期 —— 从登录, 到 spawn 一个 grasp, 到 attach 跟 agent 对话, 到中途 detach 再回来, 最后 /exit 收尾. 每一幕都把当前**舞台上的模块** 和它们**各自提供的机制**摊开. 顺着读一遍, 整个 platform 的运作就在脑子里了.

登场角色

9 个模块. 每个有自己的固定颜色, 整篇文档复用. 后面的每一幕只是这些模块的不同组合在亮灯.

User 人类用户. aprog 之外, 通过 CLI 跟系统交互.
CLI 用户机器上的 Go binary. 命令分发 + 凭据 + TUI.
Control Plane aprog VM 上的 Go service. 鉴权 / 编排 / 代理.
SQLite DB User / AuthSession / Process 三张表. 元数据 source of truth.
Snapshot disk /var/lib/aprog/proc/<pid>/ 进程状态快照. 沙箱不在时 authoritative.
Daytona Cloud 外部 sandbox provider. 镜像 + 沙箱 lifecycle + exec session.
Sandbox per-process 容器, 沙箱镜像跑起来后的环境. /home/aprog 是工作目录.
Engine Driver 沙箱内常驻 TS 进程. SDK 跟 aprog 协议的薄适配.
Engine Claude Agent SDK 的 query() 跑出来的 agent loop. 真正的"CPU".

读法提示: 每一幕开头的 chip 行展示这一幕"在台上"的模块 (颜色亮起来), 其他模块隐含在后台不出场. 看 chip 颜色快速感知"这一步谁在动".

ACT 1 · LOGIN

建立信任 —— aprog login

用户第一次使用. 给一个 username + password, 换回一个 30 天有效的 token. 之后所有命令默认带上这个 token, 不用再输密码.

User→ CLI→ Control Plane↔ DB
  1. User 在终端运行 aprog login, 看到 prompt 后输入 username + password.
  2. CLI 把账密包成 JSON, POST 到 /auth/login (此时没 token, 唯一不带 Bearer 的端点).
  3. CP 走 auth 模块: SQL 拿到 user row, 用 argon2.IDKey verify 密码, constant-time compare.
  4. CP 通过后用 crypto/rand 取 32B → base64url 当 token, INSERT 一行 auth_session (TTL 30 天).
  5. DB 持久化新 session 行. 主键是 token 自己, 不哈希 (token 本身就是 256-bit 随机串).
  6. CP 返回 {token, user_id, username, expires_at}.
  7. CLI 把 token 写入 ~/.aprog/credentials (mode 0600), 打印 Logged in as lamarck.

这一幕每个模块提供的机制

CLI 凭据采集 + 持久化 交互式 prompt (golang.org/x/term) · 0600 文件落地
Control Plane 认证 + 签发 argon2id verify · token gen · session 入库 · 限流 5/min/IP
SQLite DB 凭据 source of truth users + auth_sessions 两张表 · sliding expire
ACT 2 · SPAWN

创建一个程序进程 —— aprog spawn grasp

告诉平台 "我要跑一个 grasp 程序". 平台分配一个 pid, 准备好状态文件骨架, 但不烧 compute —— 沙箱推迟到第一次 attach 时才起.

User→ CLI→ Control Plane↔ DB + Snapshot disk
  1. CLI 读 credentials, POST /proc 带 {program: "grasp"}.
  2. CP token middleware 通过 → process 模块: 扫 ~/.aprog/programs/grasp/ 确认存在.
  3. CP 生成 pid = yyMMdd-<nanoid 4> (例 260521-x9k4).
  4. Snapshot disk 调 snapshot.Init(pid, "grasp"): mkdir 完整目录树, 写空 input.md, 写 seed meta.yml (含 program name + state_schema), 写空 state.yaml.
  5. DB INSERT processes 一行 state=spawned · sandbox_id=NULL · snapshot_path 指向上一步建好的目录.
  6. CP 返回 {pid, state:"spawned"}. 整个过程不触达 Daytona, 不烧 compute.
  7. CLI 把 pid 打印到 stdout, 退出.
Control Plane 编排 + 验证 program 存在性扫描 · pid 生成 · 调 snapshot.Init + DB 事务
Snapshot disk 骨架初始化 mkdir + seed meta.yml + 空 input.md + 空 state.yaml · mode 0755
SQLite DB 登记 processes 行 · CHECK constraint 限定 state 枚举 · snapshot_path UNIQUE
ACT 3 · ATTACH MODE 1

首次进入 —— 沙箱从无到有, agent 上线

这是所有模块都登场的一幕. 沙箱起来 · 快照注入 · driver + engine 启动 · stream proxy 建立. 完成后用户进入 TUI, 可以跟 agent 对话.

User→ CLI↔ Control Plane↔ DB Snapshot disk Daytona Sandbox Driver Engine
  1. CLI 用户 aprog attach 260521-x9k4 → POST /proc/<pid>/attach.
  2. CP 读 process row, 发现 sandbox_id=NULL · state=spawned → dispatch 到 mode 1.
  3. Daytona Create(snapshot="aprog-driver-v1") → 拿到 sandbox_id. 冷启 ~1.3s.
  4. DB UPDATE processes SET sandbox_id, sandbox_started_at, state='running'.
  5. Snapshot disk TarReader(pid) 流式产 tar (io.Pipe). 不实例化在内存里.
  6. Daytona FileSystem.UploadFileStream 接 tar reader 到沙箱 /tmp/snap.tar.
  7. Sandbox 内 ExecuteCommand: tar -xf /tmp/snap.tar -C /home/aprog. 注入完成.
  8. Daytona CreateSession(sid="aprog-<pid>") + ExecuteSessionCommand(cmd="aprog-driver", runAsync=true).
  9. Driver 启动: 读 /home/aprog/.aprog/env.json 拿 API key, 起 stdin readline iterable + stdout writer.
  10. Engine driver 调 query({prompt: iter, options}), SDK 出第一帧 SystemMessage init, 含 session_id.
  11. Driver translate.ts 把 init 翻成 {type:"session.ready", session_id, ...}, 写 stdout.
  12. CP 返回 {ws_url, ticket} 给 CLI. CLI 立即升级 WS.
  13. CP 开 Daytona logs WS, 启动 stream 反代两个 goroutine (上行 SendStdin + 下行 forward).
  14. CLI WS 收到第一帧 session.ready, TUI 进入"等待输入"状态.
Control Plane 编排者 + 代理 mode dispatch · Daytona 调用串 · UPDATE 状态 · WS 反代上线
SQLite DB 状态机锚 spawned → running 状态迁移 · 记 sandbox_id 做反查
Snapshot disk 注入源 io.Pipe 流式 tar · 不实例化在内存 · 不被修改
Daytona 沙箱 lifecycle + bridge Create · UploadFileStream · ExecuteCommand · CreateSession · ExecuteSessionCommand · 同时提供 logs WS
Sandbox 执行环境 aprog-driver-v1 镜像 · /home/aprog 工作目录 · Node 20 runtime · 不对外开端口
Engine Driver SDK ↔ aprog 适配 env.json 读 secret · stdin readline iter · stdout JSON-lines · ring buffer 兜底
Engine agent loop query() async iterator · session_id 生成 · 加载 skills 目录 · permission_mode 控制
ACT 4 · ONE TURN

一轮对话 —— thinking · tool · message · turn end

用户敲一条消息. engine 思考, 可能调几次工具读项目文件, 最后回话. 每个内部事件流式回到 CLI 实时渲染.

User→ CLI→ Control Plane→ Daytona→ Driver↔ Engine + /home/aprog 工作副本
  1. User 在 TUI 输入框敲 "读一下 main.go 看看用了哪些库", 回车提交.
  2. CLI 包成 {"type":"user.input","text":"..."} 写到 WS.
  3. CP stream 反代的上行 goroutine 收到, HTTP POST Daytona.SendSessionCommandInput.
  4. Driver stdin readline 拿到这一行, parse 后 yield 一个 SDKUserMessage 进 prompt iter.
  5. Engine query() 的 UserPromptSubmit hook 先触发 (driver 注册的) → append input.md + 推进 input_cursor.
  6. Engine 进入 agent loop. 先吐 thinking 块 (流式 delta).
  7. Driver translate 把 thinking 块翻成 {type:"agent.thinking", delta:"..."}, 写 stdout + 写 ring buffer.
  8. Daytona logs WS 把 stdout 字节流回 CP.
  9. CLI WS 收到事件, TUI 实时渲染灰色 thinking 行.
  10. Engine 决定调 tool Read, 吐 tool_use 块 → driver 翻成 agent.tool_use.
  11. Engine SDK 内部执行 Read, 读 /home/aprog 下文件 (或受限读项目目录). tool_result 块返回给 model.
  12. Engine 整合后吐 text 块 (回话) → 一直到 SDKResultMessage 标志这一轮结束.
  13. Driver ResultMessage → {type:"turn.end", ...}. CLI 收到后切回"可输入"态.
  14. /home/aprog 同时被多方写: SDK 写 session.jsonl, hook 写 input.md, program 可能写 execution-state/. 平台快照仍然不变 (滞后), 沙箱工作副本是当前 source of truth.
Control Plane 透传 + 鉴权 不解析 payload · 上行 HTTP POST · 下行 WS forward · 失败 close ws
Daytona 双向 bridge logs WS 下行 · SendStdin HTTP POST 上行 · 不感知 payload
Engine Driver 协议翻译 + 状态注入 SDK message → aprog event · hooks 写 input.md + state-skill 协议执行
Engine 推理 + 工具调度 agent loop · 内置 tools (Read/Bash/...) · 自管 session.jsonl · permission_mode 拦截
/home/aprog 工作副本 authoritative 数据面 SDK + hooks + program 三方写入 · 平台快照滞后
ACT 5 · DETACH ↻ REATTACH

暂离 + 回来 —— 沙箱不动, 兜底 ring buffer 补差

用户关笔记本走人, agent 继续跑. 半小时后回来, 看到这段时间发生的事 + 续接 live.

/detach: User→ CLI→ Control Plane · Driver (后台跑)
reattach mode 3: User→ CLI→ Control Plane→ Daytona→ Driver
  1. CLI /detach: 输入框敲 /detach → WS 发 {"type":"slash.detach"}.
  2. CP 收到 → 关 CLI WS + 关 Daytona logs WS + UPDATE last_attached_at. 不通知 driver (driver 视角下啥也没发生, 就是 stdin 没新输入).
  3. Driver 继续跑. agent 如果之前在工作 (例如还有 tool_use 没完), 继续吐事件. events.log 持续 append, ring cursor 递增.
  4. User 30 分钟后回来, 跑 aprog attach 260521-x9k4.
  5. CP 查 process row 看到 sandbox_id 非空 → mode 3. 跳过 Daytona Create / 注入 / 启 driver.
  6. CP 调 Daytona exec 跑 aprog-driver-replay --from-offset=N (N 是上次 detach 时记的 cursor). 同时开新 logs WS 连原 driver 命令.
  7. Driver replay (短命子进程) 读 ring 文件 offset N 之后的内容打到 stdout. 这些事件按时序通过 logs WS 流到 CP, 转给 CLI.
  8. CLI TUI 把 replay 段标灰 (示意 "过去"), 一旦 replay 命令退出, 切回 live 流, 续接新事件.
Control Plane 轻量切换 /detach 只动 DB 时间戳 + 关 WS · mode 3 不动沙箱, 双 exec (replay + 实时)
Engine Driver 独立续跑 不感知 CLI 来回 · ring buffer 兜底 · replay 是独立短命进程
Daytona 沙箱保活 沙箱一直在跑, 持续 per-second 计费 · 提供两个 exec session
ACT 6 · EXIT ↻ COLD RESUME

退出与冷启动 —— 状态回到磁盘, 下次启动对话续上

用户敲 /exit. 沙箱里所有东西 tar 回平台覆盖快照 (包括 session.jsonl), 沙箱销毁, compute 归零. 几天后再 attach (mode 2) 沙箱重起, 但 engine 看到 session.jsonl 后自动 resume, 对话续上.

核心机制 · 提取在前, 销毁在后

driver 进程退出 ≠ 沙箱实例销毁. 这中间有个关键窗口期: driver 已经 process.exit(0), 但 Daytona 沙箱容器还在跑, /home/aprog 文件系统完好. CP 利用这个窗口把整个工作目录 tar 出来流回平台磁盘, 原子 rename 覆盖快照, 之后才调 Daytona.Delete 销毁沙箱. 数据落地这一刻沙箱必然还活着 —— 不存在"从已销毁的沙箱里恢复数据"的反逻辑.

完整 22 步时序 + 4 阶段拆解 + 不变式见 L3 · /exit 流程 ↗.

/exit: User→ CLI→ Control Plane↔ Daytona↔ Driver → Snapshot disk → DB
cold resume (mode 2): CLI→ CP→ Snapshot→ Sandbox→ Driver→ Engine resume
  1. User 在 TUI 敲 /exit.
  2. CLI WS 发 {"type":"slash.exit"}.
  3. CP 收到 → 通过 Daytona SendStdin 把 slash.exit 写进 driver stdin (跟正常用户输入一个通路).
  4. Driver stdin 解析 slash.exit → 关 readline → prompt iterable return → SDK query() 循环退出.
  5. Engine 跑完最后一轮 (如果在 flight), 吐 ResultMessage. SDK 关 session, 写完最后一行 session.jsonl.
  6. Driver SessionEnd hook 触发: 把 session_id 写入 meta.yml.engine_session_id, fsync /home/aprog. process.exit(0).
  7. CP 检测到 driver 命令退出 (Daytona webhook 或轮询 GetSessionCommand). 进入 tar-out 流程.
  8. Daytona ExecuteCommand tar -cf /tmp/out.tar -C /home/aprog .; CP 调 FileSystem.DownloadFileStream 拉回.
  9. Snapshot disk Overwrite(pid, reader): untar 到 .swap 临时目录 → rename old → rename swap. 原子, 断电安全.
  10. Daytona Delete(sandbox_id). 沙箱实例销毁, compute 计费归零.
  11. DB UPDATE state='exited', sandbox_id=NULL, last_exited_at=NOW.
  12. CLI 收到 {"type":"session.end"}, WS 关. TUI 退出.
  13. User ⌚ 三天后, 跑 aprog attach 260521-x9k4.
  14. CP 查到 state=exited + sandbox_id=NULL → mode 2. 启沙箱 (步骤跟 mode 1 几乎一样).
  15. Snapshot disk tar 注入. 这次 tar 里包含 session.jsonl 和 meta.yml.engine_session_id.
  16. Driver 启动时读 meta.yml.engine_session_id, 加到 query options 的 resume.
  17. Engine query({resume: sessionId}) → SDK 读 session.jsonl 全量回放到上下文 → 用户敲下一句话时, 已经是"接着上次的对话".
Engine Driver 优雅退出 readline.close() = generator return = SDK session 自然结束 · SessionEnd hook 收尾
Daytona 数据回流 + 销毁 ExecuteCommand tar -cf · DownloadFileStream · Delete(sandbox)
Snapshot disk 原子覆盖 .swap → rename old → rename new · 同 fs 内 atomic · 断电可恢复
DB 回到 exited state='exited', sandbox_id=NULL · 数据永不删, 可再 attach (mode 2)
Engine resume 对话续上 query options.resume=<sessionId> · SDK 全量回放 session.jsonl · 用户感知 "我刚才说到一半"
