Harness 适配

事件流协议是 engine-neutral 的——它不长得像任何一家引擎。每款底层 harness 由一个 <name>-driver 把它私有的消息翻译成 aprog 事件流。本页给出协议要求 Driver 做的事， 以及 Claude Agent SDK 和 Codex 两套具体映射。换 harness 只换 Driver，协议与前端不动。

Driver 要做的四件事

① 翻译把引擎私有消息 → aprog 事件（turn.* / item.* / user / error）
② 盖 seq给每个事件一个单调递增 seq——两家 CLI 都不提供，Driver 用计数器合成
③ 映射 id把引擎的 item / call 标识归一成稳定 id，让 delta 能折叠、重连能按 id 替换
④ 兜策略自动应答审批（不进流）、回显 user 事件、内存持有“在飞项”供恢复
目标事件种类见 事件流协议 · 事件种类。下面两张表是 ① 的具体落法。

Codex 侧要点 1) 命令执行是三段：begin → output_delta* → end(exit_code)，映射成 aprog command 项。坑：--json 下最终 aggregatedOutput 可能为空，命令输出须从 outputDelta 重建（issue #10141）。 2) reasoning 是并行多段（summary_index 摘要分段 + 原始 CoT 的 content_index）；扁平成 thinking 时要带 part 序，否则会交错。 3) 文件 patch(patch_apply_*) 与回合级累计 turn_diff 是 Codex 独有 → 映射成 aprog file_change 项 + 回合级 artifact。 4) 原生回显 user_message——但为统一，aprog 以 Driver 自发的 user 事件为准，去重引擎回显。

跨 harness 的难点与裁决

问题ClaudeCodexaprog 裁决
全局顺序号无CLI 无（Responses API 有）Driver 统一合成 seq——这正是适配层存在的核心理由
用户回显不回显原生回显一律由 Driver 发 user 事件（echo-from-stream），去重引擎自带回显
工具 / 命令tool_use + 单独 tool_result 消息命令三段 + 结构化结果统一为 aprog tool/command 项：start → (args/output delta) → end(result)
思考 / 推理单流 thinking + 签名并行摘要分段 + 原始 CoTthinking 项；需要时 delta 带 part 序保持不交错
审批（默认放行）审批往返(需回 Op)自动批准：Driver 自动应答，审批不进流
引擎私有会话~/.claude/.../<id>.jsonl + resumerollout JSONL + codex resume都是引擎私有缓存：不权威、不进快照（见 三层数据）

所以 aprog 协议覆盖多家 harness 的关键，就是把它们的差异全部吸收进 Driver：顺序号合成、身份归一、用户回显、审批自动、引擎私有会话忽略。 前端只认 aprog 事件流，永远不知道下面是 Claude 还是 Codex。
