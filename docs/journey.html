运行全景

从用户视角走一遍：装程序 → 启动进程 → 对话 → 休眠 → 唤醒。 上层只讲「你看到什么、做什么」；每一步都能点进底层看平台幕后怎么运作。 这一页既是使用说明，也是把需求与技术细节串起来的总入口。

两个altitude 每个阶段卡片：上半是用户视角，下半是幕后一句话 + 「深入 →」链接。 想看机制就顺着链接往下钻——调用链在 生命周期流程，事件流在 事件流协议，沙箱与镜像在 沙箱与镜像，状态在 进程目录与状态。

① 安装程序

🛍 用户视角：在程序商店挑程序，点「安装」加到桌面 桌面/任务栏只显示你装过的程序。卸载只是从桌面移除——进程和数据都留着， 重新安装打开还在。安装 = 整理你自己的桌面。
  • 幕后：程序本就是一个 SKILL.md 技能包，已 publish 在平台的版本化 store 里； 「安装」只在数据库写一条归属(ownership)+ 该程序声明的 capability 授权，不搬文件
深入 · 程序=技能(skill) → 深入 · 目录与归属(DB) →
↓

② 启动进程

▶ 用户视角：打开程序、给这次运行起个名，进程就跑起来了 一个程序可以同时跑多个进程（各自独立的一次运行）。进程列表里每行显示它跑的 程序版本号（focus/选中时浮现）；点标题栏 ? 看程序说明。
  • 建进程目录 ~/.aprog/<pid>/（meta.yml + input.md + execution-state/）
  • DB 记一行 process，钉死 skill@version + 依赖闭包（不同进程可跑不同版本）
  • 起一个沙箱（程序依赖的那版镜像里已烘了引擎+driver+所需工具），注入程序闭包 + 进程目录
深入 · spawn 调用链 → 深入 · 镜像里烘了什么 → 深入 · 进程目录 →
↓

③ 对话

💬 用户视角：输入指令，看思考 / 工具调用 / 回复实时流出 你回车后，消息不是本地直接画上去——它先提交到沙箱，再顺着同一条事件流回到界面才显示。 界面只有「流」这一个真相源，多设备/多标签天然一致。
  • 引擎跑程序(技能)，产出经 Driver 翻成 engine-neutral 的 aprog 事件入流
  • 工具调用可命中 capability（PATH 上的 CLI / MCP，用注入的受限 token）
  • 程序按 input.md 推进自己的阶段(FSM)，更新 execution-state
深入 · 事件流协议 → 深入 · 用户输入 echo-from-stream → 深入 · 引擎如何映射 →
↓

④ 休眠

⏾ 用户视角：点「休眠」，释放算力；进程还在、对话区收起，目录仍可浏览 休眠的进程不占沙箱。它在列表里照常显示（含版本号），随时能唤醒。状态永不删。
  • 末次检查点 commit + git push 到 CP 托管的 bare 仓库、destroy 沙箱（让出全部资源，不用 stop；工作树在沙箱临时盘，随沙箱一起没，耐久真相在 remote）
  • DB 置 state=hibernating；事件流留在平台侧（不丢）。状态永不删——没有终止态，跑完 / kill 也只是回到 hibernating
深入 · hibernate 调用链 → 深入 · 休眠让出算力 → 深入 · 检查点与持久化 → 深入 · 进程状态存储（Git 模型）→
↓

⑤ 唤醒（再次打开）

⏻ 用户视角：点「唤醒」，进程接着上次干；历史对话照常铺出 对你来说是连续的。但底层是一个全新沙箱、引擎冷启动、空上下文——它怎么知道做到哪了？
  • 起新沙箱 + driver 自启拨入 + 注入钉死版本的程序闭包；进程目录靠 git clone 从 CP bare 仓库拉回最新检查点（默认检出 HEAD，取代旧的 pushBundle 灌回）
  • 程序按 resume 协议读状态文件接着干，绝不反向重建引擎对话上下文
  • 前端重连，按事件流游标补齐缺口 + 续 live —— 历史对话完整呈现
深入 · wake 调用链 → 深入 · 唤醒靠状态不靠对话 → 深入 · wake = git clone →

贯穿全程的两条铁律

状态 ≠ 对话 进程目录是 resume 的唯一权威；事件流只服务界面与审计。 换引擎、丢对话都不影响进程接着干。
版本钉死 进程在 spawn 时钉死 程序@版本 及其依赖闭包；改程序发新版不影响在跑/休眠的进程， 它唤醒后仍跑原版。
这两条让「休眠→唤醒」「程序持续演进」「厂商可替换」三件事互不打架——也是整套设计自洽的支点。
