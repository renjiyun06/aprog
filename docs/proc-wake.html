运行生命周期

本篇定义进程的运行生命周期：三个动词 spawn（创建）、wake（唤醒）、hibernate（休眠）， 以及连接三者的进程状态机。它是 01 · 总览 §4「进程唤醒控制流」的展开。 进程状态的权威落点为控制平面 ProcessManager 所维护的进程控制块（PCB，数据库 processes 表）； 进程的内容态以 git 仓库承载，详见 06 · 状态契约。

1 · 进程状态机

进程的稳态由单一判据决定：是否关联一个沙箱。有沙箱即 running；无沙箱则区分为 spawned（已创建、从未起过沙箱、无检查点）与 hibernating（曾运行、已释放沙箱、留有检查点）。 状态机无终止态：进程运行完毕或被 kill，均仅回到 hibernating，进程目录与产物始终保留、随时可再唤醒。

为承认异步唤醒期间的过渡，状态机设第四态 waking，作为 running 的前奏（详见 §4）。

    waking --> spawned: bring-up 失败回滚

状态枚举与定义：

状态定义沙箱稳态
spawned已创建 PCB，从未起过沙箱，进程仓库无检查点无是
running关联一个活动沙箱，引擎运行中有是
hibernating曾运行、已释放沙箱，进程仓库 HEAD 即最新检查点无是
waking异步唤醒受理后、就绪前的过渡态；sandbox_id 暂空起中否（过渡）

ProcessManager 的 ProcessState 枚举 spawned / running / hibernating / waking 四态。 程序内部的执行 FSM（phase / status）住进程目录 meta.yml，不进 PCB，与本状态机正交。

生命周期操作的状态转移与副作用：

操作前置后置对 PCB / 沙箱的副作用
spawn—spawned插入 PCB 行、钉死 program@version；创建空 bare 仓库 aprog-proc-<pid> 并回填 repo_url；sandbox_id=null，不起沙箱
wakespawned / hibernatingwaking → running置 waking 并返回；后台创建沙箱（A 平面）→ 握手 → 起引擎，ready 后转 running、回填 sandbox_id / provider / last_active_at
wake（已 running）runningrunning幂等：直接返回当前 PCB，不重起沙箱
hibernate / killrunninghibernating末次检查点（沙箱侧 git push）→ 销毁沙箱 → 置 hibernating、sandbox_id=null

2 · spawn

spawn 创建一个进程而不消耗算力：写入 PCB 行（state=spawned）并为进程创建承载其内容态的 git 仓库。 pid 由数据库自增（类比操作系统 pid）。仓库为私有 bare 仓，名由 pid 推导（aprog-proc-<pid>）； 其 clone URL 写入 PCB 的 repo_url 列——仓库名可由 pid 推导，而实际地址入库，二者不可混淆。 建库以 auto_init=false 创建空仓，以保持「spawned 无检查点」的语义。建库失败则删除该 PCB 行回滚，不留半成品。

实际仓库由 RepoGateway 收口：真实实现 GitHubRepoGateway 调 GitHub API 建私有库（不写入进程名 / program / pid 等描述性元数据）； 未配置凭证时以 MockRepoGateway 顶替，仅生成可观测的占位 URL。更换 host（如 Gitea / Forgejo）只需替换实现，编排逻辑不变。

3 · wake 控制流

wake 是唯一启动算力的动词，为 owner-only 端点。它统一覆盖两种入态——首次运行与休眠恢复——对前端透明。 接口契约如下：

方法 / 路径POST /proc/:pid/wake
鉴权 / 授权withAuth（token → User）→ authorize(user, pid, 'owner')；不存在 404、非属主 403
请求体无
响应200 + PCB（state=waking），毫秒级返回；running 经 SSE 或轮询稍后推送（见 §4）
后端落点routes/proc.ts → wake() → ProcessManager.wake(pid)（process/manager.ts）

同一 wake 据进程仓库是否已有检查点（即入态为 spawned 还是 hibernating）分叉为两条路径， 差异仅在 git clone 取回的内容；起沙箱、拉程序包、起引擎、建通道、回写 PCB 完全相同。

入态语义git clone 取回引擎启动
spawned → running首次运行：从未起过沙箱、无检查点空仓 / 初始 input从 input_cursor=0 初始化运行
hibernating → running休眠恢复：曾运行、有检查点HEAD（最新检查点）按 resume 续跑
running → running幂等：已有活动沙箱—直接返回当前 PCB，不重起

下图为完整控制流。wake 跨越两个平面：A 平面（资源平面，SandboxProvider 创建沙箱）与 B 平面（数据平面，driver 拨出的 WebSocket）。 OCI 闭包拉取与 git clone 直连存储、不经数据平面通道，图中以 driver 自指消息表示。

各环节职责与要点：

① 入口与编排。 routes/proc.ts 完成鉴权与 owner 授权后调 ProcessManager.wake(pid)。 wake 取 PCB：若已 running 则幂等返回；否则调 SandboxGateway.create 起沙箱，成功后置 running 并回填 sandbox_id / provider / last_active_at。

② A 平面：创建沙箱。 ProviderSandboxGateway.create 先据 program@version 解析出已烘镜像的 ImageRef，并由控制平面生成 bindToken（信任凭证归控制平面所有）。 PPIOProvider 调厂商 SDK 起 Firecracker MicroVM（冷启约 4 秒），在「容器已起、driver 未启」的窗口内回调 onProvisioned， 上层据 sandboxId 把 bindToken 登记进 DriverRegistry——登记早于 driver 拨号，从根上消除握手竞态。 随后经 files.write 以 root 推入 driver bundle（/root/aprog/driver.mjs），并以 background 方式后台启动 node 进程。 启动时注入的环境变量包含：APROG_BIND_TOKEN（绑定令牌）、APROG_CONTROL_PLANE_URL（控制平面回拨地址）、引擎鉴权（如 ANTHROPIC_AUTH_TOKEN，经 injectedEnv 运行时注入、不落镜像），以及可选的 NODE_EXTRA_CA_CERTS（控制平面边缘 CA 证书，使 driver 经 TLS 回拨时信任自签证书）。

③ B 平面：driver 握手。 driver 自启后第一件事即回连：据 APROG_CONTROL_PLANE_URL 推导通道地址 wss://host/<base>/v1/driver/channel， 持注入的 bindToken 拨出 WebSocket，发送 Hello 帧（protocolVersion + bindToken），控制平面据登记簿认领连接并应答 Welcome 帧（pid + mode）。 mode 取 resume（瞬时重连）或 restore（冷唤醒），driver 据此决定铺设进程目录的方式。 握手以原子单元退避重试（最多 20 次，退避封顶 3 秒），兜网络抖动。所有通信帧遵循统一格式 Frame<t,p>。

④ 握手之后的运行时编排。 按 Welcome 与控制平面下发的种子（程序引用 / 短期 registry pull token / git deploy token / 模型鉴权），driver 依次： 按 digest 拉取程序包 OCI 闭包并 untar 至 skills/；git clone 进程态至进程目录（首跑为空 / 初始 input，恢复为 HEAD）； 经 SDK 启动原生引擎；引擎就绪后回报 ready 帧。这些种子因人而异、必须握手后下发，真实凭证不出控制平面，沙箱只取短时窄 scope 的令牌。

4 · 异步唤醒

唤醒为异步：wake 立即将进程置为 waking 并毫秒级返回，bring-up（起沙箱 → 握手 → 下发种子 → 拉程序包与 clone 进程态 → 起引擎）转入后台任务； driver 启动引擎并回报 ready 后，控制平面才将进程转为 running，经前端本就订阅的 SSE / 轮询推送。

异步的动因在于：真实沙箱冷启为秒级至数十秒，若同步阻塞至就绪，HTTP 请求会长时间挂起，易超时、被代理切断或触发用户重试，且启动期间无反馈。异步唤醒的要点如下：

维度异步唤醒（waking）
返回时机受理后立即
返回态waking
HTTP 时长毫秒级
ready 信号driver 报 ready 后转 running
running 推送经 SSE / 轮询
并发保护waking 天然占位，防并发重起
失败处理销毁沙箱 + 回滚至来路（spawned / hibernating）
waking 的定位 waking 是 running 的前奏过渡，非终止态；它不是曾被否决的 exited。状态机为 spawned / hibernating → waking → running，bring-up 任一步失败则释放已起沙箱并回滚至进入前的来路态。 前端无须新增机制：它本就长连在状态 stream 上，running 仅是该流上的又一次状态变更。

5 · hibernate

hibernate 显式释放算力：进程态先检查点到 git，随后销毁沙箱，进程回到 hibernating。 因耐久真相为控制平面托管的 bare 仓库（已 git push 的检查点），休眠本身很轻；销毁沙箱后厂商成本归零，引擎私有的会话文件随沙箱消失（非权威，无妨）。 kill 与运行完毕与 hibernate 同效——均经统一的释放路径回到 hibernating，不产生终止态，仅触发方不同。 

sequenceDiagram
    autonumber
    actor U as 用户与前端
    participant CP as 控制平面
    participant PM as ProcessManager
    participant SP as SandboxProvider
    U->>CP: POST /proc/:pid/hibernate（或 kill）
    CP->>PM: hibernate(pid)
    Note over PM: 非 running / 无沙箱 → 幂等直接返回
    Note over PM,SP: 末次检查点 git push（沙箱侧）
    PM->>SP: destroy(sandboxId)
    SP-->>PM: ok（销毁沙箱）
    PM->>PM: state=hibernating, sandbox_id=null, last_active_at=now
    PM-->>U: 200 · PCB（state=hibernating）

释放路径为 ProcessManager.release → SandboxGateway.destroy → 置 hibernating、sandbox_id=null， 对非 running 或无沙箱者幂等。ProviderSandboxGateway.destroy 调 provider.destroy 按 id 销毁沙箱（幂等：沙箱已不在视作成功）。 末次检查点（quiescent 点落盘 + 末次 commit 的 git push）属沙箱侧编排，先于销毁完成。 PCB 不存 commit 指针：「最新检查点」即仓库 HEAD，可推导，故 wake 默认 clone 检出 HEAD 即取回最新检查点。 对称地，hibernate 不设过渡态：销毁通常远快于创建，且失败不影响「已让出算力」的语义。
