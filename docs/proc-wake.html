wake · 唤醒进程流程

单挑「唤醒」这一个接口,把它从前端点击到进程跑起来的整条链讲透——用于讨论: 当用户点「唤醒进程」,后端到底要做哪些事、哪些已落地、哪些待对接沙箱、还有哪个设计点要拍板。 模型背景见 数据模型 · Process,状态存储 / 取回模型(git clone/checkout)见 进程状态存储（Git 模型）,机制全景见 生命周期流程,契约见 控制平面 API。

现状：PCB 已通,沙箱 mock 唤醒的编排骨架（取 PCB → 起沙箱 → 回写 state=running → 返回最新 PCB）已落地并端到端验证; 但「真正起沙箱」这一步走 MockSandboxGateway——只造假 sandbox_id,不 git clone/checkout 取回状态、不拉 harness、不建通道。 本页用 现 标 mock 已做的、待 标接入真实沙箱后才补的。

接口签名

一个动词、一个 owner-only 端点,返回唤醒后的最新 PCB。前端无须区分「首跑」还是「恢复」——都只是点「唤醒」。

方法 / 路径POST /proc/:pid/wake
鉴权 / 授权withAuth（token→User）→ authorize(user, pid, 'owner')（非属主 403,不存在 404）
请求体无
响应200 + 最新 PCB（state=running、回填 sandbox_id/provider）
前端入口wakeProcess(pid)（web/src/stores/processes.ts）→ 就地替换列表里那行 PCB
后端落点routes/proc.ts → wake() → ProcessManager.wake(pid)（process/manager.ts）

一个接口,两种唤醒

同一个 wake 覆盖两种入态,内部靠有没有 checkpoint_ref 分叉——对前端透明,对后端是两条灌料路径:

spawned → running 首跑 从未起过沙箱,无检查点。冷启:新沙箱 + 灌初始 input,harness 首次按 state 技能写出进程目录骨架、从 input_cursor=0 跑起。
hibernating → running 恢复 跑过、留了检查点。新沙箱 + git clone/checkout <checkpoint_ref> 取回工作记忆,harness 按 resume 续跑。进程目录、产物都还在。
两条路径只在第 ③ 步「取回什么」不同(空 / 初始 input vs 该 commit);起沙箱、git clone/checkout、拉 harness、建通道、回写 PCB 全相同。 running 入态则幂等:已有沙箱,直接回当前 PCB,不重起。

完整流程(真实沙箱版)

从前端点击到事件回流,一次 wake 串起两个平面:A 平面(控制平面起停沙箱)与 B 平面(DriverChannel,事件上行 + input 下行)。逐步标注 mock 现状:

前端POST /proc/<pid>/wake
↓
① 鉴权 + 授权 + 取 PCB 现
  • withAuth 验 token → User;authorize(user, pid, 'owner')(不存在 404、非属主 403)
  • 读 PCB:running → 幂等直接返回当前 PCB(已有沙箱);否则进入起沙箱
A 平面 · 起算力 ▼↓
② 申请沙箱 现 待
  • SandboxGateway.create() → provider(Daytona…)按钉死版本的镜像起容器
  • 慢点在此:冷启容器 = 秒级~数十秒(详见 同步/异步)
  • mock 现状:立即返回 mock-sbx-<pid>-xxxx / provider=mock,无真实算力
↓
③ git clone + checkout 取回状态 待
  • git clone --depth=1 CP 托管的进程 bare 仓库 → checkout <checkpoint_ref>,工作树落在沙箱临时盘(不挂持久卷);得到进程目录 ~/.aprog/<pid>/(权威内容态:meta.yml / input.md / execution-state/)
  • hibernating 取回该 commit;spawned 无 commit,工作树为空 / 只放初始 input。耐久真相在 CP bare 仓库,不随沙箱生灭(见 Git 模型)
B 平面 · driver 拨入 ▼↓
④ 拉起 harness 待
  • driver 自启拨向 CP + create-time 绑定 + 版本握手,建 DriverChannel
  • 工作树已由第 ③ 步 git clone/checkout 取到(恢复得到该 commit;首跑则为空 / 初始 input);CP 另注 skill 闭包 + per-process 凭证
  • driver 拉起 harness:首跑写骨架从头跑,恢复按 resume 续跑
A 平面 · 收尾 ▼↑
⑤ 回写 PCB + 返回 现
  • UPDATE:state='running'、回填 sandbox_id + provider、刷 last_active_at
  • checkpoint_ref 保留(恢复后旧 commit 仍是历史真相,下次休眠 push 新 commit 后更新指向)
  • 序列化最新 PCB 返回前端 → 列表那行变 运行
之后事件经 DriverChannel 上行、CP 盖全局 seq 落库 + 扇出;前端订阅事件流(GET …/stream)渲染对话。 那是 attach/订阅的事,不属 wake 本身——wake 的职责到「沙箱跑起来、PCB=running」为止。

唤醒前后,PCB 这几列怎么变(对照 processes 表):

当前实现 vs 真实实现

mock 阶段 ProcessManager.wake() 已是真实编排的形状,只是第 ②–④ 步被 MockSandboxGateway 顶掉:

// process/manager.ts —— 现状(mock 沙箱)
async wake(pid) {
  const cur = this.get(pid)
  if (cur === undefined) return undefined          // 404 在路由层
  if (cur.state === 'running') return cur           // ① 幂等
  const { sandboxId, provider } =
    await this.sandbox.create({ pid, programId, programVersion })  // ② mock:造假 id
  // ③ git clone --depth=1 + checkout  / ④ 拉 harness:真实实现补在这里(mock 跳过)
  UPDATE processes SET state='running', sandbox_id=?, provider=?, last_active_at=?  // ⑤
  return this.get(pid)
}
接入真实沙箱时 只需把 MockSandboxGateway 换成真实 provider 实现(create 真起容器),并在 create 之后补第 ③④ 步(git clone --depth=1 + checkout <checkpoint_ref> 到临时盘 / 首跑置初始 input、拉 harness、建 DriverChannel)。 wake() 的编排骨架与 routes/proc.ts 不动——这正是 SandboxGateway 收口的目的。

待拍板:wake 是同步还是异步?

这是唤醒流程里唯一真正要先定的设计点,因为它直接挑战我们刚锁死的三态模型。 症结在第 ② 步:真实沙箱冷启秒级~数十秒。这段时间,HTTP 请求和进程状态怎么处理?

A · 同步阻塞 wake 一路做到 running 才返回。
+ 实现最简单;返回即终态,前端拿到 PCB 就是 running,无须额外订阅状态变化。
− HTTP 长挂数十秒,易超时/被代理切断/用户重试;启动期间用户无反馈;取消、并发都难做。
B · 异步 + 过渡态 waking wake 立即把进程置 waking 返回,后台起沙箱;前端订阅 stream/轮询看它转 running。
+ 即时反馈、与 HTTP 超时无关、可显示进度、可取消。
− 打破三态,需引入第 4 个状态 + 失败回滚把态退回原态;前端要处理过渡态渲染。
注意 waking 不是当初砍掉的 exited(终止态)——它是 running 的前奏,不是终点。
我的倾向:B(承认过渡态) 真实沙箱启动慢是绕不开的物理现实,假装它瞬时只会把体验与错误处理推给前端。建议: spawned/hibernating → waking → running,失败则回滚到进入前的原态(记住来路)。 若担心扩散:waking 在 UI 上仍归入「未就绪」一类(和休眠同区显示,只是带 spinner),不必到处分叉。
连带要顺一句的 对称地,hibernate 也要「落末次检查点 + destroy」,理论上也有耗时——但 destroy 通常比 create 快得多,且失败不影响「已让出算力」的语义,先不引入 hibernating 过渡态,除非实测慢到需要。

边界:失败回滚 / 幂等并发

  • 失败回滚:起沙箱成功但 git clone/checkout/拉 harness 失败,PCB 不能停在 running(否则「显示有沙箱、实际没跑」)。任一步失败 → 释放已起的沙箱 → 退回进入前的原态(spawned/hibernating),并把错误回给前端。
  • 幂等:running 入态直接返回当前 PCB(第 ① 步已做),重复点「唤醒」不重起沙箱。
  • 并发:两个 tab 同时点 wake——需要锁(或「waking 态即占位」),避免起两个沙箱。选了方案 B 时,waking 本身就是天然的并发占位。
  • 属主边界:wake 是 owner 档(起算力是花钱动作),区别于 viewer/editor(读流/投输入)。
