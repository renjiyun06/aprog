wake · 唤醒进程流程

单挑「唤醒」这一个接口,把它从前端点击到进程跑起来的整条链讲透: 当用户点「唤醒进程」,后端到底要做哪些事、哪些已落地、哪些待对接沙箱。 唤醒走异步(已拍板):wake 立即回 waking,真正就绪(running)稍后经前端本就订阅的事件流推回。 模型背景见 数据模型 · Process,状态取回(git clone)见 进程状态存储（Git 模型）,程序包拉取见 程序与发版,平面与握手见 沙箱交互界面。

现状：PCB 已通,沙箱 mock,异步待落 唤醒的编排骨架(取 PCB → 起沙箱 → 回写状态 → 返回 PCB)已落地并端到端验证; 但当前是同步形(直接回 running),且「真正起沙箱」走 MockSandboxGateway——只造假 sandbox_id,不 git clone 取回状态、不拉程序包、不拉 harness、不建通道。 本轮已定改异步(立即回 waking、running 经 stream 推)。本页用 现 标已做的、待 标接入真实沙箱后才补的。

接口签名

一个动词、一个 owner-only 端点。异步:立即返回 waking 的 PCB,不挂着等冷启。前端无须区分「首跑」还是「恢复」——都只是点「唤醒」。

方法 / 路径POST /proc/:pid/wake
鉴权 / 授权withAuth（token→User）→ authorize(user, pid, 'owner')（非属主 403,不存在 404）
请求体无
响应200 + PCB(state=waking,毫秒返回);running 不在本响应里——稍后经事件流推
前端入口wakeProcess(pid)(web/src/stores/processes.ts)→ 就地置该行为 waking(「唤醒中…」);监听 stream 待其翻 running
后端落点routes/proc.ts → wake() → ProcessManager.wake(pid)(process/manager.ts)

一个接口,两种唤醒

同一个 wake 覆盖两种入态,内部靠仓库有没有 commit(即 state 是 spawned 还是 hibernating)分叉——对前端透明,对后端是两条路径:

spawned → running 首跑 从未起过沙箱,无检查点。冷启:新沙箱 + 灌初始 input,harness 首次按 state 技能写出进程目录骨架、从 input_cursor=0 跑起。
hibernating → running 恢复 跑过、有 commit。新沙箱 + git clone 取回工作记忆(默认 HEAD = 最新检查点),harness 按 resume 续跑。进程目录、产物都还在。
两条路径只在 git clone「取回什么」不同(空 / 初始 input vs HEAD 最新检查点);起沙箱、拉程序包、起引擎、建通道、回写 PCB 全相同。 running 入态则幂等:已有沙箱,直接回当前 PCB,不重起。

完整流程(异步 · 真实沙箱版)

一次 wake 串起两个平面:A 平面(控制平面起停沙箱)与 B 平面(DriverChannel,事件上行 + input 下行)。 关键:第 ② 步就把响应回给前端(waking),③–⑧ 全在后台跑,running 最后经 stream 推回。

当前实现 vs 异步目标

mock 阶段 wake() 是同步形(直接回 running);异步目标把它拆成「立即置 waking 返回 + 后台 bring-up + stream 推 running」:

// process/manager.ts —— 异步目标(mock 现状仍同步:省略 waking、直接回 running)
async wake(pid) {
  const cur = this.get(pid)
  if (cur === undefined) return undefined            // 404 在路由层
  if (cur.state === 'running') return cur             // ① 幂等
  const from = cur.state                              // 记住来路(spawned/hibernating)以便回滚
  UPDATE processes SET state='waking'                 // ② 立即置 waking
  void this.bringUp(pid, from)                        //    后台跑 ③–⑧,不 await
  return this.get(pid)                                //    立即返回 PCB(waking)
}

// 后台 bring-up(③–⑧):
//   起沙箱(派生镜像) → driver 拨入握手 → CP 下发种子
//   → 并行(OCI pull 程序包+闭包 ‖ git clone 进程态)→ 起引擎 → 报 ready
//   成功:UPDATE state='running' + 回填 sandbox_id/provider,经 stream 推
//   失败:释放沙箱 → UPDATE state=from(回滚)→ 经 stream 报错
接入真实沙箱时 ① 把 wake() 改成上面的异步形(置 waking 即返回 + 后台 bringUp); ② 把 MockSandboxGateway 换成真实 provider(create 真起容器); ③ 在 bringUp 里补 driver 握手、种子下发、OCI 拉程序包、git clone、拉 harness、stream 推 running。 routes/proc.ts 与 SandboxGateway 收口不动。

已拍板:wake 走异步(waking 过渡态)

唤醒流程里唯一真正要先定的设计点——已定:走异步。症结在第 ③ 步:真实沙箱冷启秒级~数十秒。 同步会让 HTTP 请求挂这么久(易超时 / 被代理切断 / 用户重试,且期间无反馈),所以引入第 4 个状态 waking 承认这段过渡:

A · 同步阻塞 已否决 wake 一路做到 running 才返回。
+ 实现最简单,返回即终态。
− HTTP 长挂数十秒,易超时 / 被代理切断 / 用户重试;启动期间无反馈;取消、并发都难做。
B · 异步 + 过渡态 已选 waking wake 立即把进程置 waking 返回,后台起沙箱;前端从已订阅的 stream 看它转 running。
+ 即时反馈、与 HTTP 超时无关、可显示进度、可取消、waking 天然防并发。
− 引入第 4 个状态 + 失败回滚把态退回原态;前端要处理过渡态渲染。
waking 不是当初砍掉的 exited(终止态)——它是 running 的前奏,不是终点。
已定:B(承认过渡态) 状态机:spawned/hibernating → waking → running,失败回滚到进入前的原态(记住来路)。 前端不新增机制:它本就长连在事件 / 状态 stream 上(收对话事件),「now running」不过是那条 stream 上的又一个状态变更,顺带就收。 扩散控制:waking 在 UI 上仍归「未就绪」一类(和休眠同区显示,只是带 spinner),不必到处分叉。
连带要顺一句的 对称地 hibernate 也有耗时——但 destroy 通常比 create 快得多,且失败不影响「已让出算力」的语义,先不引入 hibernating 过渡态,除非实测慢到需要。

边界:失败回滚 / 幂等并发

  • 失败回滚:后台 bring-up 任一步(起沙箱 / 下发种子 / 拉程序包 / git clone / 拉 harness)失败,不能停在 waking 也不能进 running(否则「显示在唤醒 / 有沙箱,实际没跑」)。→ 释放已起的沙箱 → 回滚到来路原态(spawned/hibernating)→ 经 stream 把错误回前端。
  • 幂等:running 入态直接返回当前 PCB(第 ① 步),重复点「唤醒」不重起沙箱。
  • 并发:两个 tab 同时点 wake——waking 本身就是天然的并发占位:第二次进来看到 waking 即返回,不再起第二个沙箱。
  • 属主边界:wake 是 owner 档(起算力是花钱动作),区别于 viewer/editor(读流 / 投输入)。
