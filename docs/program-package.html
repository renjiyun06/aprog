程序与发版

程序对底层 harness 是一个目录，对平台是一个可版本化发布的包。本篇定义程序的本地版本管理、 包的承载形式（OCI 制品）、依赖声明与闭包解析、发布与拉取的职责划分，以及控制平面数据库的定位。 运行底座（镜像）见 03 · 镜像与沙箱。

1 · 程序的两副面孔

同一份内容，在两个视角下承担不同职责：

视角程序是什么关心的内容
harness（运行时）一个目录SKILL.md 及其 resources / scripts / references；按正文解析引用并执行。包的外壳对它不可见。
平台（发布时）一个包内容寻址、版本、依赖闭包、安装清单。可发布、可去重、可由 driver 按版本拉取部署。

一句话：包是一个版本化、去重、自带完整性校验与安装清单的目录；解包落入沙箱后，即 harness 所见的普通目录。

2 · 本地版本管理：目录模型

版本管理与镜像一致：git 跟踪的源目录为管理面，发布产物在仓库之外。程序的源目录结构如下（扁平，不再按应用 / 库分级）：

层级示例含义
programs/—所有程序的根目录
<名>/programs/state/一个程序；目录名即程序名
<版本>/programs/state/0.1.0/一个版本；目录名即版本号，人定、不可变、自包含
· SKILL.md程序入口与内容（skill 正文）
· aprog.json依赖声明（见 §4）；无此文件即无依赖
· resources/ …该程序所需材料

文件系统即管理面：新增版本 = 新建 <版本>/ 目录；查 = ls；退役 = 删目录；演化 = git log。不设独立的 registry 代码或数据库来管理版本。

程序与镜像在管理面上的对应关系：

维度镜像程序
源 / 管理面（git）images/<名>/<版本>/<供应商>/programs/<名>/<版本>/
发布动作aprog-bakeaprog-publish
产物（仓库外）PPIO template → ImageRefOCI artifact（GHCR）→ digest
版本目录名（人定、不可变）同左

3 · 包形式：OCI 制品

程序包采用 OCI（Open Container Initiative）制品格式，经 oras 推拉。复用成熟的 registry、分发与鉴权机制，且与镜像共用同一套分发层。一个程序包由一个 config 与若干 layer 组成：

组成mediaType内容
config blobapplication/vnd.aprog.program.manifest.v1+json清单（即 lockfile）：name@version、解析并钉死的依赖闭包（各 digest）、各层落地 target、install 指令
layer · skillapplication/vnd.aprog.skill.layer.v1.tar+gzip闭包中一个程序的 skill 目录（SKILL.md 及材料）
layer · bins（同上类）程序专属的预编命令（按版本随程序走；当前未启用）

OCI 的内容寻址（sha256 digest）提供不可变性与完整性；layer 级内容寻址使共享库在 registry 中只存一份，被多个程序的清单引用而自动去重。

4 · 依赖：声明 → 解析 → 取包

程序间存在依赖（如库 state 被多数程序依赖）。依赖经三个阶段兑现，复杂度集中于发布期：

阶段位置职责
声明programs/<名>/<版本>/aprog.json{"dependencies":{"state":"0.1.0"}}。键为依赖关系（供 harness 做 schema 合并，版本无关）；值为精确版本（供发布期解析）。
解析（一次，CI）aprog-publish走传递闭包、将每个依赖按 digest 钉死、拍平写入 config（即 lockfile）。版本约束求解只在此发生。
取包（每次，唤醒）driver仅拉取该已钉死的扁平闭包，不运行求解器；内容寻址使共享层每机只拉一次。

依赖声明置于 aprog.json 而非 SKILL.md：SKILL.md 为 harness 的纯 skill 内容，不应承载版本等打包信息。 程序自身的名与版本由路径 programs/<名>/<版本>/ 决定，aprog.json 仅声明所依赖的程序及其版本。 版本号只出现于声明意图（aprog.json）与产物（lockfile），源码其余部分不含版本。

5 · 发布与拉取的职责划分

“安装时可装命令”包含两个必须分开的时机，否则会把烘制镜像省下的冷启延迟重新赔回：

时机执行者 / 频次职责
publish（构建）CI，一次、可重跑承担重活：编译命令、解析依赖、按 digest 钉死、组装包并推送 GHCR。产出自包含的包。
spawn-time installdriver，每次唤醒仅 apply manifest：摆放文件、将 bins 链入 PATH、写入 env / config。幂等、离线、毫秒级。

原则与镜像中工具的处理一致：编译在发布期完成，运行期只注入预编产物；绝不在 spawn-time 执行 apt 安装或编译。

6 · driver 拉取流程

拉取一个 OCI 制品即一组 HTTP GET（Distribution 规范）。driver 不解析依赖，仅取发布期已钉死的扁平闭包：

sequenceDiagram
    autonumber
    participant D as driver
    participant R as Registry（GHCR）
    participant FS as 沙箱文件系统
    D->>R: GET /v2/<repo>/manifests/<tag 或 digest>
    R-->>D: manifest（config + layers 列表）
    D->>R: GET /v2/<repo>/blobs/<config-digest>
    R-->>D: config（lockfile：闭包与各层落地 target）
    loop 每个未缓存的 layer
        D->>R: GET /v2/<repo>/blobs/<layer-digest>
        R-->>D: blob
        D->>D: 校验 sha256 == digest
        D->>FS: 解包至 skills/<名>/
    end
    D->>FS: apply install（链接 bins、写 env / config）
    Note over D,FS: 命中本地 CAS 的共享层跳过下载，实现去重
凭证与去重 短期 token：控制平面在种子中下发仅可拉取该包仓、短时有效的 token（scope=repository:<repo>:pull），真实凭证不出控制平面。 CAS 去重：driver 按 digest 建本地内容寻址缓存，共享层（如 state）每机只下载一次。 driver 自拉：拉取即 fetch + sha256 + 解包，无需引入 oras。

7 · 控制平面数据库：投影

内容真相在 OCI 与 git，数据库为可查询的投影与平台独有的关系态，而非内容存储：

表定位
programs身份：current_version 指针、展示元数据、installations / processes 的外键落点
installations平台独有的关系态（用户 × 程序）
processes进程控制块（PCB）
program_versions薄索引 / 缓存：(program_id, version, digest)；digest 即 OCI 的不可变锁

不直接以 registry 顶替数据库：registry 为 blob 存储，不提供 current_version 指针、快速目录列举、以及 installations / processes 等关系态与 GC 所需的 SQL 连接。aprog-publish 从 git 源与 OCI digest 投影出上述表。
