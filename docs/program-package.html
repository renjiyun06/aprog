架构

程序与发版（包模型）

程序（program）= 一个 skill。对底层 harness 而言它就是一个目录； 但对平台而言,它要能版本化发布、表达依赖、带自己的命令——所以我们把它做成一个包: 一个版本化、内容寻址、自包含、带瘦安装层的产物。这页讲:本地怎么管版本(沿用镜像的目录模型)、 包用什么形式承载(OCI artifact)、依赖怎么解析、driver 怎么拉、以及为什么控制平面的库表降级成投影。

程序的两副面孔

harness 眼里 就是个目录 SKILL.md + 它的 resources/scripts/references。harness 加载这个目录、按 prose 解析引用,跑起来。包的外壳对它不可见。
同一份内容的两端↔
平台眼里 是一个包 内容寻址 + 版本 + 依赖闭包 + 安装清单 + 预编命令。能发布、能去重、能被 driver 按版本拉下来装好。
一句话:包 = 一个版本化、去重、自带完整性校验和安装清单的目录。解开落到沙箱里,就是 harness 看到的那个普通目录。

本地管理 = 目录模型(沿用镜像那套)

版本管理和镜像逐字相同:git 跟踪的源目录是管理面,发版产物在外部。 

programs/
  <类>/                  ← libraries / applications（程序比镜像多的一级分组）
    <程序名>/             ← 一个程序 = 一个目录;名即目录名
      <版本>/             ← 一个版本 = 一个自包含子目录(人定、不可变)
        SKILL.md         ← 入口 + frontmatter(声明 image 依赖、depends_on)
        …这个程序需要的材料(resources / 预编命令 / 安装清单源)
文件系统即管理面加一个版本 = 新建 <版本>/ 目录;查 = ls;退役 = 删目录;演化 = git log。不要 registry.ts / 枚举 / 单独 DB 管版本。
OCI 是产物,不是管理面就像 Daytona snapshot 之于镜像:aprog-publish 把源目录烘成 OCI artifact 推到 registry。git 是真相,registry 只存产物。
镜像程序
源 / 管理面(git)images/<名>/<版本>/ + bake.tsprograms/<类>/<名>/<版本>/ + SKILL.md
发版动作aprog-bakeaprog-publish
产物(外部)Daytona snapshot → ImageRefOCI artifact(GHCR)→ digest
版本目录名(人定、不可变)同左
发现/增/退役/历史ls / 新建目录 / rm / git同左
现状要补一处 programs/ 现在是 <类>/<名>/,缺 /<版本>/ 这层。采用本模型即把它补成 programs/<类>/<名>/<版本>/。

包形式 = OCI artifact

OCI(Open Container Initiative)是容器镜像那套打包 + registry 标准,而且通用到能装任意内容(OCI artifact,经 oras 推拉)。 拿它当程序包,等于白嫖一套成熟机制,且不用自造格式:

我们要的OCI 现成给的
不可变 + 完整性sha256 digest 内容寻址
版本tag(名@版本)
共享库只存一份layer 内容寻址 → 自动 dedup(state 那层全网一份,被多个程序引用)
安装/依赖声明manifest + config blob(塞 install 指令、依赖闭包)
别自造格式标准 spec + oras
和镜像同一套分发同一种 registry —— program 与 image 在分发层成兄弟

一个程序包(OCI artifact)的结构:

config blob = manifest名@版本 · 依赖的 image@版本 · 声明式 install 指令(files→dest / bins→PATH / env / config) · 解析并钉死的依赖闭包(各依赖 digest)。
layer · skill 闭包SKILL.md + resources/scripts/references —— harness 看到的那个目录。
layer · 预编 bins这个程序专属的命令(预编 amd64 二进制)。跨版本/跨程序复用就 dedup。
起步退路:胖 tarball 没有 registry 时可先用内容寻址 tarball + 对象存储 + presigned URL 顶着,把链路跑通。 但胖 tarball 把依赖 vendor 进单包 → state 在每个下游各存一份(重复、无共享、底层库变动要重打所有下游)。 所以目标是 OCI(layer dedup 对共享库几乎是刚需),两者都是 名@版本 + digest,平迁面很小。

publish 干重活,spawn 只瘦装

"安装时可以装命令"这句话里藏着两个不同的 install 时机,必须分开,否则把烘镜像省下的冷启延迟又赔回去:

时机谁跑 / 几次能干啥
publish(构建)CI,一次、可重跑任意重活:编命令、拉二进制、解析依赖、组装。产出 = 自包含的包(skill 闭包 + 预编 bins + manifest)。
spawn-time installdriver,每次唤醒只 apply manifest:摆文件、symlink bins 上 PATH、写 env/config。幂等、离线、毫秒级。
和 #tools 同源的纪律 这跟 沙箱 · 工具 的"编译在镜像外做完、只注入预编 amd64 产物"是同一条原则,只是从 bake-time 装进镜像,搬到 spawn-time 装进沙箱。 绝不在 spawn-time apt install / 编译——否则每次冷启都背一遍那延迟。

由此理清 program / image 的工具边界(两种工具物种):

重的 / 共享的底座 → 烘进镜像语言运行时、大编译器、引擎。慢变、跨进程共享、贵。随 image@版本 走。
程序专属命令 → 随程序包spawn 时瘦装。轻、专属、跟程序一起 rev(改它不动镜像)。
这恰好同时满足:程序能独立于镜像发版,又能自带命令。capability/plugin(三层模型第三层)将来挂在 install 钩子上、经 MCP 接,先留位、实现压后。
install 跑命令 = 安全 + 确定性面 能声明就别写脚本;非跑脚本不可时,以沙箱用户、离线、幂等跑。这层正是隔离要兜底的地方。

依赖:声明 → publish 解析钉死 → spawn 取闭包

智能程序之间有依赖:一个库 state 可能被几乎所有程序依赖。包形式必须表达这个依赖,这样 driver 拉下包、依赖也就齐了。

阶段干啥
声明SKILL.md frontmatter depends_on: [state@…, …](库本身也是 library 类 aprog,也是一个包)。
publish(一次,CI)走传递闭包、把每个依赖按 digest 钉死、把拍平的闭包写进 manifest(= 一份 lockfile)。版本约束求解只在这里,可审、确定。
spawn(driver)只拉那份已钉死的显式集合——无求解器。内容寻址 → state 每台机器只拉一次(命中 CAS 缓存)。逐个摆成 skill 目录 + apply 瘦装 + 按 state 协议合并 schema。
两层解析,别混 prose is link(运行时,给 agent):skill 正文提到 state 即引入——agent 自己会加载。 depends_on frontmatter(打包时,给 packager):机器可读,让 publish 的 resolver 知道该收哪些包。正文提及不够机器可靠,所以 frontmatter 是依赖的权威声明源(也正是 state 协议做 schema 合并时读的那个字段)。

两条依赖轴分清,兑现路径不同:

横向 · 库/程序依赖depends_on → 解析进包闭包,落地为并排的 skill 目录。driver 拉包闭包兑现。
纵向 · 镜像依赖image: 名@版本 → 运行底座,由起沙箱拉 snapshot 兑现(provider.create),不是一个子包。
state 协议的 schema 合并发生在创建时:execution 的 state_schema = 程序自身 schema + 各 depends_on 库 schema,同名 key 冲突为硬错误(见 state 技能)。

driver 怎么把包拉到本地

拉一个 OCI artifact 本质就是几个 HTTP GET(registry 的 Distribution spec)。没有魔法:

1. GET /v2/<repo>/manifests/<digest>     ← 拿 manifest(JSON):列出 config + 各 layer(digest/size/mediaType)
2. 对每个还没缓存的 blob:
   GET /v2/<repo>/blobs/<sha256:…>        ← 拿真实字节;校验 sha256 == digest
3. 解 tar 到目标路径(skill → cwd/skills/<名>;bins → /opt/aprog/bin;…)
4. apply manifest 的 install 指令(symlink bins 上 PATH、写 env/config)
短期 token 来自 CPregistry 是 token 流(scope=repository:<repo>:pull)。CP 在 welcome/种子里塞一个只能 pull 这个包仓、短期的 token,真凭证不出 CP(见 信任与启动序)。
CAS 去重内容寻址 → 本地按 digest 建 CAS。共享层(如 state)全机器只下一次,后续命中缓存跳过。依赖闭包随之落全。
driver 自拉(推荐)就是 fetch + sha256 + 解 tar,driver 本就是 HTTP 客户端,多几个 GET 即可,不必为此烘 oras。和 git clone 进程态一样直连存储,不挤 control-stream。

控制平面的库表:降级为投影,不是真相

既然程序内容发版在 OCI、源在 git,那 program_versions 还要吗?要,但定位变了: OCI/git = 内容真相;DB = 可查询的投影 + 平台独有的关系态。——和"文件系统是源、DB 是投影"是同一个模式。

为什么不能直接查 registry 顶替 DB registry 是 blob 存储,不是可查询目录库。这些它干不了: ① current_version 指针是平台决策,registry 只有 tag(用 latest 是反模式); ② 服务应用商店要快,不能每次列仓库 + 逐个拉 manifest; ③ 关系态(installations 用户×程序、processes 钉 program_version)registry 压根不知道; ④ GC join(processes → program_versions → image)是 SQL 活; ⑤ registry 抖一下,商店列表不该跟着挂。
表留不留新定位
programs(身份)留装 registry 没有的:current_version 指针 + 展示元数据(供快查)+ installations/processes 的 FK 落点
installations留平台独有关系态,registry 里没有
processes留PCB,同上
program_versions留,瘦身从"内容存储"降成薄索引/缓存:(program_id, version, digest, published_at)——digest 就是 OCI 不可变锁
子决策:image 依赖存还是派生 image 依赖其实写在包 manifest 里了。两条路:存(program_versions 留 image_name+image_version,denormalize 缓存 → GC 保持纯 SQL join,毫秒级)vs 派生(不存,spawn/GC 时 CP 从 registry 拉 manifest 读)。 倾向存——它只是 manifest 声明的一份投影(由 aprog-publish 写入,源同一份不会漂),换来 GC/spawn 不依赖 registry round-trip。这是缓存换查询性,不是真相分叉。
取代 const CATALOG 现状 catalog/programs.ts 用一个写死的 const seed 这些表,且自注"权威态本应是磁盘 skill 注册表(SKILL.md frontmatter),当前还没有"。 正解:aprog-publish 从 git 源 + OCI digest 投影出这些表,取代 const。

真相链与唤醒集成

programs/<类>/<名>/<版本>/   ← 权威源(git;你管理版本的地方)
        │  aprog-publish 读它
        ▼
 ① 解析 depends_on 闭包、digest 钉死 → 构建 OCI artifact(skill 层 + bins 层 + manifest)→ 推 GHCR
 ② 回填 (program, 版本, digest, [image 依赖]) 进 program_versions(DB 投影)
        ▼
唤醒(spawn/wake)时:
  CP: resolveImage(process→program@版本)→ provider.create(派生镜像)
  driver: 拨入握手 → CP 下发种子(模型 auth + registry/git 短期 token)
          → 并行:OCI pull 程序包闭包(横向)‖ git clone 进程态
          → 起引擎(SDK spawn)→ 报 ready
"下载程序"这一步 = driver 按 digest 拉程序包的 OCI 闭包;依赖随之落全、共享库只拉一次。完整唤醒序见 wake 唤醒流程 / 交互 · 拨入绑定注入。
