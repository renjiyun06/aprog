Web 前端 (mica)

用户真正面对的界面。一个程序打开是一个窗口，窗口分三区：左边进程列表、中间对话区、右边目录区。 它跨 NAT，只跟 Control Plane 的一个稳定 endpoint 通信，永不直连沙箱。详见 工作窗口。

关于 CLI 早期版本以 CLI（aprog ps / attach …）为主入口。现已转为 Web 前端为主； CLI 作为同协议的另一种客户端保留为附录，见 参考。

Control Plane

平台的中枢，跑在 aprog 自家 VM 上，是所有跨沙箱操作的协调点和元数据/快照的事实源。它内部分几块：

api
HTTP / WS 端点

前端的所有请求入口；attach 升级为 WebSocket 流

auth
鉴权

login / logout / token，单一鉴权面

process
进程编排

spawn / attach / detach / hibernate / wake / exit 的流程

snapshot
快照磁盘 IO

进程目录 tar-in / tar-out，平台侧持久化

sandbox
沙箱封装

Daytona SDK 封装 + Bridge 调用

stream
流反代

把 Driver 的事件流反代给前端 WS

为什么所有流量都过 Control Plane、不让前端直连沙箱？单一鉴权面、沙箱不暴露公网端口、provider 抽象不外漏。 这是一条 load-bearing 决策，见 架构 · 关键决策。

Engine Driver

沙箱内常驻的进程，跨 attach/detach 持续活着。它做两件事：

  1. 跑 Engine：在自己进程里 import { query } from '@anthropic-ai/claude-agent-sdk'，以 streaming-input 模式驱动 agent loop。
  2. 翻译协议：把 SDK 私有的 AssistantMessage / SystemMessage / ResultMessage 翻成 engine-neutral 的 aprog 事件，写本地缓冲、并 append 进进程目录的会话记录。

为什么要一个常驻 Driver、而不是裸跑 Engine？因为 detach 后前端走了、Control Plane 到沙箱的连接也该关， 但 Engine 可能还在跑或在等输入，它的产出得有东西兜住；reattach 时还要能看到“离开这段时间发生了啥”。 “SDK 消息 → aprog 事件”的翻译放在沙箱内做，让平台与引擎彻底解耦。每款引擎一个 <name>-driver 包， 对外的 wire 协议固定，内部怎么调 SDK 是 driver 自己的事。

Bridge

一次 attach 期间才存在的短命管道，无状态。Control Plane 通过 daytona exec 在沙箱里把它拉起来， 它连上 Driver 的本地 socket（/run/aprog/io.sock），把下行字节 pipe 到自己的 stdout（Daytona 流回 Control Plane）， 把上行的用户输入 pipe 回 socket。前端断开 → Control Plane 关 exec → Bridge 退出；Driver 不受影响继续兜 Engine 输出。

为什么不直接 daytona exec 启动 Driver？因为 Driver 是沙箱生命周期级别的长跑进程，不能跟某一次 attach 绑命。 Bridge 才是按 attach 起灭的一次性管道。

Engine

实际的 agent loop，当前是 Claude Agent SDK。它跑在 Driver 进程里，吃 Driver 喂来的用户输入（async iterable）， 产出消息流。它自己的会话持久化（session.jsonl）是 SDK 默认行为，但对 aprog 不权威—— 那只是沙箱生命周期内的临时缓存，沙箱一销毁就没了。详见 进程目录与状态。

Sandbox（Daytona）

per-process 的容器，外部 cloud 服务，提供文件系统、进程空间、网络和 exec API。沙箱镜像（template）里烘进了： 基础 OS + 常用工具、claude-driver、aprog-bridge、Node.js 运行时。spawn/wake 时 Control Plane 选用当前 active 镜像起一个新沙箱， 把进程目录注入 /home/aprog/。沙箱本身不对外暴露端口；任何“进沙箱”的动作都经 Daytona exec 发起。

依赖方向

单向，无循环：

前端───►controlplane.api
controlplane┈┈►drivers/*作为镜像内容，不是源码依赖
controlplane───►protocol
drivers/*───►protocol
drivers/*───►engine SDK各自的
bridge───►protocol只做 transport，不解析 payload
┈┈► 镜像内容依赖（非源码）　·　───► 源码依赖。单向无环，driver 可独立演化。

controlplane 和 drivers/* 没有源码依赖——driver 跟着沙箱镜像一起烘进去， 通过固定的 socket 协议交互。这保证 driver 可以独立演化、独立换引擎。
