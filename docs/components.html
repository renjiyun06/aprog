aprog 自己出的代码是：前端、Control Plane、Driver，外加一个共享的 protocol 包。Engine 和 Sandbox 都是外部依赖。

Web 前端 (@aprog/web)

用户真正面对的界面。一个程序打开是一个窗口，窗口分三区：左边进程列表、中间对话区、右边目录区。 它跨 NAT，只跟 Control Plane 的一个稳定 endpoint 通信，永不直连沙箱。详见 工作窗口。

关于 CLI 早期版本以 CLI（aprog ps / attach …）为主入口。现已转为 Web 前端为主； CLI 作为同协议的另一种客户端保留为附录，见 参考。

Control Plane

平台的中枢，跑在 aprog 自家 VM 上，是所有跨沙箱操作的协调点和元数据/检查点的事实源。它内部分几块：

api
HTTP / SSE 端点

前端的所有请求入口；事件流订阅走 SSE（from=<seq>）

auth
鉴权

login / logout / token，单一鉴权面；也给 git smart-HTTP 加薄鉴权门

process
进程编排

spawn / attach / detach / hibernate / wake / exit 的流程

git-host
进程 git 仓库（bare）

托管每进程的 bare repo（<pid>.git），是耐久事实源；driver git push 入、前端按 ref 拉（详）

sandbox
A 平面 · 沙箱封装

provider SDK 封装：create/destroy（hibernate=destroy、wake=create，不碰文件）

driver-channel
B 平面 · 流中枢

受理 driver 拨入、盖全局 seq 落库、扇出给前端、下发 input/控制（不再有 fs 读 / 大块传输）

为什么所有流量都过 Control Plane、不让前端直连沙箱？单一鉴权面、沙箱不暴露公网入站端口、provider 抽象不外漏。 这是一条 load-bearing 决策，见 架构 · 关键决策。

Engine Driver

沙箱内常驻的进程，随镜像 entrypoint 自启、持烘入凭证拨回 Control Plane，跨 attach/detach 持续活着。它做四件事：

  1. 当 harness 父进程：拉起 Engine（import { query } from '@anthropic-ai/claude-agent-sdk'，streaming-input 模式），掌控其生命周期。
  2. 翻译协议：把 SDK 私有的 AssistantMessage / SystemMessage / ResultMessage 翻成 engine-neutral 的 aprog 事件，局部定序后经 DriverChannel 上行（全局 seq 由 CP 落库时盖）。
  3. 承载 DriverChannel：上行事件、接收 CP 下发的 input / 控制信号（仅此两类实时往来，不再有 fs 读穿透 / 大块传输）。
  4. 管 git：在 quiescent 点 git commit + git push 进程仓库到 CP 托管的 bare repo 当检查点；冷启动 git clone 铺好进程目录（见 Git 模型）。

为什么要一个常驻 Driver、而不是裸跑 Engine？detach 后前端走了，但 DriverChannel 是 per-sandbox 常驻、不随 detach 关； Engine 可能还在跑或在等输入，它的产出得有东西兜住并继续落库；reattach 时还要能按游标看到“离开这段时间发生了啥”。 “SDK 消息 → aprog 事件”的翻译放在沙箱内做，让平台与引擎彻底解耦。每款引擎一个 <name>-driver 包，对外协议固定，内部怎么调 SDK 是 driver 自己的事。

Engine

实际的 agent loop，当前是 Claude Agent SDK。它由 Driver 作为子进程拉起，吃 Driver 喂来的用户输入（async iterable）， 产出消息流。它自己的会话持久化（session.jsonl）是 SDK 默认行为，但对 aprog 不权威—— 那只是沙箱生命周期内的临时缓存，沙箱一销毁就没了。详见 进程目录与状态。

Sandbox

per-process 的容器，外部 cloud 服务（当前 Daytona），提供文件系统、进程空间、网络和 exec API。沙箱镜像（template）里烘进了： 基础 OS + 常用工具、claude-driver（含拨回 CP 的凭证）、引擎运行时。spawn/wake 时 Control Plane 用 provider SDK 起一个新沙箱（A 平面）； 沙箱本身不对外暴露入站端口——所有交互靠沙箱内 driver 出站拨回 CP 的 DriverChannel（B 平面）。沙箱与镜像详见 沙箱与镜像、交互界面详见 沙箱交互界面。

依赖方向

单向，无循环：

前端───►controlplane.api
controlplane┈┈►drivers/*作为镜像内容，不是源码依赖
controlplane───►protocol
drivers/*───►protocol
drivers/*───►engine SDK各自的
┈┈► 镜像内容依赖（非源码）　·　───► 源码依赖。单向无环，driver 可独立演化。

controlplane 和 drivers/* 没有源码依赖——driver 跟着沙箱镜像一起烘进去， 通过固定的 protocol + DriverChannel 交互。这保证 driver 可以独立演化、独立换引擎。
