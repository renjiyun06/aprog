控制平面 API 实现

控制平面 API 定的是契约（端点、形状、错误码）；这页讲代码怎么分层去兑现它。 一句话：API 层是一张「翻译膜」，不是逻辑所在地——它只把 HTTP 翻成对子系统的调用，业务逻辑住在 ProcessManager / stream/* / driver-channel 这些子系统里。代码在 packages/control-plane/src/api/。

现状 本页描述的是骨架：目录结构、类型契约、路由器、中间件组合、一条请求的走法都已立起来且 tsc 通过； 但每个端点处理器的实体仍是 throw 'not implemented'——因为它们要调的子系统（stream/* 等）还没实现。 这正是「设计/结构先行，落地在后」的中间态。

分层：翻译膜 + 子系统

每个端点处理器做的事高度雷同，就五步——这五步不含业务逻辑：

① 鉴权withAuth → authenticate(token) → User（失败 401）
↓
② 授权authorize(user, pid, need) → 角色 ≥ 档位（非成员 403）
↓
③ 校验解析/校验入参（parsePid、body 形状）（非法 400）
↓
④ 调子系统deps.procs.spawn() / resyncThenLive() / channelFor(pid).sendInput() …逻辑在这里
↓
⑤ 序列化json / created / accepted / noContent / paged → 统一响应形状
这跟 driver 的 EngineAdapter 是同一个模式：薄 I/O 壳 + 真核。 API 层只管翻译与守门（①②③⑤），真正干活的核（④）在子系统——换传输（REST→别的）不动逻辑，测逻辑不必起 HTTP。

文件布局

按资源拆，对应 api.html 的端点分节。http.ts 只当组装根，逻辑分散到 route 模块与中间件：

api/
  http.ts              组装根：起 Bun.serve、建路由表、mount 各 route、统一分发（薄）
  router.ts            极小路由器：method + /:param 段 → handler + 路径参数
  context.ts           核心类型：Deps（子系统接缝）/ User / Role / ReqCtx / AuthCtx / Handler
  errors.ts            北面错误码 + ApiError + toErrorResponse + withErrors 包装器
  respond.ts           响应序列化：json/created/accepted/noContent/paged + parsePid
  middleware/
    auth.ts            authenticate(token→User) · authorize(user,pid,need→Role) · withAuth 包装器
  routes/
    auth.ts            /auth/login · /auth/logout
    programs.ts        /programs
    proc.ts            /proc · spawn · wake · hibernate · input · interrupt · fs（fs 读走 git repos，非 DriverChannel）
    shares.ts          /proc/:pid/shares*
    notifications.ts   /notifications · /notifications/read
  sse.ts               GET …/stream · /notifications/stream（流式，自成一路）
文件职责为什么单列
http.ts组装根：mount 所有 route 模块、把请求分发给匹配的 handler、剥版本前缀路由 ≠ 处理，二者分开；它薄到几十行
context.ts定义 Deps——API 与子系统的唯一接缝；以及 ReqCtx/AuthCtx 与处理器签名把"能调什么"收成一个注入点，便于测试替身
middleware/auth.tsauthenticate（token→User）、authorize（pid→角色）、withAuth鉴权/授权是横切关注点，所有需登录路由共用
errors.ts北面错误码枚举 + ApiError + withErrors（异常→错误信封）错误形状统一在一处；不复用南面 SandboxError
routes/*.ts每组端点一个模块；各自 mount(router) 注册路由并就地决定要不要 withAuthauth 需求归各模块自管（如 login 不需 token）
sse.ts两个 text/event-stream 长连端点 + sseResponse 帧格式化流式 plumbing 异于普通请求/响应，混进 REST 会别扭

一条请求的生命周期

以 POST /proc/:pid/input 为例，看这五步怎么串起来（每条路由都套同样的洋葱）：

Bun.serve.fetch(req)
  → router.match('POST', '/proc/42/input')        // → { handler, params:{pid:'42'} }
  → withErrors(                                    // 捕获下游任何 ApiError → 错误信封
      withAuth(                                     // authenticate(token) → AuthCtx{ user }
        input(ctx) {
          const pid = parsePid(ctx)                 // 校验 :pid（非法 → 400 validation）
          await authorize(ctx.user, pid, 'editor')  // 角色 ≥ editor（viewer → 403）
          channelFor(pid).sendInput(item)           // ④ 调子系统：DriverChannel 下行
          return accepted({ seq })                  // ⑤ 202 + 输入落库后的 seq
        }))
组合而非框架 没有引入 Web 框架——withErrors / withAuth 只是函数包装函数，在每条路由 mount 时手工套上。 需登录就 withErrors(withAuth(h))，公开端点（如 login）就 withErrors(h)。一目了然，无魔法。

Deps：API 与子系统的接缝

处理器不直接 new 任何东西，只通过 ctx.deps 调子系统。http.ts 在启动时把子系统装配进 Deps 注入：

Deps 成员谁服务哪些端点
procsProcessManagerps / spawn / wake / hibernate / 取 PCB
storeStreamStore事件流回放（盖 seq、落库）
hubStreamHub事件流 live 扇出
repos进程 git 仓库存储GET …/fs 按 ref 读已提交 tree/blob（CP 托管 bare repo）+ git smart-HTTP 端点
channelFor(pid)→ DriverChannelinput / interrupt（下行到 running 沙箱内 driver）。fs 读穿透已移除，改走 repos
接缝现状：pending 占位 stream/*、进程 git 仓库存储（repos）等子系统目前仍是接口、无实现，http.ts 暂用 pending('StreamStore') 这类 Proxy 占位注入——一旦被处理器触达就抛「子系统尚未装配」的清晰错误，而不是静默 undefined。 procs 是唯一已到位的（但其方法本身也还 throw）。

还没做 / 落地顺序

  • 已立：目录结构、Deps/ReqCtx 类型、路由器、withErrors/withAuth 组合、sseResponse 帧格式、错误信封、分页信封——全部 tsc 通过。
  • 桩：每个端点处理器体 = throw 'not implemented'，授权/鉴权同（需用户/会话存储）。
  • 前置依赖：处理器要调的子系统（stream/store·hub·resync、process/manager·lifecycle、driver-channel、进程 git 仓库存储 repos）多数还是接口——它们落地前，端点填不实。
  • 设计待定：几处端点行为还卡在 api.html 的开放问题（Q2 wake 同步/异步、Q3 状态集、Q4 进程终结、Q5 input 返回/文件）。

所以落地顺序：定开放问题 → 对齐 ProcessRecord/ProcessState 类型 → 实现子系统 → 把端点从桩填实。本骨架保证那一步到来时，只需在对应 routes/*.ts 里把 throw 换成真调用。
