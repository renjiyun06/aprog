镜像与沙箱

镜像是进程的运行底座，沙箱是承载一次运行的容器。本篇定义镜像的本地管理与烘制、base 镜像的内容、 沙箱供应商抽象，以及当前供应商 PPIO 的运行约束。镜像与程序的关系参见 02 · 程序与发版， 沙箱在唤醒时的创建时序参见 05 · 运行生命周期。

1 · 镜像与程序的分工

系统的依赖分为两条轴，兑现路径不同，不应混为一谈：

轴对象承载兑现
纵向 · 运行底座镜像（语言运行时、编译器、引擎等重而共享的底座）OCI 镜像 → 供应商 snapshot / template创建沙箱时由 provider.create 拉取
横向 · 程序依赖程序与库（轻、专属、随程序发版）OCI 程序包闭包（见 02）driver 唤醒后按 digest 拉取并部署至 skills/

判据：慢变、跨进程共享、体积大的底座烘入镜像，随镜像版本走；轻量、程序专属、随程序迭代的部分随程序包在 spawn 时部署。

2 · 本地管理与烘制

镜像与程序采用同一套目录模型：git 源目录为管理面，发布产物在仓库之外。镜像是与平台无关的逻辑实体，由各供应商分别打包：

层级示例含义
images/—所有镜像的根目录
<名>/images/base/一个逻辑镜像；目录名即镜像名
<版本>/images/base/0.1.0/一个版本（中立逻辑镜像；跨供应商共享的材料置于此层）
<供应商>/images/base/0.1.0/ppio/某供应商的打包入口目录
· bake.ts该供应商的打包脚本，直接调用其 SDK，无抽象层

发布动作为 aprog-bake <名> <版本> <供应商>，定位并执行对应的 bake.ts，产出供应商 snapshot / template。 snapshot 采用确定性命名 aprog-<名>:<版本>，供各供应商共用；控制平面据此合成 ImageRef.id，无需查表。 资源规格（CPU / 内存 / 磁盘）不在烘制期决定，而在运行时由 Sandbox.create 设定（rootfs 与资源规格解耦）。

3 · base 镜像内容

base 镜像 0.1.0 提供通用的根运行环境：

项内容
基底ubuntu:24.04
运行用户root（HOME=/root）
基础工具ca-certificates、curl、git、unzip、zip、xz-utils、build-essential
语言 / 运行时uv、bun、sdkman、nvm + Node LTS
引擎前端claude-code（全局安装）
模型路由GLM 配置烘入 /root/.claude/settings.json（base_url 与模型映射）；鉴权密钥不烘入，运行时注入

所有运行期必需的可执行文件均链接至 /usr/local/bin（原因见 §5）。

4 · 沙箱供应商抽象

沙箱经供应商抽象 SandboxProvider 收口；网关 ProviderSandboxGateway 与具体供应商解耦。 当前唯一的真实供应商为 PPIO（E2B 同构的托管 microVM，国内出站开放）。更换或新增供应商时，实现一个 SandboxProvider 并提供对应的 resolveImageRef 即可，网关本体不变。沙箱在唤醒时创建：控制平面将 driver 经 files.write 推入， 以后台进程启动，并注入 bindToken、控制平面地址、引擎鉴权与 CA 证书等环境变量（时序见 05）。

5 · PPIO 运行约束

PPIO 的托管运行时对镜像有若干约束，直接影响打包与注入方式：

约束表现对策
envd 由平台注入envd 在镜像 → microVM 转换时由 PPIO 注入，不在你的 Docker 镜像内打包时无需包含 envd；镜像与基底无关
默认运行用户固定为 userPPIO 忽略 Dockerfile 中的 USER root，默认以 user 运行provider 在每次 files.write / commands.run 显式传 user:'root'
非交互 PATH 受限envd 忽略镜像的 ENV PATH；非交互 PATH 仅含 /usr/local/bin 与默认目录（外加自动前置的 nvm node bin）driver / 引擎所需的二进制一律 symlink 至 /usr/local/bin
nvm node 路径node 安装于 /root/.nvm/versions/node/<ver>/bin/，仅在 source 后入 PATH将 node / npm / npx 链接至 /usr/local/bin 以接入常驻 PATH
凭证 每进程 / 每沙箱的凭证（bindToken、引擎鉴权、git 凭证等）一律运行时注入，绝不烘入镜像。
